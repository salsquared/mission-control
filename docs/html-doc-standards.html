Documentation Standard

HTML Documentation Standards

The canonical dos and don'ts for authoring repository documentation as self-contained, browsable HTML. This file is both the rulebook and the reference template — copy its structure when converting an existing doc.

Living standard Applies to: all repos Last updated 2026-05-28

01Purpose & scope

Every active document under docs/ is converging on one consistent, navigable, diagram-friendly HTML format. This standard exists so that any reader — or any agent — lands in a doc and immediately knows how to read it, where the table of contents is, and how to deep-link a section.

This document plays two roles at once:

  • Rulebook — the dos and don'ts below are binding for every HTML doc in the repo.
  • Template — you asked for one file, not a separate skeleton: this file is the skeleton. §02 reproduces its scaffold verbatim; "View Source" on this page is the fastest way to start a new doc.
Companion This standard governs format. Its sibling, Documentation Across Dev Phases, governs content — which kind of doc to write (plan → architecture → implementation) and when. Read both before converting a doc.

02File anatomy (copy me)

Every doc shares one scaffold. The shared look & behavior live in docs/assets/doc.css and docs/assets/doc.js; a doc links them rather than embedding its own styles. Start a new doc by copying this: 

<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1" />
  <title>Doc Title · Mission Control Docs</title>
  <link rel="stylesheet" href="assets/doc.css" />
</head>
<body>
  <div class="layout">
    <aside class="sidebar">
      <div class="brand"><span class="dot"></span> Mission Control Docs</div>
      <nav class="toc" aria-label="Table of contents">
        <a href="#intro">01 · Intro</a>
        <!-- one link per <h2>; add class="sub" for nested <h3> -->
      </nav>
    </aside>
    <main class="content">
      <header class="doc-header">
        <p class="eyebrow">Category</p>
        <h1>Doc Title</h1>
        <p class="lede">One-sentence summary.</p>
        <div class="meta">Last updated 2026-05-28</div>
      </header>
      <section id="intro">
        <h2><span class="num">01</span>Intro</h2>
        <p>…</p>
      </section>
      <footer class="doc-footer">…</footer>
    </main>
  </div>
  <script type="module" src="assets/doc.js"></script>
</body>
</html>
Do Link the shared assets/doc.css + assets/doc.js. One edit there restyles every doc — the whole point of converting the repo.
Don't Inline a full <style> block or per-doc <script>. That re-duplicates the theme into every file and defeats single-source maintenance.
Path note Assets live in docs/assets/ and docs live in docs/, so the link is the relative assets/doc.css. Keep all converted docs in docs/ so this relative path holds. The <script> goes last in <body> and must be type="module" (it imports Mermaid).

03Naming & location

  • One doc = one file = one <h1>. Don't merge two topics into a file or split one topic across files.
  • Filenames are kebab-case .html named for the topic, not the phase: architecture.html, resume-pipeline.html — not phase2.html.
  • Live in docs/; shared assets in docs/assets/. Nothing in the repo root.
  • When converting, keep the same base name (architecture.md → architecture.html) so existing cross-references only need an extension bump.

04Structure & headings

The heading tree and the table of contents are the same tree. Keep them in lockstep:

  • Exactly one <h1>, inside .doc-header.
  • One <section id="…"> per <h2>; the id is kebab-case and stable.
  • <h3> for subsections (also id'd if you want them in the TOC as class="sub" links).
  • Don't skip levels (no <h2> → <h4>). doc.js scroll-spy + anchor generation rely on ids being present on h2/h3.
ids are a contract Other docs deep-link to your-doc.html#section-id. Renaming an id silently breaks those links. Treat ids like public API — add freely, rename rarely.

05Semantic HTML

Do Use <header>, <nav>, <main>, <section>, <figure>/<figcaption>, <table>, <aside>. Use <strong>/<em> for meaning and <ul>/<ol> for real lists.
Don't Build a <div> soup, use <b>/<i> for emphasis, fake a list with <br/>-separated paragraphs, or nest a second <main>.

The class names in doc.css (.callout, .badge, .table-wrap, .diagram) are the vocabulary — reach for an existing one before inventing a new class. If you genuinely need a new component, add it to doc.css so the next doc can reuse it.

06Diagrams (Mermaid)

Node-based graphs must be Mermaid, never ASCII art (the repo-wide rule from CLAUDE.md). doc.js auto-renders any <pre class="mermaid">. Wrap it in a <figure class="diagram"> with a caption: 

<figure class="diagram">

Do Use <br/> for line breaks inside labels. Quote parens in edge labels: -->|"fetch()"|. Parens inside quoted node labels (["text()"]) are fine.
Don't Use \n in labels (renders literally), leave parens unquoted in |…| edge labels (parse error), or paste a screenshot of a diagram instead of the Mermaid source.

07Tables

This repo's docs lean heavily on tables (status matrices, decision logs, callsite inventories). Use a real <table> with a <thead>, wrapped in .table-wrap so it scrolls horizontally on narrow screens instead of breaking the layout:

ElementUse it forDon't
<thead><th>The header row of any data tableStyling a plain <td> to look like a header
.table-wrapThe scroll container around every tableLetting a wide table overflow the page grid
<table>Tabular data & comparisonsPage layout (use CSS grid in doc.css)

08Code & references

  • Block code: <pre><code>…</code></pre>. Escape &lt; / &gt; inside it.
  • Inline code: <code> for identifiers, paths, commands.
  • File references use the repo's greppable path:line form — e.g. lib/applications/ingest.ts:131 — so a reader can jump straight to source.
  • Optional syntax tint: wrap tokens in <span class="t-tag|t-attr|t-str|t-com|t-kw">. Purely decorative — skip it for short snippets.
Don't Paste a screenshot of code or a terminal. It isn't selectable, searchable, diffable, or accessible. Always use real text in a <pre>.

09Callouts

Four flavors, one markup shape: <div class="callout flavor"><span class="label">…</span> …</div>.

Do Affirmative guidance — the recommended path.
Don't Anti-patterns — the thing to avoid.
Note Context, cross-references, asides.
Warning Footguns, irreversibility, contracts that break if ignored.

Pair a Do with a Don't using .ddgrid (two-column on wide screens, stacked on mobile). Use callouts for genuinely load-bearing points — if every paragraph is a callout, none of them stand out.

10Links & cross-references

Do Link sibling docs relatively (./architecture.html), deep-link sections (./architecture.html#data-flow), and add rel="noopener" to any target="_blank" external link.
Don't Hardcode absolute filesystem paths (/home/user/…), localhost URLs, or link to a .md file that has already been converted to .html.

During the migration, a not-yet-converted target may still be .md — that's fine; update the link to .html in the same PR that converts the target.

11Accessibility & responsiveness

  • lang="en" on <html>; charset + viewport in <head>.
  • Every <img> has alt text; every <figure> has a <figcaption>.
  • Don't override the theme with low-contrast custom colors — doc.css already meets contrast on the dark palette.
  • Don't set fixed pixel widths; the grid in doc.css is responsive (sidebar collapses below 880px). Let it do its job.
  • Motion is gated by prefers-reduced-motion in doc.css — don't add JS animations that ignore it.

12Metadata & freshness

The .doc-header .meta row carries the doc's status and freshness:

  • Status badge — Living standard, Stable, or Draft.
  • Last updated uses an absolute ISO date (2026-05-28) — never "last week" or "recently" (repo-wide rule).
  • The <footer class="doc-footer"> links back to this standard and to the source doc it was converted from.

14Pre-commit checklist

Before committing a new or converted doc, confirm:

  • Links assets/doc.css + assets/doc.js; no inline theme.
  • Exactly one <h1>; every <h2>/<h3> has a stable id.
  • The .toc has one link per <h2> (+ .sub links for key <h3>s).
  • Diagrams are Mermaid (<br/> not \n; parens quoted in edge labels); no ASCII art or screenshots.
  • Tables wrapped in .table-wrap with a real <thead>.
  • Code in <pre><code> with </> escaped; file refs in path:line form.
  • .meta has a status badge + an absolute ISO last-updated date.
  • Opens cleanly in a browser; sidebar scroll-spy highlights the right section; diagrams render.
