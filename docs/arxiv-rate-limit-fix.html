arXiv Rate-Limit — Permanent Fix

Design + implementation plan. arXiv 429s recur because dev and prod run on one box, share one IP, and never share research-fetch state — so every paper is fetched twice and the two token buckets burst-collide. This plan removes all three load multipliers. Status: PLAN ONLY — not yet built (2026-05-31). All OQ1–OQ10 resolved 2026-05-31 — design fully pinned; cleared to build. (OQ10: a dedicated withSharedCache() wrapper; the generic withCache is left untouched.)

Current state & refresh rates

Every arXiv-touching route caches for one hour

All research routes wrap their handler in withCache(..., { ttlSeconds: 3600 }) — no per-route variation. On top of the TTL, a per-process token bucket (lib/arxiv/rate-limit.ts) paces every arXiv fetch at ARXIV_RATE_PER_MIN=7 (one request per ~8.6s, no burst, queue cap 20). The 7 is already half the 15 default — a prior band-aid acknowledging the shared-IP problem in the .env.* comments.

Live fan-out is AI + Physics only

Only AIView and PhysicsView call these routes. SpaceView doesn't touch arXiv; crypto/space topics in the pick routes are dead paths. So the live arXiv key set is 6 keys:

  • Physics → yesterday, week, review, historical (4 keys — this is the "physics has at most 4 calls" the user noted)
  • AI → review, historical (2 keys; its listings go to HuggingFace, not arXiv)

react-query window-focus refetches are absorbed by the server cache (they hit withCache, not arXiv), so they are not a contributor.

Root cause — three multipliers

Why the 7/min throttle doesn't save us

The throttle paces requests but never reduces how many distinct fetches happen. Three independent multipliers stack on top of the 6 keys:

  1. No cross-tier dedup — the ×2. CACHE_BACKEND=sqlite is on, but lib/repositories/cache-entries.ts writes L2 through prisma.cacheEntry, which is the per-tier DB (dev.db vs prod.db). Dev and prod keep separate caches and each fetches every key independently. Every arXiv key is pulled twice.
  2. Uncoordinated buckets — burst collisions. The token bucket lives on globalThis per process (lib/arxiv/rate-limit.ts:33). Dev's and prod's buckets are phase-independent, so two requests can hit the shared IP <3s apart even though the average (14/min combined) is under arXiv's limit. arXiv 429s on the instantaneous spacing, not the average.
  3. Re-fetching immutable picks + a 1h TTL on weekly data. review/historical persist only paperId (see recordReviewPick: data: { paperId, topic, weekStart }). So even when the week's pick is already chosen, the route re-hits arXiv via id_list= and Semantic Scholar every cache-miss just to re-render metadata that never changes — hourly, per tier. A 1h TTL on data that turns over daily (listings) or weekly (picks) means ~24 refetches/day for data that changes once.
The exact same problem was already solved for Gemini. lib/ai/llm-cache.ts (design: docs/archive/cross-tier-llm-dedup.html) exists precisely because "dev and prod run the same code on one box against the same data, so every cacheable step would call Gemini twice." Research never got that treatment. This plan ports the pattern.

The permanent solution

Three layers, each killing one multiplier

All three are committed (OQ4: build Layer 3 now, not gated on measurement). Build order stays Layer 2 → Layer 1 → Layer 3 — Layer 2 is nearly free and lands the biggest volume cut first — but all three ship in this effort. Layer 1 includes cross-tier single-flight (OQ3) and folds in /api/research/hf (OQ2); Layers 1 + 3 both build on a shared SQLite base (OQ5).

Layer 2 — persist picks + raise TTLs

Persist pick metadata so picks never re-hit arXiv

Add nullable metadata columns to both pick models. Nullable so existing rows migrate cleanly — a NULL triggers the legacy id_list fetch once, which then backfills the row.

  • app/api/research/review/route.ts + historical/route.ts — when existingSelection exists and has metadata, return it directly and skip both the arXiv id_list fetch and the Semantic Scholar enrichment. Only when metadata is NULL (pre-migration row) do the one-time id_list fetch, then backfill. On a fresh search-and-pick, persist the metadata in the same recordPick call.

Likely solves the 429s outright — but Layers 1 and 3 still ship (OQ4): they remove the dev+prod root cause structurally, not just the symptom.

Layer 1 — shared cross-tier cache

A dedicated withSharedCache() wrapper, withCache left untouched

The only reason dev and prod don't already share L2 is that cache-entries.ts points at the per-tier Prisma DB. Fix: a best-effort store on a third SQLite file (neither tier's DB), as a thin adapter over the fat shared base (OQ5/OQ7 — a generic keyed store with the reservation protocol; see §7). The 4 research routes reach it through a dedicated withSharedCache() wrapper (OQ10) — a sibling of withCache; the generic wrapper every other cached route uses is not edited. Per OQ2 this covers /api/research/hf too, and per OQ3 it carries the full cross-tier single-flight protocol from day one. lib/ai/llm-cache.ts refactors onto the same base this effort (OQ8).

Why a wrapper and not { read, write }. Cross-tier single-flight (OQ3) requires the cache to wrap the arXiv fetch itself (reserve → lead/compute → finish), so two tiers can't both call arXiv on a simultaneous miss. A read/write store can't express that — the handler would run on both tiers before either writes. So the store is compute-mediating (getOrCompute), which is exactly the fat base's reserve/lead/follow (OQ7), and the wrapper drives it.
// NEW lib/research/shared-cache.ts — thin getOrCompute() adapter over the fat base
//   - data/research-cache.db, WAL, busy_timeout, guarded dynamic import of better-sqlite3
//   - reserve/lead/follow single-flight (OQ3) so simultaneous dev+prod misses fetch ONCE
//   - BEST-EFFORT: any init/IO failure degrades to a direct compute() (uncached)
//   - schema: research_cache(key, data, expiry, status, reserved_at, done_at)

}

Layer 3 — shared cross-tier rate bucket

One paced line for both processes, file-locked

Replace the globalThis per-process bucket with a single-row SQLite bucket both processes draw from atomically — so the two tiers can't burst-collide on the IP.

// EDIT lib/arxiv/rate-limit.ts (or NEW lib/arxiv/shared-bucket.ts wired in)
//   table: arxiv_bucket(id INTEGER PRIMARY KEY CHECK(id=1), tokens REAL, last_refill INTEGER)
//   acquireArxivSlot():
//     BEGIN IMMEDIATE              -- cross-process write lock (busy_timeout handles contention)
//       refill tokens by elapsed * rate, cap at 1 (strict pacing, no burst)
//       if tokens >= 1: tokens -= 1; waitMs = 0
//       else:           reserve next slot, advance last_refill, compute waitMs
//     COMMIT
//     await sleep(waitMs)          -- pacing happens in-process, lock released first
//   BEST-EFFORT: file/native failure → fall back to the current globalThis bucket.
Env semantics change. ARXIV_RATE_PER_MIN currently means per-process (7 each → ~14 combined). With a shared bucket it becomes the single shared rate — set it to ~14 (still under arXiv's ~20/min = 1-per-3s) and update the explanatory comments in both .env.development and .env.production. Getting this wrong (leaving 7 as "shared") needlessly halves throughput; leaving 14 as "per-process" re-creates the collision. Call it out in the commit.

Committed for this effort (OQ4: build now, not gated on measurement). The bucket calls the base's low-level openDb() seam (open / WAL / degrade) and keeps its own single-row BEGIN IMMEDIATE refill logic (OQ9) — no duplicated open/degrade code. Keep the per-process globalThis bucket as the best-effort fallback so a shared-file failure can never block arXiv calls.

Risks & rollback

Known risks

  • better-sqlite3 ABI mismatch (after an nvm Node switch) — both new stores use the same guarded dynamic-import + best-effort degrade as llm-cache.ts, so a load failure runs uncached / per-process, never crashes. This is a hard requirement, not a nicety: correctness must never depend on either shared file being up.
  • Two-DB migration drift — the new columns must land on prod.db too, or the scheduler/prod route throws Prisma P2021/unknown-column. Run migrate deploy on prod in the same step.
  • Env-semantics flip (Layer 3) — see the warning in §6; the shared-vs-per-process meaning of ARXIV_RATE_PER_MIN changes. Easy to get backwards.
  • Stale pick metadata — once persisted, a pick's title/summary never refresh. That's correct (arXiv abstracts are immutable), but if a pick row is ever wrong, clearing the row re-triggers a fresh fetch.

Rollback

Each layer is independent and reversible. Layer 2: revert TTLs + route conditional (columns can stay, harmless when unread). Layer 1: revert the 4 routes from withSharedCache back to withCache — they fall back to per-tier prisma L2 (generic wrapper was never touched). Layer 3: drop the shared-bucket call — acquireArxivSlot uses the globalThis bucket. No layer is load-bearing for correctness; all three are pure optimizations over a working baseline.
