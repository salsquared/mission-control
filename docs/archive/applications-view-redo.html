Applications View — Single-Track Switch Redo

The Applications dash currently renders two full copies of the same UI — one per track (career / side) — stacked down the page across four section headers. This redo collapses them into one set of cards driven by a track switch at the top. Pure presentation change: the data layer, the API, and the three cards (already track-aware) are untouched. The only new asset is the switch; the only new state is which track is active.

✅ SHIPPED 2026-05-29 — archived. Built as designed: the four sections collapsed to one "Applications Pipeline" section with a config-driven TRACKS segmented switch, a single <CardGrid columns={2}> stack (Interviews → Kanban → New Postings · Watchlists → Account Status), and the new device-local applicationsTrack store field (persist v8→9). Refinements added during review, beyond this plan: (1) per-track presentation is now color-only — title + icon are FIXED across tracks (Kanban = suitcase "Pipeline Kanban", New Postings = newspaper "Postings", Watchlists = eye "Watchlists"), since the title/icon swap on toggle read as jarring; (2) the Kanban career accent moved blue → cyan to match the other two cards; (3) a random-per-card track-colored radial glow on the three switchable cards — origin seeded by card id (stable across reloads), cyan (career) / amber (side) — added via a new wrapperStyle passthrough on Card + CardGrid.CardItem (a background-image layered over the canonical bg-black/40); (4) NewPostingsCard's top-right count subtitle tinted to the track color. Verified tsc --noEmit clean; live on dev :4101. Prod (:3101, compiled) needs npm run build + restart to pick it up.

Why — the doubling problem

Current shape

Today components/views/ApplicationsView.tsx mounts four <Section>s and both tracks' cards simultaneously. The kanban, watchlists, and new-postings cards each appear twice (once per track), and the page reads as two near-identical halves. The shared cards — Upcoming Interviews and Account Status — live up top inside the career section, far from where the side user is looking.

flowchart TB
  s1["Section: Applications Pipeline"] --> k1["Kanban — career"]
  s1 --> cal1["Upcoming Interviews — shared"]
  s1 --> acc1["Account Status — shared"]
  s2["Section: Job Discovery"] --> wl1["Watchlists — career"]
  s2 --> np1["New Postings — career"]
  s3["Section: Side Pipeline"] --> k2["Kanban — side"]
  s4["Section: Side Discovery"] --> wl2["Watchlists — side"]
  s4 --> np2["New Postings — side"]

Everything needed for the merge already exists: ApplicationsKanbanCard, NewPostingsCard, and WatchlistsCard each take an optional track prop and carry per-track TRACK_PRESETS (title / icon / color / empty copy). The view just renders each one twice instead of once-with-a-toggle.

Scope. No card component, schema, repository, or API route changes. The redo lives almost entirely in ApplicationsView.tsx, plus one device-local field in the Zustand store. (As shipped, two shared primitives also gained a tiny optional wrapperStyle passthrough — see the SHIPPED banner.)

Target layout

One section, a switch, a single card stack

Keep the single “Applications Pipeline” section header. Drop the Job Discovery, Side Pipeline, and Side Discovery headers entirely. Directly under the title sits the track switch; under that, one stack of cards that re-points to the active track when the switch flips.

flowchart TB
  hdr["Header — Applications Pipeline"] --> sw{"Track Switch:  Career  |  Side"}
  sw --> cal["Upcoming Interviews — shared · full width"]
  cal --> kan["Pipeline Kanban — active track · full width"]
  kan --> np["New Postings — active track · half"]
  kan --> wl["Watchlists — active track · half"]
  np --> acc["Account Status — shared · full width · bottom"]
  wl --> acc

Card order, top to bottom (the user-specified sequence):

  1. Upcoming Interviews — shared across tracks (interviews are interviews; one Gmail account). First thing under the switch, full width.
  2. Pipeline Kanban — active track, full width (8 status columns want the room).
  3. New Postings + Watchlists — active track, sitting side-by-side 50/50 (matching today's discovery grid; collapses to stacked on mobile).
  4. Account Status — shared, full width, pinned at the very bottom.

Grid mechanics

The whole stack is one <CardGrid columns={2}>. In a 2-column grid, colSpan: 2 = full width and colSpan: 1 = half; with grid-flow-row-dense the order resolves to full → full → [half · half] → full without manual row math:

const cards: CardItem[] = [
  { id: "calendar",     colSpan: 2, className: "max-h-[40vh]", content: /* Upcoming Interviews */ },
  { id: "kanban",       colSpan: 2, className: "max-h-[50vh]", wrapperStyle: trackGlow("kanban", activeTrack),       content: /* Kanban  track={activeTrack} */ },
  { id: "new-postings", colSpan: 1,                            wrapperStyle: trackGlow("new-postings", activeTrack), content:  },
  { id: "watchlists",   colSpan: 1,                            wrapperStyle: trackGlow("watchlists", activeTrack),   content:  },
  { id: "conn-status",  colSpan: 2,                            content: /* Account Status */ },
];
Why key={activeTrack}. The card frame (the CardItem.id-keyed wrapper) stays mounted across a switch, but the inner card remounts — so per-card UI state (page number, search box, filter panel, bulk-select mode) resets cleanly instead of carrying “career” search text over onto the side feed.

The track switch

Config-driven segmented control

The switch is the one new asset: a segmented pill control rendered between the section header and the grid. It is driven by a single TRACKS array so that renaming a track or adding a third is a one-line edit, not a UI rewrite (the user flagged “potentially more or renamed tracks”).

const TRACKS = [
  { id: "career", label: "Career", icon: Mail,      activeClass: "bg-blue-500/20  text-blue-200  border-blue-400/40" },
  { id: "side",   label: "Side",   icon: Briefcase, activeClass: "bg-amber-500/20 text-amber-200 border-amber-400/40" },
] as const;
flowchart LR
  sw["Track switch click"] -->|"setApplicationsTrack(id)"| store["useAppStore.applicationsTrack"]
  store -->|"track"| kan["ApplicationsKanbanCard"]
  store -->|"track"| np["NewPostingsCard"]
  store -->|"track"| wl["WatchlistsCard"]

Active pill takes the track's activeClass; inactive pills are muted (bg-black/30 border-white/10 text-white/50) with a hover lift. Each pill shows the track's lucide icon + label.

Adding / renaming a track later touches: APPLICATION_TRACKS (the Zod enum in lib/schemas/applications.ts) → PostingsTrackKey + DEFAULT_POSTING_FILTERS_BY_TRACK (store) → the TRACK_PRESETS map in each of the three cards → the TRACKS array here. The switch and layout pick it up for free.
As shipped: color-only presets. The per-card TRACK_PRESETS now keep title + icon identical across tracks; only color fields (iconColorClass / addBtnClass / subtitle tint) diverge. Plus a seeded radial-gradient glow (trackGlow(cardId, track)) and the track-tinted Postings count subtitle — so a switch recolors the cards rather than relabeling them.

State & data flow

One new device-local field

Add applicationsTrack to the DevicePrefsSlice in components/providers/state/index.ts — per-device, persisted to localStorage alongside activeViewId / postingFilters (a switch preference, not cross-device truth). Default "career".

// DevicePrefsSlice
applicationsTrack: PostingsTrackKey;            // default "career"
setApplicationsTrack: (t: PostingsTrackKey) => void;

// persist partialize  → add  applicationsTrack: state.applicationsTrack
// version 8 → 9        → migrate returns  applicationsTrack: persisted.applicationsTrack === "side" ? "side" : "career"

Queries stay as-is

The view keeps both apps queries (career + side) enabled. That preserves the existing dual-cache optimistic handleStatusChange and the kanban's “move to other track” bulk action without a rewrite — the kanban simply reads whichever cache matches activeTrack. The discovery cards already fetch their own per-track data internally, so they go from two instances mounted to one — strictly fewer live queries than today.

The two AddApplicationModal instances (career + side) collapse to one with defaultTrack={activeTrack}.
