Fetcher Health — Dedicated Store

The Fetcher Health card on the Internal Systems dash is structurally unable to show what it claims to. It derives per-host counts by regex-parsing one PM2 log file on demand — the web process's — so it never sees the continuous background crawling (which runs in the scheduler processes), empties its per-host table after an hour of web quiet while the 1-day badge still reads 100%, and has broken repeatedly on log-format drift. This plan replaces the log-parsing with a small SQLite store every process writes to at the moment of fetch — reusing the best-effort cross-process pattern already proven by data/llm-cache.db (though, unlike that one, this store is per-tier by data — OQ5). Same plumbing, but a typed table you query instead of unstructured text you regex.

The symptom

What the user sees

The card shows “No fetcher activity in the last hour” (and the 6h badge reads —), while the 1d badge simultaneously reads 100%. Meanwhile the in-app log viewer shows fetches happening constantly. The card looks like it “was keeping track until something happened.”

Nothing happened. The card is doing exactly what it was built to do — it's just measuring the wrong process over the wrong window, off a stale snapshot. The contradiction is baked into the design, not a transient glitch.

Two distinct defects produce this. (1) Wrong source — it reads only the web process's log, never the scheduler's, where the continuous crawling lives. (2) Window mismatch — the per-host table is built from the last 1 hour only, but the badges span 1h / 6h / 1d, so the table empties long before the 1d badge does. Both are confirmed against live logs in §2.

Why it's broken — from the code

How the card is computed today

app/api/system/fetcher-health/route.ts is lazy and log-derived. On each request (behind withCache(handler, 3600)) it:

  1. Picks one PM2 log file by NODE_ENV — mission-control-out.log (prod) or mission-control-dev-out.log (dev) — via pm2LogPath().
  2. Reads the whole file, JSON.parses each line, keeps {ts, msg} entries within 24h.
  3. Classifies by message prefix: [EXTERNAL API] / [CACHE HIT] / [CACHE MISS] → ok; [CACHE FALLBACK] → fallback; [SCRAPER BROKEN] → broken.
  4. Extracts a host via parseHostFromLog() (a regex over the message).
  5. Builds totals for 1h/6h/1d, but the per-host map only from entries in the last 1 hour (route.ts:96).

The client (FetcherHealthCard.tsx) then caches that with staleTime: 1h + gcTime: 1h and no refetch on focus/reconnect. So a displayed snapshot can be ~1–2h stale with nothing refreshing it.

Defect 1 — it reads the wrong process

The “continuous fetching” is watchlist crawls + news polling, and those run in the scheduler processes — not the web process the route reads. I reproduced the route's exact logic against every relevant log file (snapshot at 22:17 UTC, 2026-05-31):

Defect 2 — the table window is shorter than the badges

The per-host table is gated on tsMs >= cutoffs['1h'] (route.ts:96); the badges aggregate 1h/6h/1d. So when the web process goes quiet for an hour, the table empties — firing the “No fetcher activity in the last hour” empty state (FetcherHealthCard.tsx:155) — while a 1d badge can still read 100% off older entries. That is the exact contradiction the user reported.

Defect 3 — log fragility (the recurring tax)

The route is layered with scar tissue from past breakage: parseHostFromLog() carries comments about the old loose (\S+\.\S+) regex matching papers..., api..., and arXiv IDs like 2506.13777 as bogus “hosts.” It also only reads the current out.log — PM2-logrotate rolls daily into dated files, so the “1d” window already collapses to near-empty at every midnight rotation. And a verbosity quirk means “ok” means different things per tier: [CACHE HIT]/[CACHE MISS] are gated off in dev (cache.ts:38), so the dev card counts only [EXTERNAL API] while prod also counts cache hits.

Defect 4 — “ok” counts the attempt, not the outcome

loggedFetch() emits its [EXTERNAL API] line before the call (external-fetch.ts:26 — console.info(...) then return fetch(...)), so the card counts a fetch as “ok” the instant it's attempted — regardless of whether the upstream returned 200, 429, 500, or the connection threw. A host that's down and returning 500s for everything still shows 100% healthy. The card answers “did we try?” when the question is “did it work?” — and it conflates a true upstream touch with a cache hit (which never touched the upstream at all). This is the defect the redesign's recording semantics (§5) exist to fix.

Note — there's already a placeholder for this. scheduler/index.ts:138 reads // Future: ... fetcher-health-roll (1 min). A dedicated fetcher-health mechanism was always the intended direction; this plan is that mechanism (minus the roll-up job — see OQ11).

The decision — logs vs. a store

The reframe

The real choice is not “logs vs. a new system.” The card already uses a shared store for cross-process aggregation — the PM2 log file is the only thing that lets the web route see the scheduler's fetches (and it can't even do that today). The question is just: should that shared store be unstructured text you regex, or a structured table you query?

Framed that way, a store isn't extra architecture — it's the same architecture made reliable. And the overhead is small here, because the pattern is already running in prod:

  • data/llm-cache.db is a cross-tier, best-effort, WAL SQLite that both web tiers and both schedulers already open and write to (lib/ai/llm-cache.ts). The hard part — cross-process writes from the scheduler — is solved and live.
  • better-sqlite3 is already a dep, already in serverExternalPackages, already loaded via a guarded dynamic import() that degrades to a no-op on ABI mismatch.
  • scheduler/jobs/llm-cache-prune.ts is the housekeeping precedent — a daily, idempotent, both-tiers prune.
Why not just patch the logs (Option 1)? It would make the card correct (read scheduler + web, align windows) and reuse existing logging for free — a legitimate, cheaper choice. But it keeps monitoring permanently coupled to logging format, which is the documented source of every past bug here, and it can't cleanly fix the log-rotation window. Given the chokepoints are few and the SQLite plumbing is copy-paste-shaped, the store ends the fragility class rather than patching this instance. This doc proposes the store; Option 1 stays the fallback if §9 reveals the store isn't worth it.

Proposed design — the shared store

A dedicated SQLite file — not the Prisma app DB

New module lib/fetcher-health/store.ts, structurally a sibling of lib/ai/llm-cache.ts: a memoized async initStore() opening data/fetcher-health.db (WAL, busy_timeout, synchronous=NORMAL) via guarded dynamic import("better-sqlite3"), with the same best-effort, never-load-bearing contract — any init/write failure degrades to a silent no-op, never throws on the fetch path.

Why a separate file and not a Prisma table in dev.db/prod.db. Per-fetch rows don't belong in the app DBs for two concrete reasons: (1) lib/prisma.ts's $allOperations middleware console.infos every query (on in prod), so an insert per fetch would flood the in-app log viewer with thousands of [DATABASE] Executing create on FetchEvent lines/day; (2) prisma/prod.db is backed up as mailbox-equivalent (it holds the Gmail refresh token), and ephemeral 48h telemetry has no business in that encrypted, Drive-mirrored snapshot. A gitignored data/*.db sidesteps both and keeps best-effort isolation clean (a store hiccup can't break a fetch). We reuse the llm-cache.db pattern — cross-process best-effort better-sqlite3, written by the web and scheduler process of a tier — but this store is per-tier by data (OQ5): one file, a tier column, reads filtered to the serving tier — not cross-tier-merged like the LLM cache.

What gets recorded (OQ1/OQ2/OQ3 decided): a row only for a real upstream touch and its outcome — never a cache hit. Four kinds:

  • ok — upstream returned 2xx/3xx.
  • error — upstream returned non-2xx (a 500/429 is an error) or the request threw.
  • fallback — served last-good stale data after the handler failed (serveStale).
  • broken — a scraper returned 0 items (ScraperBrokenError).

Cache hits are deliberately not stored — a hit never touched the upstream, so it isn't fetcher health, and hit/miss rate is already tracked by cacheStats in lib/cache.ts (surfaced on /api/system). No 5th cache bucket; if the card wants hit-rate for context it reads that existing surface (see OQ3).

Raw-event table (one row per fetch outcome — see OQ7 for raw-vs-bucketed):

CREATE TABLE IF NOT EXISTS fetch_event (
    id     INTEGER PRIMARY KEY AUTOINCREMENT,
    ts     INTEGER NOT NULL,             -- Date.now() at the fetch
    host   TEXT    NOT NULL,             -- e.g. "boards.greenhouse.io"
    kind   TEXT    NOT NULL,             -- 'ok' | 'error' | 'fallback' | 'broken'  (cache HITS not recorded)
    tier   TEXT    NOT NULL,             -- 'dev' | 'prod'  — card filters to the serving tier (OQ5 decided)
    source TEXT                          -- process: 'web' | 'scheduler'  (OQ4: card defaults to union)
);
CREATE INDEX IF NOT EXISTS idx_fetch_event_ts ON fetch_event(ts);

The public surface is two calls:

export type FetchKind = 'ok' | 'error' | 'fallback' | 'broken';

// Hot path — fire-and-forget, swallows every error. tier/source read from env
// (MC_SCHEDULER_TIER / NODE_ENV) inside the store, not passed by every caller.
export function recordFetchOutcome(host: string, kind: FetchKind): void;

  health: Record<string, HealthEntry>;   // per-host, over the selected window
  totals: Record<WindowKey, HealthEntry>; // 1h / 6h / 1d   — each {ok,error,fallback,broken}
}>;

Reads are a windowed aggregate, cheap and rotation-proof:

SELECT host, kind, ts
FROM fetch_event
WHERE ts >= :cutoff1d AND tier = :servingTier;   -- per-tier (OQ5)
-- The route buckets each row into 1h/6h/1d by its ts (so the four counts
-- per window come from one read); the per-host table uses the user-selected
-- window instead of a hard-coded 1h (fixes Defect 2). At a few thousand
-- rows/24h this is trivially cheap; pre-aggregating in SQL is a later option.

Recording — the fetch chokepoints

Where recordFetchOutcome() goes

The same handful of places that already emit the log lines the route parses today — so this is re-pointing existing instrumentation, not scattering new calls:

ChokepointFileRecords
loggedFetch() — await the responselib/external-fetch.ts:23ok on 2xx/3xx, error on non-2xx or a thrown request (host from the URL). The single source of truth for upstream outcomes.
logExternalCall() — SDK pre-flight markerlib/external-fetch.ts:36ok (attempt); SDK callers can report failures via a paired call — deferred (see note)
cache hitlib/cache.ts:320,341not recorded — not an upstream touch; cacheStats already owns hit-rate (OQ1)
cache misslib/cache.ts:354not recorded here — the miss runs the handler, whose real upstream call is captured by loggedFetch (recording here too would double-count)
serveStale()lib/cache.ts:414fallback
ScraperBrokenErrorlib/fetchers/errors.ts:3broken

Recording lives at loggedFetch (the real fetch, where the Response status is known) rather than the cache layer — so the per-call outcome is accurate and there's no double-counting. loggedFetch / logExternalCall already fan out across the fetchers (rss, scrape, snapi, google-news, custom), lib/postings/liveness.ts, and the research/space API routes — so wiring these two helpers covers both the web routes and the scheduler crawls in one place.

Gap to close — the finance routes bypass loggedFetch. app/api/finance/route.ts:33-36 and app/api/finance/history/route.ts hit Pulsar with bare fetch() and rely on withCache's upstreamHost (the cache line) for host attribution. Since we stop recording cache lines (OQ1), those hosts would vanish from the card unless converted. Fix: route the finance fetches through loggedFetch (a ~1-line swap per call) — added as task P2.1. company-news needs no change: it delegates to adapter.fetch(), and the registry fetchers (lib/companies/*, lib/fetchers/*) already use loggedFetch.
SDK markers (logExternalCall) record an attempt only. Unlike loggedFetch, the SDK-wrapping helper never sees a Response, so it can't know 2xx-vs-500. v1 records ok on the marker; a per-caller upgrade (wrap the SDK call, record error on throw) is a small follow-up, not v1. Most outcome-bearing traffic flows through loggedFetch anyway.
Host extraction moves to the source. Today the host is recovered by regex from a rendered log string (fragile — Defect 3). At the chokepoint we have the real URL in hand — so recordFetchOutcome takes a clean hostname and parseHostFromLog() is deleted entirely.

The route + card rewrite

Route reads the store, not the log

app/api/system/fetcher-health/route.ts drops all file I/O + regex parsing and calls readFetcherHealth(Date.now(), servingTier) — servingTier from NODE_ENV so each tier's card shows only its own fetches (OQ5). The withCache wrapper can shrink dramatically (the store read is already <1ms) — OQ9 covers the new TTL. The ENOENT “PM2 log file not found” branch is replaced by a “store disabled” note when init failed.

Response shape DOES change now (OQ3 → 4 buckets). FetcherHealthEntrySchema in lib/schemas/system.ts gains an error field ({ ok, error, fallback, broken }), so the health map + all three totals windows carry it. That's a contract change: regenerate docs/apis.html (the api-docs-smoke pre-push gate will fail otherwise), and the card grows an Error column. Earlier draft said “shape unchanged” — that was before the error bucket was decided.

Card — source filter, error column, live window

Three card changes, all decided:

  • Source filter (OQ4): a segmented All · Web · Scheduler control — the config-driven segmented-control pattern from ApplicationsView — scopes the badges + per-host table; flipping it refetches with a source param. So you can tell whether it's the live dashboard's fetches or the background crawls that are healthy/broken.
  • Error column (OQ2): a red Error column beside Broken so 500s/timeouts are visible at a glance; the health-% denominator becomes ok / (ok + error + fallback + broken).
  • Live window (OQ10): the three badges (1h/6h/1d) become clickable and drive the per-host table window — killing the “table empty but 1d=100%” contradiction — and the client cache drops to near-live (~30s) with focus-refetch, so a fresh snapshot actually reflects recent fetches.

Retention & the prune job

Self-bounding by a daily prune

We only ever query the last 24h, so the table self-bounds: a new scheduler/jobs/fetcher-health-prune.ts (mirroring llm-cache-prune.ts) deletes fetch_event rows older than 48h (OQ8 decided), registered in scheduler/index.ts JOBS as a daily, both-tiers, idempotent, best-effort job. At a few thousand fetches/day with a 2-day window, the table stays small (single-digit MB).

Best-effort all the way down. Like the LLM cache: if the store can't init on a tier (ABI mismatch, unwritable path), recordFetchOutcome is a no-op, readFetcherHealth returns empty with a note, and the prune reports disabled: true. The fetch path never sees an error from this subsystem.

Testing

Hermetic smoke

A new scripts/tests/hermetic/fetcher-health-store-smoke.ts (modeled on llm-cache-smoke.ts, using a temp FETCHER_HEALTH_PATH + a reset seam): seed events across the 1h/6h/1d boundaries, assert per-host counts and window totals across all four buckets (ok/error/fallback/broken), assert the per-tier filter (a dev row never shows in a prod read), assert the prune drops only old rows, and assert a disabled store degrades to empty without throwing. Add a focused check that loggedFetch maps a 500 to error and a 200 to ok (the OQ2 contract). Wired into pre-push.sh's SUITES.

docs/apis.html MUST be regenerated (npm run gen:api-docs) because the error bucket changes the fetcher-health response schema — the api-docs-smoke hermetic gate fails on a stale apis.html. Commit the regenerated file alongside the schema change.

Open Questions

These shape the design before any code. Each has my recommendation, but they're genuinely yours to call — flag any you'd decide differently.

Semantics — what does "healthy" even mean?

OQ1 — ✓ DECIDED: cache hits do NOT count as health. A hit means we did not touch the upstream, so it isn't a health signal — record only real upstream touches. Hit/miss rate stays owned by the existing cacheStats tracker (lib/cache.ts → /api/system), so nothing is lost. (Folded into §4, §5.)
OQ2 — ✓ DECIDED: record the real response outcome. A 500/429 is an error, not an ok. loggedFetch awaits the response and maps 2xx/3xx → ok, non-2xx or a thrown request → error. (Clarification on your “is it just tracking whether we get a response at all?” — today it's worse than that: it logs before the call, so it counts the attempt, not even the response. The fix records the actual outcome. Folded into §2 Defect 4, §5.)
OQ3 — ✓ DECIDED: four buckets, no 5th cache bucket. ok / error / fallback / broken — error and broken stay distinct (upstream-returned-bad vs scraper-got-nothing). On your 5th-bucket idea: we don't store cache hits at all — the cacheStats tracker already counts hit/miss (so a stored cache bucket would double-book it), and a hit isn't a fetcher touch. Sub-decision ✓: cache hit-rate stays off this card (default) — it remains on the existing cache-telemetry surface, not duplicated here. (Folded into §4, §6.)

Scope & granularity

OQ4 — ✓ DECIDED: surface the web vs. scheduler distinction. The card gets a segmented filter All · Web · Scheduler (reusing the config-driven segmented-control pattern from ApplicationsView) that scopes the badges + per-host table; the source column (already in the §4 schema) drives it. Lets you see at a glance whether it's the live dashboard's fetches or the background crawls that are healthy/broken — they fail for different reasons. (Folded into §4, §6, §10.)
OQ5 — ✓ DECIDED: per-tier. Record a tier column; each tier's card shows only its own fetches (prod card = prod fetches), avoiding the 2× inflation from both schedulers crawling the same sources. (Folded into §4 schema, §6 route.)
OQ6 — ✓ DECIDED: group by hostname. Raw hostname (boards.greenhouse.io, api.lever.co) — unambiguous, zero extra wiring, same as today. Revisit a logical-label grouping (greenhouse, linkedin) only if the host list reads noisily. (Not to be confused with the source = web/scheduler column from OQ4 — different axis.)

Storage mechanics

OQ7 — ✓ DECIDED: raw events. One row per fetch outcome — simple, flexible windows, pruned to 48h (OQ8). The §4 schema is this. (Folded into §4.)
OQ8 — ✓ DECIDED: 48h retention. One day of slack past the 24h query window — survives a missed prune tick, trivial disk. (Folded into §7.)
OQ9 — ✓ DECIDED: direct sync write. recordFetchOutcome writes each call straight through (matches llm-cache); negligible at our volume, and watchlist bursts are already 3–10s-paced. Revisit only if it surfaces in perf-monitor. (Folded into §4, §5.)

Rollout & the route's cache

OQ10 — ✓ DECIDED: near-live + clickable window. Drop to ~30–60s server withCache + ~30s client staleTime with refetch-on-focus (the store read is <1ms, so the old hour-long caches were only there to avoid re-parsing a multi-MB log). The per-host table window follows the selected badge (clickable 1h/6h/1d) instead of a hard 1h. (Folded into §6, §10.)
OQ11 — ✓ DECIDED (default applied): drop the roll-up idea. Here's what the reference was, since it was unclear: scheduler/index.ts:138 carries a leftover comment — // Future: weekly-paper-pick (Mon 09:00), fetcher-health-roll (1 min). — a note-to-self that someone once imagined a scheduler job named fetcher-health-roll running every minute to periodically compute and store an aggregate (“roll up”) of fetcher health into a summary table. That made sense under the old log-parsing world, where reading health was expensive, so you'd precompute it on a timer. This design makes it unnecessary: we write one row per fetch as it happens (§4) and the route aggregates with a cheap GROUP BY on read — nothing to precompute, no timer job to add. So we drop the idea and delete that dead comment (the only scheduler job we add is the 48h prune in §7 — housekeeping, not a roll-up). The alternative — a periodic roll-up table for long-horizon trend history — is only worth it if OQ8 grows to 7d+ and you want sparklines; easy to veto back to that if you change your mind.
OQ12 — ✓ DECIDED: start fresh. No backfill — the card fills within minutes of deploy given crawl cadence; one-time log parsing would be throwaway code. Card is briefly empty right after deploy, then self-populates. (Folded into §10 P6.)
OQ13 — ✓ DECIDED: fully delete the log-parsing path. Once the store is live, pm2LogPath() / parseHostFromLog() / the file read all go — no log fallback (keeping it would reintroduce the fragility we're removing). Clean replace. (Folded into §6, §10 P3.)

Adjacent / optional
