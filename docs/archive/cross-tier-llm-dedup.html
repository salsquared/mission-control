mission-control

Cross-tier LLM call deduplication

dev and prod run the same code on the same machine against the same data, so every LLM-backed step (email parsing, posting classification, …) calls Gemini twice for identical input — and bills twice. This is the design for calling Gemini once per unique input and having the other tier reuse the result. Scoped to email ingestion first (the hardest case), then extended to every callsite through a single chokepoint. Implementation spec for a handoff agent — see §9 for coordination with the in-flight Gmail-push work.

Problem & goal

Why we pay twice

  • Two tiers, one machine, same inputs. mission-control (prod, :3101 / prod.db) and mission-control-dev (dev, :4101 / dev.db) — plus their two schedulers — run identical code. Scheduler jobs like job-watcher classify the same postings in both tiers today; once the Gmail push fan-out lands (see docs/archive/gmail-realtime-push.html), both webhooks will parse the same email too.
  • Per-process rate buckets. lib/ai/rate-limit.ts stashes its token bucket on globalThis (line 38) — so it's shared within a process, but dev and prod each get their own. Two tiers ⇒ effectively 2 × GEMINI_RATE_PER_MIN against Google. A cache reduces call volume; it does not unify the buckets (that's the gateway's job — §7).

Goal & non-goal

Goal: for a given (model + rendered prompt + output schema), Gemini is called once; the other tier reuses the result — even when both tiers fire near-simultaneously.

Non-goal: a shared ingestion service. Both tiers must keep ingesting independently so both DBs (and both kanbans) stay populated. We share only the expensive, deterministic-ish sub-computation (the LLM call), not the ingestion flow around it. The unit of sharing is the LLM result, keyed on its input.

Design overview

Content-addressed cache + shared SQLite + in-flight reservation

A thin wrapper, llmCached(req, compute), sits in front of every real Gemini call. It computes a content hash of the request, checks a shared SQLite file both tiers open, and either returns a cached result, leads the computation, or waits for the other tier's in-flight computation.

// cache key — everything that determines the output, nothing that doesn't
key = sha256([
  model,                                  // resolved model id (not the optional input)
  system ?? "",
  user,                                   // the fully-rendered prompt
  String(temperature ?? ""),
  String(maxOutputTokens ?? ""),
  JSON.stringify(zToJsonSchema(schema)),  // output contract (same serializer api-docs uses)
].join(""));

Two properties fall out of putting the rendered prompt + model in the key:

  • Auto-invalidation. Edit a prompt template or bump a model ⇒ new key ⇒ never a stale hit. TTL becomes pure housekeeping (§7), not correctness.
  • Cross-tier consistency bonus. dev and prod now return the same classification for the same input instead of independently drifting.
Zero new deps. better-sqlite3 is already in node_modules (synchronous, single-file, perfect for a one-box KV). Node is v24 — node:sqlite exists too but is still experimental; prefer better-sqlite3.

The shared store

A third SQLite file, neither tier's DB

dev.db and prod.db are separate, so the cache lives in its own file both processes open — e.g. data/llm-cache.db (gitignored; it's a cache, not state). WAL mode gives many readers + a serialized writer; busy_timeout makes a brief cross-process write contention wait rather than throw.

CREATE TABLE IF NOT EXISTS llm_cache (
  key         TEXT PRIMARY KEY,
  name        TEXT,            -- callsite name (chatJSON.name / 'parse-application-email') — observability
  model       TEXT,
  status      TEXT NOT NULL,   -- 'pending' | 'done'
  result      TEXT,            -- JSON string; NULL while pending
  reserved_at INTEGER NOT NULL,-- ms epoch — lease clock for stale-leader takeover
  done_at     INTEGER
);
CREATE INDEX IF NOT EXISTS idx_llm_cache_done_at ON llm_cache(done_at);
// module init (lib/ai/llm-cache.ts)
const db = new Database(process.env.LLM_CACHE_PATH ?? "data/llm-cache.db");
db.pragma("journal_mode = WAL");
db.pragma("busy_timeout = 5000");
db.pragma("synchronous = NORMAL");   // cache durability is non-critical; favor speed
Path must resolve identically from both processes. dev and prod have the same CWD (the repo root, under PM2), so a repo-relative path is fine — but make it overridable via LLM_CACHE_PATH and assert the resolved absolute path on startup so a CWD surprise fails loud instead of silently giving each tier its own file (which would defeat the whole thing).

The reservation protocol

Why read-through alone is not enough

The Gmail fan-out delivers the same email to both webhooks within ~the same second. A naive read-through cache would double-miss (both check, both find nothing, both call Gemini). To actually hit "once" we need an in-flight reservation so exactly one tier leads and the other waits.

Leader / follower / steal / fallback

stateDiagram-v2
  [*] --> Check
  Check --> Return: row.status == done
  Check --> Reserve: no row
  Check --> Follow: row pending
  Reserve --> Lead: INSERT OR IGNORE won
  Reserve --> Follow: lost race (someone inserted first)
  Lead --> Return: compute() ok, write done
  Lead --> Release: compute() threw (DELETE row, rethrow)
  Follow --> Return: poll sees done
  Follow --> Steal: pending older than LEASE_MS
  Follow --> Fallback: waited > MAX_WAIT
  Steal --> Lead: conditional UPDATE won
  Steal --> Follow: lost steal
  Fallback --> Return: compute() myself (rare duplicate)
  Release --> [*]
  Return --> [*]
      
export async function llmCached<T>(
  req: { key: string; name: string; model: string },
  compute: () => Promise<T>,
  opts?: { leaseMs?: number; maxWaitMs?: number; pollMs?: number },
): Promise<T> {
  const LEASE = opts?.leaseMs ?? 30_000;   // > worst-case Gemini latency incl. retries
  const MAX_WAIT = opts?.maxWaitMs ?? 8_000;
  const POLL = opts?.pollMs ?? 150;

  const hit = getRow(req.key);
  if (hit?.status === "done" && hit.result) return JSON.parse(hit.result) as T;

  if (tryReserve(req)) return await lead(req, compute);  // INSERT OR IGNORE won → leader

  const deadline = nowMs() + MAX_WAIT;
  while (nowMs() < deadline) {
    await sleep(POLL);
    const r = getRow(req.key);
    if (r?.status === "done" && r.result) return JSON.parse(r.result) as T;   // leader finished
    if (!r) { if (tryReserve(req)) return await lead(req, compute); continue; } // released → re-lead
    if (r.status === "pending" && nowMs() - r.reserved_at > LEASE) {            // stale leader
      if (trySteal(req.key, r.reserved_at)) return await lead(req, compute);    // atomic takeover
    }
  }
  return compute();   // fallback: leader stalled past MAX_WAIT — accept a rare duplicate over hanging
}

async function lead<T>(req, compute): Promise<T> {
  try { const out = await compute(); finish(req.key, JSON.stringify(out)); return out; }
  catch (e) { release(req.key); throw e; }   // DELETE so the next caller can lead cleanly
}

All three mutations are atomic in SQLite even across processes:

  • tryReserve → INSERT INTO llm_cache(...) VALUES(..., 'pending', ...) ON CONFLICT(key) DO NOTHING; .changes === 1 means I won.
  • trySteal → UPDATE ... SET reserved_at=? WHERE key=? AND reserved_at=? (the old value); .changes === 1 means I won the takeover.
  • finish / release → single-row UPDATE / DELETE.

Failure modes & why each is safe

ScenarioOutcome
Both tiers arrive simultaneouslyOne wins INSERT OR IGNORE and computes; the other follows and reuses. One Gemini call.
Leader crashes mid-computeRow stuck pending; first follower past LEASE_MS steals it via conditional UPDATE and re-leads. No permanent stall.
Leader errors (4xx/exhausted retries)Leader DELETEs the row and rethrows; next caller leads cleanly. Errors are never cached.
Leader slow but alive (> MAX_WAIT)Follower falls back to computing itself — a rare duplicate, strictly better than hanging.
Corrupt/locked cache fileWrapper catches store errors and computes directly (cache is best-effort, never load-bearing).
Cache is best-effort. Any store-layer exception (open/read/write) must degrade to "just call compute()". Dedup is an optimization; correctness never depends on the cache being up.

Integration points

One wrapper, two chokepoints

Every Gemini call already funnels through exactly two helpers — wrap both and the whole app is covered:

  • lib/ai/gemini.ts:chatJSON (line 219) — the generic structured-output path used by every callsite except email. Its options (name, system?, user, schema, model?, temperature?, maxOutputTokens?) are exactly the cache-key inputs.
  • lib/email-parser.ts:parseApplicationEmail (line 144) — the SDK-bypassing email path. Key on (MODEL_ID, the rendered prompt, the parse schema).

Where exactly — and move the rate slot inside the leader

Check the cache before acquireGeminiSlot(). On a hit (or as a follower), you spend no rate token and make no API call. The slot acquisition moves into the compute() closure so only the leader pays for it:

// inside chatJSON, replacing the direct `await acquireGeminiSlot(); ...call...`
const key = cacheKey({ model, system: opts.system, user: opts.user,
                       schema: opts.schema, temperature: opts.temperature,
                       maxOutputTokens: opts.maxOutputTokens });
return llmCached({ key, name: opts.name, model }, async () => {
  await acquireGeminiSlot();          // only the leader acquires a token
  return runGeminiCallWithRetries();  // the existing body, unchanged
});

parseApplicationEmail wraps identically around its acquireGeminiSlot() + generateContent body, keeping its existing safeTrack Lunary events inside the closure so a cache hit isn't traced as a model call (or is traced as a cache-hit event — implementer's call; note it in docs/llm-calls.html).

What to cache (and what not to)

Default-on with an opt-out

Most callsites are deterministic extraction/classification (email parse, posting parse, employment-type classify, resume selection) — safe and high-value to cache. A few are intentionally generative and re-rolled for variety; freezing those would be a regression.

Audit docs/llm-calls.html before flipping chatJSON default-on. Add a cache?: boolean (default true) to ChatJSONOptions and set cache: false on any "suggest N things"-style callsite. Per the anti-repetition rule, those calls already fold the user's known items into the prompt — so identical input genuinely should reuse, but if a callsite is meant to vary on re-ask with identical input, it must opt out. Treat the inventory as the checklist; tag each row cache-on/off.

Rate-limit & the gateway evolution

What the cache fixes vs. what it doesn't

The cache cuts duplicate calls, which indirectly relieves rate pressure — but the two tiers still run independent rate buckets. Unifying the bucket (and centralizing observability/cost, or sharing across machines) is the job of a future LLM gateway: one PM2 process both tiers POST to over localhost.

The cache is the on-ramp, not a throwaway

Because callsites only ever touch llmCached(req, compute), promoting the store behind a gateway later doesn't change a single callsite — only the wrapper's body swaps a SQLite read for a fetch('http://localhost:PORT/llm'). Build the cache now; reach for the gateway only when you specifically need a single shared rate bucket or cross-host sharing.

Testing

Hermetic smoke (add to pre-push SUITES)

  • Single-flight: fire N concurrent llmCached(sameKey, compute) where compute increments a counter; assert the counter is 1 and all N callers get the same result.
  • Leader error not cached: first compute throws → assert row released and a second call re-leads (counter increments again).
  • Stale takeover: seed a pending row with reserved_at older than LEASE_MS; assert a fresh caller steals + computes.
  • Cache-down degradation: point LLM_CACHE_PATH at an unwritable path; assert calls still succeed via direct compute.
  • Cross-process (optional): spawn two tsx child procs against one temp LLM_CACHE_PATH, both calling the same key; assert one compute total via a shared side-file counter.

Use a fake compute (no real Gemini) so it stays hermetic. Append the file to the SUITES array in scripts/pre-push.sh.
