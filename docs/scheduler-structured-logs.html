Scheduler Logs — Structured Output + In-App Visibility

The two scheduler processes (mission-control-scheduler-{dev,prod}) do all the continuous background work — watchlist crawls, classification sweeps, digests, prunes — but their logs are plain text, untimestamped, and invisible to the in-app log viewer. The Internal Systems dash shows only the web process's logs, because the logger monkey-patch lives in a Next.js-only startup hook the scheduler never runs, and the log viewer reads a per-process ring buffer + the web PM2 file. This plan gives the scheduler structured JSON log output (with a per-line source/tier field), mutes its per-query verbose noise, and bridges its logs into the in-app viewer — live (near-real-time) and historical — via a per-tier shared SQLite store, the same cross-process pattern as llm-cache.db / fetcher-health.db.

STATUS — DECIDED, ready to implement (2026-05-31). All open questions resolved with the user this session (see §5). Spun out of the Fetcher Health store plan (its OQ14): structured scheduler logging is orthogonal to that card but was surfaced by the same root cause. No code written yet; the firmed-up build plan is in §6.
Relationship to the Fetcher Health store + card. The Fetcher Health plan makes fetcher health independent of log parsing (it records structured outcomes to data/fetcher-health.db and the FetcherHealthCard aggregates them). This doc is the general fix for scheduler observability — every other [SCHEDULER:*] line, not just fetches. They share a root cause but neither blocks the other. The view boundary between them is settled in OQ9: there is no separate “fetcher log” view — fetcher monitoring is the card; raw fetcher log lines live in this general viewer, source-filtered.

The problem

Scheduler logs are a blind spot

Open the Internal Systems dash's log viewer and you see the web process's activity — HTTP requests, cache events, Gmail webhooks. You do not see any of the scheduler's work: which watchlists crawled, what the classifier swept, whether a prune ran, why a job errored. To debug the scheduler today you have to pm2 logs mission-control-scheduler-prod from a terminal — the in-app surface, which CLAUDE.md calls “the canonical observability surface,” simply doesn't have it.

And even at the terminal, the scheduler's lines are plain text with no per-line timestamp ([SCHEDULER:prod] running job-watcher), so they can't be parsed, windowed, or correlated by any tool — which is exactly what made the Fetcher Health card unable to read them.

Why — from the code

The logger only initializes in the web runtime

The structured logger is armed by instrumentation.ts — Next.js's register() hook, gated on NEXT_RUNTIME === 'nodejs'. It calls lib/logger.ts:initLogger(), which monkey-patches console.{log,info,warn,error} + process.stdout/stderr.write to (1) push entries into a 50-deep ring buffer on globalThis, (2) fan out to SSE listeners, and (3) emit a structured {ts, level, msg} JSON line to stdout (logger.ts:83).

The scheduler (scheduler/index.ts) is a plain tsx process. It never imports instrumentation.ts (that's a Next.js mechanism), so initLogger() never runs there. Result: its console.* calls go straight to stdout as plain text, no JSON envelope, no ring buffer, no SSE.

The viewer reads per-process state + the web file

Two read paths, both web-scoped:

  • Live — app/api/system/logs/route.ts is an SSE stream over getLogs() + subscribeToLogs(). That ring buffer lives on the web process's globalThis — a different process from the scheduler, so even if the scheduler had its own buffer the web stream couldn't see it.
  • Historical — app/api/system/logs/historical/route.ts reads a PM2 file hardcoded to mission-control-out.log (the prod web log). It never opens a scheduler file — and, because PM2_LOG_PATH is unset in both env files, it reads the prod web file even on the dev tier (the dev viewer's “load older” already shows prod's logs).

Two facts that shape the fix

Surfacing scheduler logs naively would make the viewer worse, not better — two code facts force the design:

  • The prod scheduler is loud. It runs with NODE_ENV=production (ecosystem.config.cjs), and the verbose-log gates key off exactly that (lib/prisma.ts:26, lib/cache.ts:40). So it emits a [DATABASE] Executing … line for every Prisma query plus [CACHE]/[API Request] lines — the Fetcher-Health diagnosis counted 2913 such markers in the prod scheduler log alone. Piping that into the viewer unfiltered drowns the signal. → muted in OQ7.
  • The [SCHEDULER:tier] prefix covers almost nothing. Only ~6 lines emitted directly by scheduler/index.ts carry the tag (startup, running <job>, the P2021-disable warn, the generic fail, shutdown, ready). Every content line — [job-watcher] processed …, [liveness], [classify-pending…], [DATABASE] — is emitted from job code with no marker. Text-prefix filtering can't separate scheduler from web. → structured source/tier field in OQ3.

Foundation — structured logger in the scheduler

Call initLogger() at scheduler startup

Layer one: import and call initLogger() at the top of scheduler/index.ts (before the JOBS loop). Immediately the scheduler's every console.* gains the JSON envelope on stdout — so its PM2 file becomes timestamped + machine-parseable, and (critically) it grows the ring buffer + listener fan-out that the bridge in §4 drains. We call the full initLogger(), not a lite emit-only variant: the live tail needs the ring buffer anyway, so a second logging code path would be dead weight to maintain (resolved in OQ8's reasoning).

Scope the call narrowly. Call only initLogger() — not the rest of instrumentation.ts's register(), which does web-only setup (Lunary, the Pulsar WS relay, shutdown drains). The scheduler has its own lifecycle; we only want the logging patch.

Stamp source + tier at init

initLogger() auto-derives a process identity once and stamps it on every line, so the viewer can filter cleanly without parsing message text. The discriminator is process.env.MC_SCHEDULER_TIER (set only in the two scheduler processes):

const schedTier = process.env.MC_SCHEDULER_TIER;            // 'dev' | 'prod' | undefined
const source = schedTier ? 'scheduler' : 'web';
const tier   = schedTier ?? (process.env.NODE_ENV === 'production' ? 'prod' : 'dev');

The JSON envelope (logger.ts:83) and the LogEntry type both gain source + tier; addRawLog stamps them too. Old lines without the fields still parse (back-compat default). This gives the viewer an All · Web · Scheduler filter that mirrors the one chosen for the Fetcher-Health card.

Mute the scheduler's verbose noise

So the bridge carries signal, the scheduler must stop generating per-query spam — not just hide it downstream. Add && !process.env.MC_SCHEDULER_TIER to the two verbose gates so the scheduler never emits [DATABASE]/[CACHE]/[API Request], regardless of its NODE_ENV:

// lib/prisma.ts:26  &  lib/cache.ts:40
const verbose =
  (process.env.NODE_ENV === 'production' && !process.env.MC_SCHEDULER_TIER)
  || process.env.DEBUG_PRISMA === '1';        // (DEBUG_VERBOSE_LOG for cache.ts)

Web tiers are untouched. DEBUG_PRISMA / DEBUG_VERBOSE_LOG remain the opt-in escape hatch when you do want to see the scheduler's queries. The job-level summaries ([job-watcher] processed …, [liveness] verdicts, errors) are not gated — those are the signal we want.

Chosen architecture — the scheduler→web bridge

Plan B, scheduler-only sink

Structured stdout alone does not put scheduler logs in the viewer — the ring buffer is per-process. The bridge is a per-tier shared SQLite store, data/logs.db (gitignored, WAL, best-effort), the same cross-process pattern as data/llm-cache.db and data/fetcher-health.db. Three properties of the chosen shape:

  • Scheduler-only sink. Only the scheduler writes to logs.db (via a subscribeToLogs() listener registered in the scheduler — not inside the shared initLogger, so web never writes). The web in-memory ring + SSE stay exactly as they are: instant, battle-tested, zero new risk to the working path.
  • Live = near-live poll. The web SSE route keeps streaming its own ring (web rows, instant) and additionally polls logs.db every ~1s for rows past a monotonic cursor (id > lastSeenId), pushing each into the same SSE stream, source-tagged. ~1s lag is invisible for a log tail.
  • Historical = file + store. The historical route reads web rows from the (per-tier-corrected) PM2 file and scheduler rows from logs.db (a time-window query, ~48h), merges by ts, source-tagged. The scheduler's own PM2 file stays JSON-structured for terminal pm2 logs but is not parsed by the route — logs.db is its in-app source.

Plan C (push-to-web HTTP) was rejected: it couples the two processes at runtime (web must be up; logs vanish on a web rebuild; no persistence), against the best-effort philosophy of the existing shared-SQLite stores. Single-source-of-truth (both processes write logs.db, web reads it for everything) was rejected too — it would put the un-muted web firehose through a synchronous SQLite insert on the request path and add ~1s latency to web logs (full rationale in OQ8).

Retention. A daily logs-prune job trims logs.db to 48h, mirroring fetcher-health-prune (slots into scheduler/index.ts JOBS next to it). 48h is a default, easily tunable — post-mute the scheduler's write volume is modest, so a longer window is cheap if “what did the scheduler do this week” turns out to matter.

Decisions (resolved Open Questions)

Decision log

All resolved with the user 2026-05-31. OQ1–OQ6 carry over from the proposal (their answers below); OQ7–OQ9 were surfaced during review. Recorded here as the decision log.

OQ1 — Scope → in-app visibility, not terminal-only. Structured PM2 files alone (a legitimate 20-min fix) aren't the goal; the in-app viewer is the whole point. Structured stdout is the foundation (§3), not the deliverable.
OQ2 — Transport → live tail via Plan B (shared SQLite). Not historical-only. data/logs.db bridges scheduler→web; the web SSE route polls it (~1s) for live, the historical route reads a window from it. Matches the house cross-process pattern (llm-cache.db, fetcher-health.db): best-effort, no runtime coupling, survives either process restarting. Plan A (file-merge) is folded in as the historical half; Plan C (push-to-web) rejected.
OQ3 — Source tagging → structured source + tier field. Auto-derived in initLogger from MC_SCHEDULER_TIER (absent ⇒ source=web, tier from NODE_ENV). Text-prefix parsing rejected: only ~6 lines/process carry [SCHEDULER:tier]; all job-body lines are unprefixed (see §2). The field also feeds the viewer's All · Web · Scheduler filter and is required for the live path (no file boundary to infer source from).
OQ4 — Double-emit → keep both lines. initLogger emits the JSON line and the original plain line; keep both in the PM2 file for readable terminal pm2 logs. The in-app surface no longer depends on the scheduler's file (its in-app source is logs.db), and post-mute (OQ7) the volume is modest — pm2-logrotate already bounds size.
OQ5 — Historical hardcoded path → fix in-scope. historical/route.ts:12 always reads the prod web mission-control-out.log; correct it to pick mission-control{,-dev}-out.log by NODE_ENV/tier so the dev viewer reads dev. (The original draft pointed at the old fetcher-health route's NODE_ENV file-selection as the pattern — dropped: that log-parsing route has since been deleted by the now-shipped Fetcher-Health store work. Reuse the per-tier helper that work established, e.g. currentTier() in lib/fetcher-health.)
OQ6 — Notification relationship → visibility first. No alerting in this plan. The structured source/tier/level logs become the substrate a future notification system reads (scheduler errors, a tier disabling a job on P2021 are prime alert candidates) — hook that later.
OQ7 — Verbose noise in the scheduler → mute at the gate. (New.) Add && !MC_SCHEDULER_TIER to the verbose gates in lib/prisma.ts + lib/cache.ts so the scheduler never generates [DATABASE]/[CACHE]/[API Request] (it runs NODE_ENV=production on prod, which would otherwise turn them on — 2913 markers in one log). Web unchanged; DEBUG_PRISMA/DEBUG_VERBOSE_LOG remain the opt-in.
OQ8 — logs.db scope → scheduler-only bridge. (New.) The web ring + SSE stay untouched; logs.db carries only scheduler rows. Single-source rejected: it would route the un-muted web [DATABASE]-per-query firehose through a synchronous better-sqlite3 insert on the request path, add ~1s latency to web logs, and widen the best-effort blast radius (a logs.db failure would take down the web tail too). Scheduler-only is the surgical choice and doesn't rebuild a working layer.
OQ9 — Fetcher-view boundary → no separate fetcher log view. (New.) Fetcher monitoring = the FetcherHealthCard (aggregated ok/error/fallback/broken per host, from fetcher-health.db). Raw fetcher log lines = this general viewer, filtered by source=scheduler + free-text. A dedicated fetcher-log view would re-introduce the log-parsing the fetcher-health store exists to retire. Optional future affordance (not v1): a drill-down link from a red host in the card → the logger pre-filtered to that host.

Implementation tasks

Build checklist

P1Structured logger in the scheduler + source/tier + mute

(a) lib/logger.ts: add source + tier to LogEntry and the JSON envelope (addLog @ :83 and addRawLog); auto-derive once at init from MC_SCHEDULER_TIER (fallback NODE_ENV → {prod,dev}). Old lines without the fields still parse.

(b) scheduler/index.ts: import { initLogger } + call it before the JOBS loop — only initLogger(), not the rest of register().

(c) lib/prisma.ts:26 + lib/cache.ts:40: add && !process.env.MC_SCHEDULER_TIER to the verbose gate (OQ7); leave DEBUG_* overrides.

Verify: restart both scheduler tiers; confirm {ts,level,msg,source,tier} lines and the absence of [DATABASE]/[CACHE] spam in mission-control-scheduler-{dev,prod}-out.log.

P2The logs.db store (scheduler-only sink)

New lib/logs-store.ts, modeled on lib/ai/llm-cache.ts / the fetcher-health store: guarded dynamic better-sqlite3 import (in serverExternalPackages), WAL, data/logs.db, table LogLine(id INTEGER PK AUTOINCREMENT, ts, level, source, tier, msg). Best-effort: any store failure degrades to a no-op (no throw on the logging path). Exposes insert(entry), since(cursorId, limit), window(fromMs, toMs), prune(cutoff).

Register a subscribeToLogs(insert) listener in the scheduler only (after initLogger() in scheduler/index.ts) — never inside the shared initLogger, so the web process never writes. Add data/logs.db* to .gitignore.

P3Live — web SSE merges scheduler rows

app/api/system/logs/route.ts: keep the in-memory ring + subscribeToLogs (web rows, instant). Add a setInterval(~1000ms) poll of logs.db for id > cursor, enqueueing each as a 'new' SSE event (source-tagged); seed the cursor from the latest id at connect. Include recent logs.db rows in the 'initial' payload, merged with the web ring and sorted by ts. Clear the interval on req.signal abort.

P4Historical — per-tier web file + scheduler from logs.db

app/api/system/logs/historical/route.ts: fix pm2LogPath() to select mission-control{,-dev}-out.log by NODE_ENV (OQ5). Additionally query logs.db for the requested window, merge with the parsed web-file entries, sort by ts, keep the existing level/from/to filters + .slice(-1000). Carry source/tier through.

P5Viewer — source/tier badge + All · Web · Scheduler filter

components/views/InternalView.tsx: thread source/tier through the live (sysLogs) and historical (historicalLogs) state + types; render a small source/tier badge per row; add an All · Web · Scheduler segmented filter over the log card (mirrors the Fetcher-Health card filter). Preserve the existing reverse-order + “load older” behavior and the SSE reconnect/backoff.

P6Prune job

New scheduler/jobs/logs-prune.ts, modeled on fetcher-health-prune.ts: daily, delete logs.db rows older than 48h; no-op when the store failed to init on this tier. Register in scheduler/index.ts JOBS next to fetcher-health-prune.

P7Verify end-to-end

Both tiers: scheduler activity visible in the in-app viewer (live ~1s + historical); the All/Web/Scheduler filter works; no [DATABASE] spam from the scheduler; the web live tail unchanged (instant); terminal pm2 logs still readable; logs.db prunes at 48h. Add a hermetic smoke for the store (insert → since/window → prune) and wire it into pre-push.sh.
