Scheduler Logs — Structured Output + In-App Visibility

The two scheduler processes (mission-control-scheduler-{dev,prod}) do all the continuous background work — watchlist crawls, classification sweeps, digests, prunes — but their logs are plain text, untimestamped, and invisible to the in-app log viewer. The Internal Systems dash shows only the web process's logs, because the logger monkey-patch lives in a Next.js-only startup hook the scheduler never runs, and the log viewer reads a per-process ring buffer + the web PM2 file. This plan gives the scheduler structured JSON log output (the easy, high-value half) and lays out options for surfacing scheduler logs in the in-app viewer (the cross-process half).

The problem

Scheduler logs are a blind spot

Open the Internal Systems dash's log viewer and you see the web process's activity — HTTP requests, cache events, Gmail webhooks. You do not see any of the scheduler's work: which watchlists crawled, what the classifier swept, whether a prune ran, why a job errored. To debug the scheduler today you have to pm2 logs mission-control-scheduler-prod from a terminal — the in-app surface, which CLAUDE.md calls “the canonical observability surface,” simply doesn't have it.

And even at the terminal, the scheduler's lines are plain text with no per-line timestamp ([SCHEDULER:prod] running job-watcher), so they can't be parsed, windowed, or correlated by any tool — which is exactly what made the Fetcher Health card unable to read them.

Why — from the code

The logger only initializes in the web runtime

The structured logger is armed by instrumentation.ts — Next.js's register() hook, gated on NEXT_RUNTIME === 'nodejs'. It calls lib/logger.ts:initLogger(), which monkey-patches console.{log,info,warn,error} + process.stdout/stderr.write to (1) push entries into a 50-deep ring buffer on globalThis, (2) fan out to SSE listeners, and (3) emit a structured {ts, level, msg} JSON line to stdout (logger.ts:83).

The scheduler (scheduler/index.ts) is a plain tsx process. It never imports instrumentation.ts (that's a Next.js mechanism), so initLogger() never runs there. Result: its console.* calls go straight to stdout as plain text, no JSON envelope, no ring buffer, no SSE.

The viewer reads per-process state + the web file

Two read paths, both web-scoped:

  • Live — app/api/system/logs/route.ts is an SSE stream over getLogs() + subscribeToLogs(). That ring buffer lives on the web process's globalThis — a different process from the scheduler, so even if the scheduler had its own buffer the web stream couldn't see it.
  • Historical — app/api/system/logs/historical/route.ts reads a PM2 file hardcoded to mission-control-out.log (the prod web log). It never opens a scheduler file.

The easy win — structured stdout

Call initLogger() at scheduler startup

The high-value, low-risk half: import and call initLogger() at the top of scheduler/index.ts (before the JOBS loop). Immediately the scheduler's every console.* gains the {ts, level, msg} JSON envelope on stdout — so its PM2 file becomes timestamped + machine-parseable, and any tool (including the Fetcher Health reader, if it ever wanted scheduler data) can consume it. This alone fixes the “unstructured, untimestamped” complaint.

Watch the double-emit. initLogger's addLog writes the JSON line and calls the original console.* (the human-readable line) — so each scheduler log becomes two file lines (JSON + plain), the same as the web process today. That doubles scheduler log volume; pm2-logrotate handles size, but it's worth a deliberate choice (see OQ4: keep both, or JSON-only for the scheduler).
Scope the call narrowly. Call only initLogger() — not the rest of instrumentation.ts's register(), which does web-only setup (Lunary, the Pulsar WS relay, shutdown drains). The scheduler has its own lifecycle; we only want the logging patch.

The hard part — in-app visibility

Cross-process is the real work

Structured stdout (above) does not put scheduler logs in the in-app viewer — the ring buffer is per-process and the historical route reads only the web file. Bridging that gap needs cross-process transport. Options, cheapest first:

  • (A) Historical-merge — generalize logs/historical to read + merge the scheduler PM2 files alongside the web file (sorted by ts, tagged by source). Cheap; depends on the easy-win JSON format. Historical only — no live tail of scheduler activity.
  • (B) Shared-store ring buffer — the scheduler writes log entries to a shared SQLite (the llm-cache.db / fetcher-health pattern); the web live + historical routes merge from it. Gives near-live cross-process tail; more moving parts + a prune.
  • (C) Push to web — the scheduler POSTs (or IPC-streams) entries to the web process, which injects them into its existing ring buffer + SSE. True live tail; couples the two processes at runtime.

Recommended split. Ship the easy win + (A) historical-merge as v1 (covers “let me see what the scheduler did” with minimal risk), and treat live cross-process tail (B/C) as a follow-up only if you actually want to watch scheduler activity stream in real time. See OQ2.

Open Questions

Defaults given; all yours to redirect.

OQ1 — Scope: structured stdout only, or also in-app visibility? Options: (a) just the easy win — scheduler logs become structured/timestamped in PM2 files, parseable but still terminal-only; (b) also surface them in the in-app viewer. My take: (b) — the whole point is the in-app surface, and (A) historical-merge is cheap. But (a) alone is a legitimate 20-minute fix if you only care about parseability.
OQ2 — If surfacing: which transport? (A) historical-file-merge, (B) shared SQLite, or (C) push-to-web (per §4). My take: (A) for v1 — historical merge of the scheduler PM2 files, no live tail. Add (B/C) later only if real-time scheduler tailing earns its keep. Want live tail from day one?
OQ3 — Tag log lines by source/tier? Scheduler lines already self-prefix [SCHEDULER:dev] / [SCHEDULER:prod], but a structured source/tier field (vs. parsing the prefix) makes filtering in the viewer clean. Options: (a) rely on the existing text prefix; (b) add a structured field to the JSON envelope. My take: (a) for v1 — the prefix is already there and the viewer can filter on it; structured fields are a refinement.
OQ4 — Double-emit: keep both lines, or JSON-only for the scheduler? initLogger emits JSON + the original plain line. Options: (a) keep both (matches web; human-readable pm2 logs still works); (b) a scheduler-only mode that emits JSON only (halves volume, but raw pm2 logs reads as JSON). My take: (a) — consistency with the web process and readable terminal output beats the volume saving; pm2-logrotate already bounds size.
OQ5 — Historical route hardcodes the prod web file. logs/historical/route.ts:12 always reads mission-control-out.log — so even the web historical view is wrong on the dev tier (and the “fix” here would inherit that). Options: (a) fix it to pick the file by NODE_ENV (like the fetcher-health route does) as part of this work; (b) leave it, separate bug. My take: (a) — it's in the blast radius and a one-line fix; do it here.
OQ6 — Relationship to a future notification system. You mentioned a broader notification system coming. Scheduler errors (a job throwing, a tier disabling a job on P2021) are prime alert candidates. Options: (a) keep this purely about visibility, hook alerts later; (b) emit a structured error signal now for the future system to consume. My take: (a) — visibility first; the structured logs become the substrate that future system reads.
