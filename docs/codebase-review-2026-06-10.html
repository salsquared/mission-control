Security findings

Open sign-in + unscoped global tables

HIGH · lib/auth.ts:40-78 · The signIn callback returns true unconditionally — there is no email/domain allowlist. Any Google account on the internet can complete OAuth through ms-prod.salsquared.xyz and get a valid NextAuth session minted by PrismaAdapter.

User-scoped data (applications, resumes, profile, Gmail-derived rows) is not exposed — those routes filter by session.user.id. But four tables have no userId column: Task, LifeGoal, GlobalSetting, SavedPaper (prisma/schema.prisma:498/518/69/13). Any authenticated stranger can read and mutate the owner's tasks, life goals, dash settings, and saved papers via /api/tasks, /api/goals, /api/settings, /api/research/saved — and burn Gemini/Gmail quota on the un-rate-limited /api/research/import and /api/applications/backfill.

Fix. Gate sign-in on an owner allowlist (mechanism: OQ1) and scope the four global tables per userId (scope: OQ2) — both decided in §7. Decision-free regardless: add checkUserRateLimit to research/import + backfill.

Next.js 16.1.1 pinned behind security fixes

HIGH · package.json:61 · next is exact-pinned at 16.1.1; npm audit reports a batch of advisories fixed in 16.2.x — RSC cache poisoning (GHSA-wfc6-r584-vfw7), XSS via CSP nonces, request smuggling in rewrites, SSRF via WebSocket upgrades, several DoS. The app is publicly reachable through the Cloudflare tunnel, so the cache-poisoning/XSS/DoS class is live. The middleware-bypass advisories do not apply (no middleware.ts — auth is route-level).

Fix: bump next + eslint-config-next to 16.2.9 (a minor within v16) and re-run the hermetic gate. Never run npm audit fix --force here — it would downgrade next-auth to v3.

Scraped-SVG XSS on the AI dash

HIGH · app/api/ai/llmleaderboard/route.ts:77 → components/cards/ai/LLMLeaderboardCard.tsx:135 · The leaderboard route scrapes lmarena.ai and returns raw SVG markup per org ($.html(svgNode)); the card injects it with dangerouslySetInnerHTML, 50 rows per render. innerHTML won't run <script>, but it does wire SVG event-handler attributes (onload=, onerror=, <animate onbegin=>) — a compromised or simply redesigned upstream page means arbitrary JS in an authenticated session whose cookies guard Gmail-scoped APIs.

Fix: sanitize server-side (DOMPurify with an SVG allowlist) before returning orgLogo, or map orgName → local logo assets and drop the scraped markup — strategy choice is OQ3. The other two dangerouslySetInnerHTML uses in the repo (app/layout.tsx:47, lib/resumes/templates/ats-plain.tsx:189) are static constants — fine.

Probe redirects are fetched before validation

MEDIUM · lib/postings/liveness.ts:186-214 + lib/security/url-guard.ts:124-127 · Liveness probes use redirect: "follow" and only validate res.url after the chain completes. A malicious or compromised posting host can 302 the probe into http://127.0.0.1:3101/… or the LAN — the GET is issued to the internal target; assertSafeResponseUrl merely refuses to read the body. Note requireLocalOrSession trusts localhost, so a redirect to localhost:3101 is an authenticated-equivalent GET. sourceUrl is partially third-party-controlled (scraped careers pages / ATS data).

Fix: switch to a manual-redirect loop that runs assertExternalHttpUrl on each Location before following (cap ~5 hops). DNS rebinding stays a documented out-of-scope residual.

No security response headers

MEDIUM · next.config.ts (absent) · Nothing sets X-Frame-Options/frame-ancestors, X-Content-Type-Options, Referrer-Policy, or any CSP — on an app serving authenticated pages with session cookies over a public tunnel. Clickjacking is live today.

Fix: add a headers() block in next.config.ts (X-Frame-Options: DENY, nosniff, Referrer-Policy: strict-origin-when-cross-origin) or set them at the Cloudflare layer; CSP optional given Next's inline scripts.

MC_LIVENESS_BYPASS unguarded in prod

MEDIUM · lib/postings/liveness.ts:458-462 · The hermetic-test override short-circuits all probes. If it ever leaks into the prod scheduler env (an ecosystem.config.cjs edit, an --update-env mishap), MC_LIVENESS_BYPASS=closed mass-closes every stale candidate and cascades kanban cards with zero network evidence. The comment says "production MUST NOT set this"; nothing enforces it.

Fix: refuse the bypass (loud warn, return null) when MC_SCHEDULER_TIER === "prod" or NODE_ENV === "production".

Correctness bugs

C3 re-probe sweep: inert at steady state + cursor jam

HIGH · scheduler/jobs/job-watcher.ts:581-650 · On feat/closed-jobs — fix before merging PR #1. Two compounding problems, verified in source:

  • Selection contradicts Gap A. The C3 comment says its purpose is closed-but-still-listed postings (rows the fetch keeps returning, so they never go stale). But the candidate filter excludes everything in seenExternalIds — i.e. exactly the still-listed rows. For aggregator kinds (LinkedIn/Indeed), "present in the search feed" ≠ "detail page still accepting applications"; that conflation is the gap C3 was built to close. As written, C3 can only probe rows absent from this fetch but <6h fresh — a sliver the stale probe reaches 6h later anyway.
  • Cursor jam. take: c3Budget happens before the seen/staleProbed filter, and lastProbedAt is stamped only on rows actually probed. Filtered-out rows keep lastProbedAt = NULL forever, so they permanently occupy the front of the ORDER BY lastProbedAt ASC window. At steady state (window full of currently-listed rows) c3ToProbe is empty every tick and the rotation never advances — C3 does literally nothing.

Fix: stamp lastProbedAt (or a separate cursor field) on skipped candidates too — or filter before take — and drop the seenExternalIds exclusion for aggregator kinds (keep it for first-party ATS kinds, where feed presence genuinely implies alive). Consider also stamping lastProbedAt in the stale-probe path so stale-probed-alive rows don't sit at the cursor front. (Cursor mechanics: OQ6; seen-exclusion scope: OQ7.)

Bot-wall redirects read as "closed" — amplified by the cascade

HIGH · lib/postings/liveness.ts:243 (LinkedIn), :260 (Indeed), :384-385 (Workday HTML fallback) · Flagged independently by two lanes; verified. probeLinkedin returns "closed" whenever the final URL lacks /jobs/view/ — and this check fires before the carefully-gated body-marker logic (which correctly returns "unknown" without positive alive evidence). LinkedIn routinely 302s guest scrapes to /authwall or /signup when bot-detecting — an HTTP 200 after follow, no 429 — which therefore reads as positive closure evidence. Same shape for Indeed (off /viewjob → closed) and the Workday fallback (/login redirect → closed).

Pre-cascade this only flipped a posting row; with Pillar C→A, a single bad verdict now auto-moves the linked INTERESTED kanban card to CLOSED — user-visible. This re-introduces the false-close class the 2026-05-25 probe gate was built to kill, through the one heuristic with no positive-evidence requirement:

flowchart LR
  probe["GET sourceUrl
(guest scrape)"] -->|"302 to authwall / login,
HTTP 200 after follow"| check["finalUrl lacks /jobs/view/"]
  check --> verdict["verdict: closed"]
  verdict --> flip["JobPosting.status = closed"]
  flip --> cascade["closeApplicationsForClosedPostings()"]

(user-visible kanban move)"]

Fix: classify redirect targets — authwall/login/signup/challenge/consent hosts+paths → "unknown"; only a redirect to the board root / jobs-search page counts as "closed". Optionally require two consecutive closed verdicts across ticks before flipping. (Classification: OQ4; confirmation policy: OQ5.)

Ingest side-effect checkpoints stamp on failure

HIGH · lib/applications/ingest.ts:489-520 + lib/calendar/sync.ts:55-147 + lib/repositories/applicationEvents.ts:70-92 · syncEventToGcal is internally best-effort: auth failure and API failure are caught and return null instead of throwing. Ingest's try { await syncEventToGcal(…); stamp gcalSyncedAt } catch {…} therefore always stamps the checkpoint — the catch is dead code. A transient Gcal outage or expired token permanently marks the interview event as mirrored, and the documented contract ("failure leaves the checkpoint null so the next ingest retries", PB-5 + CLAUDE.md) is broken: the calendar event is silently never created.

Same defeat for notifiedAt: maybeNotifyForApplicationEvent swallows dispatchNotification errors internally (console.warn only), so a failed OFFER/INTERVIEW notification is stamped as delivered and never retried. At-most-once still holds (dedupKey); at-least-once — the checkpoint's stated purpose — does not.

Fix: gate each stamp on the side-effect's return value (non-null gcal id / success boolean), or add a throwOnError mode for the ingest caller. (Mechanism: OQ8.)

Errored Gmail messages are permanently dropped

MEDIUM · app/api/gmail/webhook/route.ts:118-147 + lib/email-parser.ts:212-226 · When ingestGmailMessage returns {action:"errored"} (gmail.get failure, classifier failure), the loop counts it, advances lastSyncedHistoryId to the envelope id, and 200s. Pub/Sub redelivery is dedup-blocked by WebhookDelivery, and the next push resumes past the message. Compounding: parseApplicationEmail has no retry — unlike chatJSON's withRetry — so one transient Gemini 429 loses the email until a manual "Scan Inbox".

Fix: don't advance the watermark past the oldest errored message (or persist errored msgIds for a scheduler retry sweep), and give the email parser the same withRetry treatment as chatJSON. (Recovery policy: OQ9.)

dev + prod both mirror the same email to one Google Calendar

MEDIUM · lib/calendar/sync.ts:106-131 · Both tiers receive the same Gmail push, ingest into their own DBs, and mirror future events to the user's single primary calendar. The idempotency id is sha1(ApplicationEvent.id) — a per-tier cuid — so within-tier dedup cannot work across tiers: one interview email ⇒ two calendar events. Email has the EMAIL_ENABLED master switch; Gcal writes have no analog.

Fix: add a GCAL_SYNC_ENABLED master switch (0 in .env.development), or dedup via an extendedProperties query keyed on something cross-tier-stable (emailMsgId + kind). (OQ10.)

Scheduler: overlap, P2021 blindness, non-atomic cascade

MEDIUM · three related gaps:

  • scheduler/index.ts:216-241 — setInterval with no "still running" flag: a slow job-watcher tick (per-watchlist crawl + up to 30 probes × 1.5s + jitter) overlaps its next tick. The per-watchlist RUNNING map dedups same-watchlist work, but two interleaved loops defeat the fragile-source serial pacing and double external load. Three-line guard fixes it.
  • scheduler/index.ts:222-229 vs job-watcher.ts:793-799 — runDueWatchlists converts every per-watchlist throw into a RunResult.error, so a P2021 from inside processOneInner never reaches the scheduler's schema-behind disable handler → the exact every-10-minutes error spam the mechanism exists to prevent. Bonus: a thrown processOneInner never updates lastRunAt/lastError, so the watchlist retries every tick.
  • job-watcher.ts:521-539 + scheduler/index.ts:244-250 — posting close and cascade are separate statements, SIGTERM is an immediate process.exit(0) (every pm2 restart), and a cascade throw is swallowed. A posting can land status="closed" with its INTERESTED card never cascaded — and nothing ever revisits it (the helper is only called with just-closed ids). Fix: a cheap idempotent reconcile at the top of each run: Application.status="INTERESTED" AND posting.status="closed" → cascade.

Frontend: dash-restore race + render-phase store write

MEDIUM · two items:

Verified solid

Auth & guards

  • Route coverage is complete — every handler under app/api/** calls a guard (requireLocalOrSession, requireSession, session-or-service-token, or OIDC); the full route→guard table was built and no unguarded handler exists. Guards run outside withCache, so cache hits can't bypass auth.

Residual config dependency: no httpHostHeader origin override in cloudflared.

Open Questions

Unresolved decisions

Every finding above whose fix has more than one defensible shape gets an OQ here — one OQ = one decision, lettered options, mutually exclusive, pick exactly one. Findings with a single obvious fix (Next bump, MC_LIVENESS_BYPASS prod guard, security headers, unused-dep removal, manual-redirect SSRF loop, pre-push fail-on-missing, dead-code deletion) carry no OQ and wait in the fix list. Resolve all OQs before the fix list in the next section is authored — a checklist written against open forks bakes in unchosen decisions.

OQ1 — Where should the sign-in allowlist live? (→ §2.1)
  • (a) Env-var allowlist. The signIn callback returns profile.email === process.env.OWNER_EMAIL (or membership in a comma-separated ALLOWED_SIGNIN_EMAILS), which bounces every other Google account at the OAuth door on both tiers. Pros: one-line diff, no schema change, instantly effective, config sits with the other secrets in untracked .env. Cons: one more required env var (fresh-machine recovery must set it — same class as PUBSUB_SERVICE_ACCOUNT_EMAIL); adding a second account later means an env edit + restart.
  • (b) DB-driven allowlist. Only emails already present in User (or rows with an allowed flag) may sign in, bootstrapped from the existing owner row, which makes sign-in policy travel with the database instead of the environment. Pros: no new env var; future multi-account is a row insert, no restart. Cons: more code (bootstrap/first-user logic), and a wiped or restored-empty DB locks everyone out — a recovery footgun.

✓ Decided (2026-06-11): (a) — env-var allowlist in the signIn callback. The threat is "anyone on the internet can sign in", not account lifecycle; the one-line env check closes it immediately and matches how the webhook already gates its signer via env (PUBSUB_SERVICE_ACCOUNT_EMAIL).

OQ2 — Scope the four global tables (Task, LifeGoal, GlobalSetting, SavedPaper) per userId? (→ §2.1)
  • (a) Scope all four. Add userId (+ index) to each table, backfill existing rows to the owner, derive the id from the session in each route, and filter every repo query — so even an authenticated stranger sees only their own (empty) rows and the data model becomes genuinely multi-user-correct. Pros: defense-in-depth independent of OQ1; removes the four standing exceptions to the "everything is user-scoped" rule. Cons: one migration + four repos + their routes; GlobalSetting's single-row id='global' design (and the /api/settings If-Match optimistic-concurrency path) must be reworked to per-user rows.
  • (b) Scope three, keep GlobalSetting global. Task/LifeGoal/SavedPaper get userId; the settings row stays single-row global, protected only by the OQ1 allowlist. Pros: closes the personal-content exposure while avoiding the settings-row + concurrency rework; smaller diff. Cons: keeps one unscoped mutable surface — anyone who ever passes the allowlist can still rewrite dash order/hues (app config, not personal data).
  • (c) Don't scope; rely on OQ1 alone. No schema change; the allowlist is the single gate. Pros: zero migration work. Cons: single-layered defense; the review's HIGH finding stays mitigated rather than fixed.

OQ9 — Recovery policy for messages that error during webhook ingest? (→ §3.4)

What "errored" actually covers. ingestGmailMessage returns errored on a gmail.get failure (ingest.ts:120), a classifier failure (:154), or a rare P2002-recovery dead end (:437); the webhook also catches unexpected per-message throws (route.ts:131-137). Adding withRetry to parseApplicationEmail is decision-free and happens regardless, which shrinks the residual to: multi-minute Gemini outages, quota exhaustion, deterministic parser crashes on a specific email (poison), and DB-locked windows. Three facts from the code constrain the options: (1) the route flattens all history records and advances the watermark once, to the envelope id, after the loop (route.ts:117-152) — there is no per-record bookkeeping today; (2) a C4 stale-history fallback already exists: if the watermark is older than Gmail's retained history window (~7 days), history.list 404s and the route re-seeds the watermark to the envelope id, acks 200, and designates manual Scan Inbox as the recovery (route.ts:97-115); (3) the backfill (/api/applications/backfill) already re-runs the same idempotent ingest over a relevance-filtered 180-day window, capped at 200 messages — re-running it cannot duplicate anything.

  • (a) Hold the watermark. Track per-history-record success in the webhook loop and advance lastSyncedHistoryId only to the record before the oldest errored message; the resume logic already prefers the older of (checkpoint, envelope), so every subsequent push re-walks from the held point and re-attempts. Already-ingested messages re-walk as cheap checkpoint skips, but each walk re-pays a gmail.get per message past the hold point, and the errored message itself re-pays a full parse (a Gemini call × its retries) on every push until it succeeds. Pros: transient blips (minutes–hours) self-heal on the next push with no new tables; at-least-once holds within Gmail's retention window. Cons: the guarantee silently expires at the C4 horizon — hold past ~7 days and the existing 404 handler force-advances the watermark, abandoning the held range to manual backfill anyway, so (a) converges to (c) with extra moving parts; needs the per-record bookkeeping the loop doesn't have; a poison message means repeated parse spend + added latency on every push for up to a week; dev and prod wedge independently (separate watermarks).
  • (b) Errored-message retry queue. New table (e.g. FailedIngest(msgId @id, userId, firstFailedAt, attempts, lastError)) written on an errored outcome; a scheduler job (the registry already runs 11) retries each row through the same idempotent ingest with backoff (15m → 1h → 6h, cap ~5 attempts), dispatches one dedupKey'd notification on final give-up, drops the row on a gmail.get 404 (email deleted), and prunes like WebhookDelivery (30d). The watermark advances exactly as today. Pros: poison handled structurally (attempt cap + notify, no repeated webhook-path spend); webhook latency stays flat; failures become a visible inventory (surfaceable on the Internal Systems dash); immune to the 7-day horizon since retry is by msgId, not historyId. Cons: the most machinery — migration + table + scheduler job + smoke; a second ingest entry point to keep consistent.
  • (c) Accept the drop, loudly. Keep today's watermark behavior; on an errored outcome dispatch a dedupKey'd in-app notification ("N emails failed ingest today — run Scan Inbox to recover") instead of today's ring-buffer console.warn; recovery is the existing one-click idempotent backfill. Pros: near-zero new machinery; consistent with the system's own design — the C4 path already names Scan Inbox as the recovery for the analogous history-gap case; a failure burst (Gemini outage) becomes one counted notification rather than silence. Cons: at-least-once stays unmet between the failure and the user clicking; edge: backfill finds messages via the relevance-shaped Gmail query (buildGmailQuery), so a failed message that query wouldn't match isn't re-found (rare — the query approximates the same relevance gate the webhook path applies).

✓ Decided (2026-06-12): (b) — errored-message retry queue, with the schedule tuned for minimal Gemini spend per Sal's requirement ("the solution which calls Gemini the least and retries as little as possible automatically"): exponential ×4 backoff (5 m → 20 m → ~1.3 h → ~5.3 h → ~21 h → ~3.6 d), attempt cap ~6 — cumulatively ~4.7 days, comfortably inside the 7-day ceiling, each scheduled attempt paying exactly one parse call (no inner withRetry on queue retries — the backoff is the retry). Every failed attempt sounds an alarm — alarm side built now, delivery deferred. Each failure emits a per-attempt alarm event carrying which attempt just failed and at what time the next retry is scheduled ("Email ingest failed (attempt 3/6) — retrying ~14:05"; the final one announces give-up + Scan Inbox as manual recovery). For now only the alarm side is built — the queue row carries nextRetryAt/attempts/lastError and each failure logs a structured warn — and it deliberately does not plug into the notification dispatcher yet (per Sal 2026-06-12: the notification system hasn't been refactored into a robust feature; hold back). Routing these alarms into notifications/the frontend is a noted follow-up once that refactor lands. Worst case per failed message: ≤7 parse calls, ever. Chosen over (a) because (a)'s retry cadence is coupled to incoming pushes — it re-pays a parse on every new email, uncontrollably, then silently expires at the C4 horizon; the queue is the only option whose retry timing we control, making it the minimal-automatic-token design that still recovers without manual action.
