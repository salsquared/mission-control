Gmail real-time push — wiring plan

The Applications dash claims it "auto-syncs via Gmail & Pub/Sub", but new mail only lands when you press Scan Inbox. The receive-side plumbing (OIDC verify → dedup → history.list resume → ingest) is fully built and has never run. This plan wires the two missing ends — registering the Gmail watch() and keeping it alive — so arrival → kanban becomes real-time. Plan only; no code is changed yet.

Diagnosis — what's actually broken

Evidence

This is not a watch that expired after 7 days — the real-time path was never wired end-to-end. Four independent checks, all confirming the webhook has never fired even once:

CheckResultMeaning
gmail.users.watch() calls in app code0 (only a comment in a debug script)Nothing ever tells Gmail to push notifications
WebhookDelivery rows in prod.db0The webhook's first action (INSERT) has never executed
user.lastSyncedHistoryIdnullThe checkpoint the webhook advances was never set
[GMAIL WEBHOOK] lines in prod logs0 ever (vs. 4 [BACKFILL])Log capture works; only the manual scan has ever run
PUBSUB_AUDIENCE env varMISSINGEven a delivered push would 500 at webhook/route.ts:18 before any work
Note. The UI copy in ApplicationsView.tsx:282 ("Auto-syncs via Gmail & Pub/Sub API") and the login screen's "live Pub/Sub polling" are aspirational. The only ingestion path that has ever run is the manual POST /api/applications/backfill behind the Scan Inbox button — which scans the last 180 days, capped at 200 messages.

The two missing ends

The webhook (app/api/gmail/webhook/route.ts) and the ingest pipeline (lib/applications/ingest.ts) are complete and correct. What's absent:

  • Registration — nobody calls gmail.users.watch(), so Gmail has no Pub/Sub topic to publish change notifications to.
  • Renewal — Gmail watch subscriptions expire after 7 days. Even a one-time manual watch() would go silent a week later. There is no renewal job.
  • Receive config — PUBSUB_AUDIENCE and PUBSUB_SERVICE_ACCOUNT_EMAIL are unset, so the webhook can't accept a push. (PUBSUB_WEBHOOK_SECRET is set but the current OIDC-based route never reads it — dead env from an older design.)
  • Public reachability — Pub/Sub push needs a public HTTPS endpoint. The webhook must be served on the cloudflared public hostname and must not sit behind Cloudflare Access (the route does its own OIDC verification — it deliberately does not use requireLocalOrSession).

Target architecture

End-to-end flow

Once wired, a new email triggers a push within seconds; the webhook resumes from the per-user watermark and replays the existing ingest. The scheduler keeps the 7-day watch alive with a daily re-arm (7× safety margin).

Both tiers, one mailbox — the fan-out

dev.db and prod.db are both linked to the same Google account (salsalcedo4321@gmail.com, refresh token present in both). Gmail permits exactly one active watch() per mailbox — calling watch() again replaces the prior registration (that's how renewal works). So dev and prod cannot each point at their own topic; the daily renewals would clobber each other and only the last-armed tier would receive pushes.

The standard fix is one topic, two push subscriptions. Gmail publishes each change once to the single topic; Pub/Sub fans it out to both tier webhooks. Each tier walks history from its own per-DB lastSyncedHistoryId and ingests into its own DB. Dev's EMAIL_ENABLED=0 keeps notification emails from firing twice.

This doubles email LLM classification (each tier parses the same email once). That's a deliberate, separate concern — see docs/cross-tier-llm-dedup.html, which makes Gemini fire once per unique input across both tiers. The two efforts are parallel-safe; until dedup lands, the 2× email cost is harmless and temporary.

What stays untouched

The webhook body, dedup, history.list resume logic, per-event checkpoints, and the whole ingest pipeline are unchanged — they were built correctly for a push that never came. The Scan Inbox backfill also stays as the manual catch-up / disaster-recovery path.

Code changes (the half I build)

C1 — lib/gmail/watch.ts (new): the registration helper

One small module. Calls users.watch() against the configured topic, scoped to the INBOX label (application mail lands there). Seeds lastSyncedHistoryId only when it's null — on a renewal the webhook owns that watermark, and overwriting it would skip events between the last checkpoint and "now". No-ops cleanly when the topic env var is unset (so dev never arms a watch on the shared Google account).

import { google } from "googleapis";
import { getGoogleAuthClient } from "@/lib/googleapis";
import { prisma } from "@/lib/prisma";

// Set in .env (SHARED by both tiers) — dev and prod arm the SAME topic, so the
// renewals are idempotent rather than clobbering each other (Gmail allows one
// watch per mailbox; see §2.2). Format: projects/<project>/topics/<topic>
const TOPIC = process.env.GMAIL_PUBSUB_TOPIC;

export async function registerGmailWatch(userId: string) {
  if (!TOPIC) {
    console.warn("[gmail-watch] GMAIL_PUBSUB_TOPIC unset — skipping watch registration");
    return null;
  }
  const auth = await getGoogleAuthClient(userId);
  const gmail = google.gmail({ version: "v1", auth });
  const res = await gmail.users.watch({
    userId: "me",
    requestBody: {
      topicName: TOPIC,
      labelIds: ["INBOX"],
      labelFilterBehavior: "INCLUDE",
    },
  });
  const historyId = String(res.data.historyId);
  const expiration = String(res.data.expiration); // ms-epoch, ~7d out

  // Seed the watermark ONLY on first arm. On renewal the webhook is the owner;
  // clobbering it with "now" would drop anything between checkpoint and now.
  const u = await prisma.user.findUnique({
    where: { id: userId },
    select: { lastSyncedHistoryId: true },
  });
  if (u && !u.lastSyncedHistoryId) {
    await prisma.user.update({ where: { id: userId }, data: { lastSyncedHistoryId: historyId } });
  }
  console.info(`[gmail-watch] armed user=${userId} historyId=${historyId} exp=${expiration}`);
  return { historyId, expiration };
}
Scope. users.watch() works with the existing gmail.readonly scope already granted in lib/auth.ts — no re-consent needed.

C2 — scheduler/jobs/gmail-watch-renew.ts (new): the daily re-arm

Modeled on the existing stateless sweeps (e.g. webhook-delivery-prune.ts). Finds every user with a linked Google account that has a refresh token and re-arms each. Idempotent — calling watch() again just resets the 7-day clock against the same topic.

import { prisma } from "@/lib/prisma";
import { registerGmailWatch } from "@/lib/gmail/watch";

export interface GmailWatchRenewResult { processed: number; renewed: number; failed: number; }

export async function runGmailWatchRenew(): Promise<GmailWatchRenewResult> {
  const accounts = await prisma.account.findMany({
    where: { provider: "google", refresh_token: { not: null } },
    select: { userId: true },
  });
  const userIds = [...new Set(accounts.map((a) => a.userId))];
  let renewed = 0, failed = 0;
  for (const userId of userIds) {
    try {
      if (await registerGmailWatch(userId)) renewed++;
    } catch (e) {
      failed++;
      console.warn(`[gmail-watch-renew] failed for user ${userId}:`, (e as Error).message);
    }
  }
  return { processed: userIds.length, renewed, failed };
}

Register it in scheduler/index.ts at a 24h interval. The existing startup-stagger (setTimeout(tick, 10_000 * (i+1))) means it also re-arms within ~seconds of every scheduler restart — so a deploy or reboot self-heals the watch:

{
  name: 'gmail-watch-renew',
  intervalMs: 24 * 60 * 60 * 1000, // daily — Gmail watch expires in 7d (7x margin)
  run: async () => {
    const r = await runGmailWatchRenew();
    if (r.processed > 0) {
      console.info(`[gmail-watch-renew] ${r.renewed}/${r.processed} re-armed, ${r.failed} failed`);
    }
  },
},
Both tiers arm the same topic. GMAIL_PUBSUB_TOPIC lives in .env (shared), so the dev and prod schedulers both re-arm the same watch target every day — idempotent, no clobber. This is the only safe shape because dev.db and prod.db are linked to the same mailbox and Gmail permits one watch per mailbox (see §2.2). Verify both scheduler processes see the topic var (PM2 env-file loading) — see §4.

C3 — sign-in trigger in lib/auth.ts (best-effort)

After the signIn callback persists fresh tokens, fire a one-shot arm so connecting (or re-consenting) an account is live immediately instead of waiting for the next daily tick. Wrapped so a watch failure never blocks sign-in. This is belt-and-suspenders; the scheduler is the primary keeper.

// inside signIn({ account }), after the existing token-persist block:
try {
  const acct = await prisma.account.findFirst({
    where: { provider: "google", providerAccountId: account.providerAccountId },
    select: { userId: true },
  });
  if (acct) await registerGmailWatch(acct.userId);
} catch (e) {
  console.warn("[auth] gmail watch on sign-in failed:", (e as Error).message);
}
return true;

C4 — webhook 404 recovery (optional hardening, phase 2)

If the webhook is down > 7 days, lastSyncedHistoryId goes older than Gmail's history window and history.list returns 404. Today that surfaces as a 500. Add a catch: on 404, re-seed lastSyncedHistoryId from the envelope's historyId and return 200 (the next push catches up; the manual backfill recovers the gap). Not required for first light — call it out as a follow-up.

C5 — what does NOT change

  • No schema migration. Stateless daily renewal needs no new column. (Considered + rejected: a gmailWatchExpiration field to renew only near expiry — pointless micro-optimization for a single-user system; daily re-arm is simpler and self-healing.)
  • No webhook body changes (beyond optional C4) — it already resumes from the watermark, dedups, and ingests correctly.
  • No new Gemini callsite — so no Lunary/Promptfoo/docs/llm-calls.html work.

C6 — tests

  • Hermetic smoke for runGmailWatchRenew's DB selection + error isolation, with registerGmailWatch stubbed (no live Gmail). Append to the SUITES array in scripts/pre-push.sh.
  • Live probe under scripts/tests/probes/ that calls registerGmailWatch against the real account and prints the returned historyId/expiration — diagnostic only, exit-zero not a contract.

Google Cloud + env (the half you do)

G1 — Pub/Sub topic + Gmail publish grant

In the same GCP project as your OAuth client:

# 1. Create the topic
gcloud pubsub topics create gmail-push

# 2. Let Gmail's system service account publish change notifications to it
gcloud pubsub topics add-iam-policy-binding gmail-push \
  --member="serviceAccount:gmail-api-push@system.gserviceaccount.com" \
  --role="roles/pubsub.publisher"

G2 — Two push subscriptions (one per tier) → public webhook URLs

Both tiers receive from the same topic via separate subscriptions. Push needs public HTTPS endpoints, so each is a cloudflared hostname (not mc.local — that's LAN-only and unreachable by Pub/Sub). Pick (or create) a service account for Pub/Sub to sign the OIDC token with (e.g. pubsub-mc-publisher@<project>.iam.gserviceaccount.com):

# prod subscription
gcloud pubsub subscriptions create gmail-push-prod \
  --topic=gmail-push \
  --push-endpoint="https://ms-prod.salsquared.xyz/api/gmail/webhook" \
  --push-auth-service-account="pubsub-mc-publisher@<project>.iam.gserviceaccount.com" \
  --push-auth-token-audience="https://ms-prod.salsquared.xyz/api/gmail/webhook"

# dev subscription (ms-dev.salsquared.xyz already routes to :4101)
gcloud pubsub subscriptions create gmail-push-dev \
  --topic=gmail-push \
  --push-endpoint="https://ms-dev.salsquared.xyz/api/gmail/webhook" \
  --push-auth-service-account="pubsub-mc-publisher@<project>.iam.gserviceaccount.com" \
  --push-auth-token-audience="https://ms-dev.salsquared.xyz/api/gmail/webhook"
The audience must match the env var byte-for-byte, per tier. Each subscription's --push-auth-token-audience must equal that tier's PUBSUB_AUDIENCE exactly (scheme, host, path, no trailing slash). A mismatch → every push 401s.
Don't gate either webhook path behind Cloudflare Access. If the tunnel applies an Access policy to /api/gmail/webhook on either host, Pub/Sub can't reach it. The route authenticates itself via the OIDC JWT — leave it open at the edge.

G3 — env vars: topic shared, audience per-tier

The topic is identical for both tiers (shared .env); only the audience differs (each tier's own public host, in its tier env file):

# .env  (shared by both tiers)
GMAIL_PUBSUB_TOPIC=projects/<project>/topics/gmail-push
PUBSUB_SERVICE_ACCOUNT_EMAIL=pubsub-mc-publisher@<project>.iam.gserviceaccount.com
# PUBSUB_WEBHOOK_SECRET — remove; unused by the OIDC route

# .env.production
PUBSUB_AUDIENCE=https://ms-prod.salsquared.xyz/api/gmail/webhook

# .env.development
PUBSUB_AUDIENCE=https://ms-dev.salsquared.xyz/api/gmail/webhook

Then restart all four processes so both web tiers (webhooks) and both schedulers (renewal) pick them up:

pm2 restart mission-control mission-control-dev \
  mission-control-scheduler-prod mission-control-scheduler-dev --update-env
Verify each scheduler sees the topic. The renewal job runs in both mission-control-scheduler-dev and -prod tsx processes — confirm GMAIL_PUBSUB_TOPIC + the Google OAuth client vars are visible in each (PM2 env-file loading), or that tier's daily re-arm silently no-ops.

Verification & rollback

First-light checklist (per tier — verify dev first)

  1. After restart, each scheduler's startup tick (~10s) logs [gmail-watch] armed … with a historyId + expiration ~7 days out.
  2. user.lastSyncedHistoryId in that tier's DB flips from null to a real value.
  3. Send yourself an application-shaped test email. Within seconds: a [GMAIL WEBHOOK] log line appears on both tiers (the topic fans out to both subscriptions), a WebhookDelivery row is created in each DB, and both kanbans update without pressing Scan Inbox.
  4. Confirm only one notification email is sent (prod), not two — dev's EMAIL_ENABLED=0 mutes it.
  5. Watching the same email a second time (Pub/Sub at-least-once) logs dedup hit and returns 200 — no double-ingest.

Rollback

Fully reversible and low-blast-radius, per tier: delete that tier's Pub/Sub subscription (its pushes stop) — or unset GMAIL_PUBSUB_TOPIC in shared .env to disarm both (renewals no-op, the watch lapses in ≤7d). The manual Scan Inbox path is exactly what it is today. No data migration to undo. To kill only dev pushes while keeping prod live, just delete the gmail-push-dev subscription.

Schema migrations & the dual-DB

This feature needs no migration — the renewal job reads only the long-stable Account + User tables, and the Gmail watch() registration is entirely DB-independent. For future migrations generally, the existing dual-DB discipline already covers it:

  • Per-DB apply. prisma migrate dev against dev.db, then DATABASE_URL="file:./prod.db" npx prisma migrate deploy for prod, then pm2 restart the affected processes. dev and prod migrate independently — exactly why "works in both" is testable on dev first.
  • Graceful lag. If a future migration adds a column the watch-renew job reads and one tier is behind, the scheduler's existing P2021 guard disables just that job on the lagging tier (one loud warning) until you migrate-deploy + restart. During that window the watch lapses within ≤7d and that tier falls back to the manual scan — no crash loop, and the other tier is unaffected.
  • Watch survives DB lag. A schema lag only ever affects the ingest write on that tier, never the Gmail-side subscription — pushes keep arriving; a bad ingest is caught per-message and logged.
