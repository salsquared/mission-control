Context & problem
Design-doc convention (delete before publishing). Sections follow the decision-making flow and the tail order is fixed: context → design → risks & rollback → Open Questions (second-to-last) → Tasks (last, right before the footer). Open Questions are OQ-numbered (OQ1, OQ2… — never Q) and must be resolved with the user before any implementation code is written. The Tasks checklist needs ≥2 nested levels (phase → task → sub-task, e.g. P1.2.3) — never flat.

Problem

What's broken, missing, or painful today — grounded in the current code/state (cite files + line numbers, not vibes). Why is this worth doing now? What's the cost of not doing it?

Goals & non-goals

What this doc commits to delivering, and — just as important — what it explicitly does not (deferred to a later phase, or out of scope entirely). Naming non-goals up front keeps the design honest and the Tasks list bounded.

  • Goal: <the outcome that defines done>
  • Non-goal: <the tempting-but-deferred thing> — deferred to <when / why>

Design

Approach

The chosen design. Start from the existing code's grain — what abstraction/pattern already exists that this extends (e.g. the per-tier SQLite house pattern, the withCache wrapper, the fetcher registry)? Describe data model, API surface, and control flow. Use a diagram for anything with more than ~3 moving parts:

flowchart TB
  trigger["entry point"] -->|"step"| work["does the thing"]
  work --> store["persists / emits"]
  store --> done(["done"])

Note the touch list: every file/module this changes (schema, Zod, repository, route, UI, scheduler job, tests, generated docs). The reviewer should be able to predict the diff from this section.

Alternatives considered

The other viable designs and why they lost. This is where "minimal v1" lives — for each component, state why it's load-bearing or why the heavier alternative was deferred. A rejected alternative with a clear reason often becomes an Open Question or a non-goal.

Risks & rollback

Risks & mitigations

What could break — data loss, race conditions, log-viewer spam, prod/dev divergence, schema-behind tiers, external-API bursts. For each, the mitigation already in the design.

Rollback

How to back this out if it misbehaves in prod — feature flag, revert + rebuild + pm2 restart, migration-down, or "best-effort so it degrades to a no-op." If a migration is involved, note the backup step and that both DBs must be handled.

Open Questions

Unresolved decisions

Genuine design forks not yet settled — pulled from real conditional choices in the plan (scope cuts, deferred abstractions, "verify load-bearing first" items). Don't pad with fake questions. Each is OQ<n>: put the question alone in the highlighted callout banner, then a following <div class="oq-detail"> holding the options as a <ul class="oq-options"> list. Letter every option — lead each <li> with <strong>(a)</strong>, (b), … so each choice is nameable (OQ<n>a, OQ<n>b) and one bulleted line reads far better than an inline "A vs B" sentence. The trailing Lean: line (and, once settled, the ✓ Decided. line) must name the pick by its letter first — e.g. Lean: (b) — then say what that means and give at least one sentence of justification. Keep the banner to the question, not the discussion. Resolve all OQs with the user before writing implementation code. Deciding an OQ appends, never replaces: keep the lettered oq-options list under the question (the record of what was weighed), swap the trailing Lean:/Status: line for the ✓ Decided. line (still leading with the chosen letter), and flip the banner to callout tip.
