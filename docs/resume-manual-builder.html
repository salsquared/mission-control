Manual Resume Builder

Replace the keyword-scoring auto-selector with a hand-curated, per-Canon bullet selection — so a Security-Officer job never leaks onto an ML/AI resume, because it's structurally absent from that Canon, not just unlucky in a score.

Context & problem

Problem

The resume generator picks which bullets go on a tailored resume by counting posting-keyword overlap. lib/resumes/select.ts:scoreBullet awards TAG_WEIGHT = 2 per matched tag and SUBSTRING_WEIGHT = 1 per keyword found in the bullet text; the entity's aggregate score is the sum over its bullets. This has a fundamental ceiling: keyword overlap cannot distinguish domain-relevant experience from domain-irrelevant experience.

Concretely, an ML/AI Engineering posting is dense with soft words — communication, collaboration, data, reporting, scheduling, problem-solving. A Security-Officer bullet like “Coordinated radio communication and filed daily incident reports” matches several of them, scores nonzero, and is kept. Two more rules push irrelevant experience onto the page regardless of relevance:

  • The work-role “spine” is always kept (pickWorkRoleSpineIndex + dropZeroScoreEntities in select.ts) so the Experience section is never empty — which force-includes the most-recent job even at zero score.
  • All Education is always kept regardless of score.

The net effect is the user's actual complaint: security-guard experience appears on an ML/AI Engineering resume. The pipeline is elaborate — auto-tag → score → LLM section/entity ordering (resume-tagline callsite) → Flash bullet rewrite → two-phase one-page pruner (lib/resumes/one-page.ts) — yet it is wrong at the one thing that defines a good resume: relevance of what's included. No prompt-tuning fixes a keyword-matching ceiling. Relevance is a human judgement, and the user wants to make it.

Good news on blast radius: only the selection layer is broken. The renderer consumes a tidy document shape (ResumeProps in lib/resumes/templates/ats-plain.tsx: { profile, tagline, sections{workRoles,projects,education}, extras{skills,languages,hobbies}, sectionOrder }) and turns it into PDF/DOCX. The fixed-selection plumbing already exists for Canons — lib/canons/specialize.ts:reconstructSelection rebuilds a nested ResumeSelection from a flat stored selection + a frozen profile snapshot, dropping entities that no longer exist. We swap the brain (how the selection is decided) and reuse the body (resolve → optional rewrite/tagline → compose → render → persist).

Goals & non-goals

  • Goal: a manual builder that shows the user's entire profile (work roles, projects, education, and skills/languages/interests) and lets them hand-pick what goes on a resume.
  • Goal: the selection is saved per-Canon and re-renderable — “ML Engineer” and “Security Officer” are separate curated selections.
  • Goal: a per-entity lock state (always-in / out) so relevance is a property you set, not something the algorithm re-decides each run.
  • Goal: AI rewrite + tagline survive as opt-in toggles (default off → verbatim render); the old scoring pipeline survives as a “Suggest from posting” pre-fill button.
  • Goal: reuse the existing render / specialize / persistence / staleness / versioning machinery unchanged.
  • Non-goal: drag-to-reorder of sections/entities/bullets — deferred (OQ8); v1 inherits profile order.
  • Non-goal: a live in-builder page-count preview — deferred (OQ9); v1 reports page count after render.
  • Non-goal: deprecating or removing the one-off URL/paste generator (GenerateResumeCard) — deferred (OQ12).
  • Non-goal: cover letters, multi-template, per-company personalization — out of scope.

Design

Approach

Extend the Canon. Today a Canon owns one reusable resume whose fixed bullet set lives implicitly inside the latest GeneratedResume.selections (a flat JSON array), produced by the auto-pipeline at “Regenerate” time. We make that selection a first-class, user-editable definition on the Canon — a new Canon.selection JSON field (OQ6) — and replace “Regenerate”'s auto-pipeline with a manual editor. Generate then becomes the specialize flow almost verbatim: resolve the saved selection against the live profile → (optionally) rewrite + tagline → compose → render → persist.

Data model

The Canon.selection JSON (parsed by the repository, like resumeEntityIds already is). Resolved best-effort against the live profile at render — a referenced entity or bullet that's since been deleted is silently dropped, exactly as reconstructSelection does today.

{
  "version": 1,
  "sectionOrder": ["experience", "projects", "education", "skills", "languages", "interests"],
  "sectionsOff": ["interests"],          // section types the user toggled entirely off
  "entities": {
    "<workRoleId>": { "kind": "workRole", "state": "always", "bulletIds": ["b1", "b3"] },
    "<projectId>":  { "kind": "project",  "state": "auto",   "bulletIds": ["b7"] },
    "<guardRoleId>": { "kind": "workRole", "state": "never",  "bulletIds": [] }
  },
  "extras": { "skillItems": ["PyTorch", "..."], "languages": ["Spanish"], "hobbies": [] }
}

Per-entity state (OQ5) is the crux of the fix — three values:

  • always (🔒) — always rendered; “Suggest from posting” cannot remove it.
  • auto (default) — eligible; “Suggest from posting” may toggle it in/out based on score.
  • never (🚫) — never rendered; “Suggest” must respect this and never re-add it.

The Security-Officer job set to never on the ML Canon is the structural fix: no re-score, no spine rule, no “Suggest” pass can put it back. Within an included entity, bulletIds is the ordered set of bullets to render (OQ7 — per-bullet granularity).

Alternatives considered

  • Per-bullet global locked/excluded flags (already exist). Rejected: those are global — they affect every resume. The whole point is “include this job on my Security resume but never on my ML one,” which is inherently per-Canon. Manual selection lives at the Canon, not the bullet.
  • Relational CanonSelectionEntry table. Rejected for v1: the stored selection is already JSON (GeneratedResume.selections), the resolve-against-live-profile pattern already exists, and a JSON column matches the house pattern with no join model to maintain. Revisit only if we need to query selections across canons.
  • Keep the auto-pipeline and bolt on a manual exclude list. Rejected: doesn't deliver the “show everything, pick what you want” UX the user asked for, and leaves the spine/soft-keyword leaks intact for everything not explicitly excluded.
  • Drag-reorder + live page preview in v1. Deferred (OQ8/OQ9): real work, not load-bearing for the relevance fix. v1 inherits profile order and reports page count after render.

Rollback

Canon.selection is nullable and additive, and the manual generate path is gated on its presence — so backing out the route branch makes every canon fall through to the existing auto-pipeline (degrades to today's behavior, a no-op). To revert: drop the route branch + overlay, revert + npm run build + pm2 restart mission-control. The column can stay (harmless) or be removed with a down-migration after a prod.db backup. No generated artifacts or existing GeneratedResume rows are mutated by this feature.

Open Questions

Unresolved decisions

The four forks below the line are decided (this session's design conversation). OQ5–OQ13 are open; each carries a Lean. Resolve all open OQs with the user before writing implementation code. OQ5/OQ6/OQ7 define the data model and gate Phase 1, so settle those first.

OQ1 — What does the AI still do on the manually-picked bullets?
✓ Decided: manual selection is the core; rewrite + tagline are opt-in toggles, default off → deterministic verbatim render. The selection problem is separate from wording, so wording stays optional polish.
OQ2 — What does “locked aka always there” mean?
✓ Decided: lock is per-entity (🔒 ML Engineer always, Security Guard never), not per-section-type. Relevance is a state you set on an entity.
OQ3 — Is a manual selection saved and reusable?
✓ Decided: saved per-Canon, re-renderable. Canons already model the user's job-search lanes; “ML resume vs security resume” lives there.
OQ4 — What happens to the existing auto-scoring pipeline?
✓ Decided: it survives as an optional “Suggest from posting” pre-fill button — a starting guess the user edits, not the decider.
OQ5 — Entity state cardinality: three states or two?
Options: (A) three — always / auto / never; or (B) two — in / out, where “Suggest” may re-propose any out entity. Lean: A. The third state (never) is what stops “Suggest from posting” from re-adding the Security-Officer job every run; with B the user re-excludes it forever. Status: open.
OQ6 — Where is the selection stored?
Options: (A) a single nullable JSON column Canon.selection; (B) a relational CanonSelectionEntry table; (C) reuse the existing deferred Canon.pinnedEntityIds/sectionOrder fields. Lean: A — matches the house JSON-selection pattern, resolved best-effort against the live profile; no join model. Status: open.
OQ7 — Bullet granularity in v1: per-bullet, or entity-level only?
Options: (A) per-bullet checkboxes within an included entity; (B) include an entity ⇒ all its bullets render. Lean: A — picking which bullets show is core to a good resume and cheap given the data is already there. Status: open.
OQ8 — Ordering control in v1?
Options: (A) inherit profile position order + a fixed DEFAULT_SECTION_ORDER; (B) drag-reorder sections/entities/bullets. Lean: A for v1, defer B. Status: open.
OQ9 — Length / one-page handling?
Options: (A) drop the auto one-page pruner in manual mode (the user owns length) and report page count after render; (B) keep the auto-pruner; (C) live in-builder page preview. Lean: A — auto-pruning contradicts “manual” (it would silently drop the user's picks). Defer C. Status: open.
OQ10 — UI surface for the builder?
Options: (A) a full-screen Window-class overlay launched from the Canon row; (B) a new dash/view; (C) inline expansion inside CanonsCard. Lean: A — the full profile tree is too big for a card; an overlay is the established escape-the-grid pattern. Status: open.
OQ11 — “Suggest from posting” input?
Options: (A) the Canon's own keywords; (B) a pasted/linked specific posting; (C) both. Lean: C — default to canon keywords, allow pasting a specific posting to bias the suggestion. Status: open.
OQ12 — Fate of the one-off generator (GenerateResumeCard URL/paste/pipeline)?
Options: (A) keep it as the quick non-canon throwaway path; (B) fold it into the builder; (C) deprecate it. Lean: A for now — the canon builder is the curated path; revisit once the builder lands. Status: open.
OQ13 — Rewrite scope when the toggle is on?
Options: (A) keep today's behavior — rewrite only posting-matching bullets, pass no-match bullets through verbatim (rewrite.ts pre-filter); (B) rewrite every included bullet. Lean: A — avoids cross-domain invention and saves tokens; tagline still needs a posting/keywords. Status: open.
