Manual Resume Builder

Retire the keyword-scoring auto-selector and replace it with a hand-curated, per-Canon bullet selection — so a Security-Officer job never leaks onto an ML/AI resume, because it's structurally absent from that Canon, not just unlucky in a score.

Context & problem

Problem

The resume generator picks which bullets go on a tailored resume by counting posting-keyword overlap. lib/resumes/select.ts:scoreBullet awards TAG_WEIGHT = 2 per matched tag and SUBSTRING_WEIGHT = 1 per keyword found in the bullet text; the entity's aggregate score is the sum over its bullets. This has a fundamental ceiling: keyword overlap cannot distinguish domain-relevant experience from domain-irrelevant experience.

Concretely, an ML/AI Engineering posting is dense with soft words — communication, collaboration, data, reporting, scheduling, problem-solving. A Security-Officer bullet like “Coordinated radio communication and filed daily incident reports” matches several of them, scores nonzero, and is kept. Two more rules push irrelevant experience onto the page regardless of relevance:

  • The work-role “spine” is always kept (pickWorkRoleSpineIndex + dropZeroScoreEntities in select.ts) so the Experience section is never empty — which force-includes the most-recent job even at zero score.
  • All Education is always kept regardless of score.

The net effect is the user's actual complaint: security-guard experience appears on an ML/AI Engineering resume. The pipeline is elaborate — auto-tag → score → LLM section/entity ordering (resume-tagline callsite) → Flash bullet rewrite → two-phase one-page pruner (lib/resumes/one-page.ts) — yet it is wrong at the one thing that defines a good resume: relevance of what's included. No prompt-tuning fixes a keyword-matching ceiling. Relevance is a human judgement, and the user wants to make it.

Good news on blast radius: only the selection layer is broken. The renderer consumes a tidy document shape (ResumeProps in lib/resumes/templates/ats-plain.tsx: { profile, tagline, sections{workRoles,projects,education}, extras{skills,languages,hobbies}, sectionOrder }) and turns it into PDF/DOCX. The fixed-selection plumbing already exists for Canons — lib/canons/specialize.ts:reconstructSelection rebuilds a nested ResumeSelection from a flat stored selection + a frozen profile snapshot, dropping entities that no longer exist. We swap the brain (how the selection is decided) and reuse the body (resolve → optional rewrite/tagline → compose → render → persist).

Goals & non-goals

  • Goal: a manual builder that shows the user's entire profile (work roles, projects, education, and skills/languages/interests) and lets them hand-pick what goes on a resume — per-entity and per-bullet.
  • Goal: the selection is saved per-Canon and re-renderable — “ML Engineer” and “Security Officer” are separate curated selections.
  • Goal: no automated selection at all — the keyword scorer is retired from the selection path (OQ4). What's on the resume is exactly what the user checked; nothing is auto-decided, scored, or force-kept.
  • Goal: AI rewrite + tagline survive as opt-in toggles (default off → verbatim render).
  • Goal: a live page-count indicator in the builder (OQ9) so the user owns length deliberately.
  • Goal: reuse the existing render / specialize / persistence / staleness / versioning machinery unchanged.
  • Non-goal: any “Suggest” / auto-prefill in v1 — deferred (OQ4); if it returns it'll be an LLM relevance pass, not the keyword scorer.
  • Non-goal: drag-to-reorder of sections/entities/bullets — deferred (OQ8); v1 inherits profile order.
  • Non-goal: deprecating or removing the one-off URL/paste generator (GenerateResumeCard) — deferred (OQ12).
  • Non-goal: cover letters, multi-template, per-company personalization — out of scope.

Design

Approach

Extend the Canon. Today a Canon owns one reusable resume whose fixed bullet set lives implicitly inside the latest GeneratedResume.selections (a flat JSON array), produced by the auto-pipeline at “Regenerate” time. We make that selection a first-class, user-editable definition on the Canon — a new Canon.selection JSON field (OQ6) — and replace “Regenerate”'s auto-pipeline with a manual editor. Generate then becomes the specialize flow almost verbatim: resolve the saved selection against the live profile → (optionally) rewrite + tagline → compose → render → persist. The manual path never calls selectBullets, so the spine rule, the all-education-kept rule, and the one-page pruner simply don't run — the resume is exactly the checked set.

Data model

The Canon.selection JSON (parsed by the repository, like resumeEntityIds already is). Resolved best-effort against the live profile at render — a referenced entity or bullet that's since been deleted is silently dropped, exactly as reconstructSelection does today.

{
  "version": 1,
  "sectionOrder": ["experience", "projects", "education", "skills", "languages", "interests"],
  "sectionsOff": ["interests"],          // section types toggled entirely off
  "entities": {
    "<workRoleId>": { "kind": "workRole", "bulletIds": ["b1", "b3"] },
    "<projectId>":  { "kind": "project",  "bulletIds": ["b7"] }
  },
  // (OQ5) optional forward-compat: "excluded": ["<guardRoleId>"]
  "extras": { "skillItems": ["PyTorch", "..."], "languages": ["Spanish"], "hobbies": [] }
}

Selection is binary (the consequence of retiring Suggest — OQ4/OQ5). Each entity in entities is included, and its bulletIds are exactly the bullets that render, in that order. An entity not listed is excluded. There's no state / lock field because v1 has no automated process to gate against — “always-in” is just “checked” and “never” is just “unchecked”.

The Security-Officer job is simply absent from the ML Canon's entities map — and with no scorer, no spine rule, and no Suggest in the picture, nothing can put it back. That's the structural fix. (OQ5 open: whether to also persist an explicit excluded id-list as forward-compat insurance, so a possible future LLM Suggest wouldn't re-propose deliberate excludes.) Within an included entity, bulletIds is the ordered set of bullets to render (OQ7 — per-bullet granularity).

Alternatives considered

  • Keep the keyword scorer as a “Suggest from posting” pre-fill. Rejected (OQ4): it's the exact mechanism being replaced — keyword overlap is why the guard bullet matched in the first place. A pre-fill built on it inherits the flaw. If a suggestion affordance is ever wanted, it should be a semantic LLM relevance pass, not the scorer.
  • Three-state entity model (always / auto / never). Rejected (OQ5): those states only earned their keep when a Suggest pass needed fencing. With no automated mutation, they collapse to a plain checkbox; carrying them would be dead complexity.
  • Per-bullet global locked/excluded flags (already exist). Rejected: those are global — they affect every resume. The whole point is “include this job on my Security resume but never on my ML one,” which is inherently per-Canon. Manual selection lives at the Canon, not the bullet.
  • Relational CanonSelectionEntry table. Rejected for v1: the stored selection is already JSON (GeneratedResume.selections), the resolve-against-live-profile pattern already exists, and a JSON column matches the house pattern with no join model to maintain. Revisit only if we need to query selections across canons.
  • Drag-reorder in v1. Deferred (OQ8): real work, not load-bearing for the relevance fix. v1 inherits profile order.

Rollback

Canon.selection is nullable and additive, and the manual generate path is gated on its presence — so backing out the route branch makes every canon fall through to the existing auto-pipeline (degrades to today's behavior, a no-op). To revert: drop the route branch + overlay, revert + npm run build + pm2 restart mission-control. The column can stay (harmless) or be removed with a down-migration after a prod.db backup. No generated artifacts or existing GeneratedResume rows are mutated by this feature.

Open Questions

Unresolved decisions

Most forks are now decided (this session's design conversation). Only two remain genuinely open: OQ5 (final nod on pure-binary vs a forward-compat exclude layer — gates the Phase 1 data model) and the OQ9 sub-point (live-preview fidelity). The five marked ✓ Defaulted were taken to my stated lean to keep momentum — say the word to revisit any. Resolve OQ5 before writing Phase 1 code.

OQ1 — What does the AI still do on the manually-picked bullets?
✓ Decided: manual selection is the core; rewrite + tagline are opt-in toggles, default off → deterministic verbatim render. The selection problem is separate from wording, so wording stays optional polish.
OQ2 — What does “locked aka always there” mean?
✓ Decided (revised): lock is per-entity, not per-section-type. But once Suggest was dropped (OQ4) there's no automated process to lock against, so “lock” collapses to a plain include/exclude checkbox. The per-entity (and per-bullet) granularity stands; the separate “lock” affordance does not.
OQ3 — Is a manual selection saved and reusable?
✓ Decided: saved per-Canon, re-renderable. Canons already model the user's job-search lanes; “ML resume vs security resume” lives there.
OQ4 — What happens to the existing auto-scoring pipeline?
✓ Decided (revised): no “Suggest” button in v1. The keyword auto-selector (selectBullets) is retired from the selection path — not demoted to a pre-fill. (Expanding OQ11 showed a keyword-overlap Suggest inherits the exact flaw that motivated this redesign.) If a suggestion affordance returns later it'll be an LLM relevance pass (semantic, respects deliberate excludes), scored against canon keywords — explicitly deferred.
OQ5 — Pure binary, or binary + a forward-compat exclude layer?
With Suggest removed (OQ4) the original always/auto/never collapses to a plain included/excluded checkbox per entity and per bullet. Options: (A) pure binary — an entity is either in entities (with chosen bulletIds) or absent; (B) binary + an explicit excluded id-list recording deliberate excludes, as forward-compat insurance so a future LLM Suggest wouldn't re-propose the guard job. Lean: A (truly minimal); B costs one array now to save a future annoyance. Status: open — needs final nod; gates Phase 1.
OQ6 — Where is the selection stored?
✓ Defaulted (to lean): a single nullable JSON column Canon.selection — matches the house JSON-selection pattern, resolved best-effort against the live profile; no join model. (Alternatives: relational table; reuse the deferred pinnedEntityIds/sectionOrder fields.)
OQ7 — Bullet granularity in v1?
✓ Decided: per-bullet checkboxes within an included entity (the user's “select the ones I want”). Picking which bullets show is core to a good resume and cheap given the data is already there.
OQ8 — Ordering control in v1?
✓ Defaulted (to lean): inherit profile position order + a fixed DEFAULT_SECTION_ORDER; defer drag-reorder of sections/entities/bullets to a later phase.
OQ9 — Length / one-page handling?
✓ Decided: the manual path drops the one-page auto-pruner (the user owns length), and the builder shows a live page indicator as boxes are toggled. Open sub-point: fidelity — a true PDF render-on-edit (exact, ~seconds each) vs a fast in-browser HTML/height estimate that's confirmed-exact at Generate. Lean: fast estimate live + exact count at Generate. Status: sub-point open.
OQ10 — UI surface for the builder?
✓ Decided: a full-screen Window-class overlay launched from the Canon row — the full profile tree is too big for an in-grid card, and an overlay is the established escape-the-grid pattern (like the AI Companion).
OQ11 — “Suggest from posting” input?
✓ Resolved — moot. No Suggest in v1 (OQ4), so there's no input to choose. If an LLM relevance Suggest is built later, the input is canon keywords by default with an optional pasted posting to refine.
OQ12 — Fate of the one-off generator (GenerateResumeCard)?
✓ Defaulted (to lean): keep it as the quick non-canon throwaway path for now; the canon builder is the curated path. Revisit once the builder lands.
OQ13 — Rewrite scope when the toggle is on?
✓ Defaulted (to lean): keep today's behavior — rewrite only posting-matching bullets, pass no-match bullets through verbatim (rewrite.ts pre-filter); avoids cross-domain invention and saves tokens. Tagline still needs a posting/keywords.
