LLM Calls & Cost Optimization

The operational reference for every Gemini callsite in mission-control — the model fleet, the per-callsite inventory + token budgets, the observability wiring — and how we keep the free-tier quota a sip, not a chug: the cost model, the optimizations already in place, the live backlog, and what we deliberately leave alone. Most LLM work here is mechanical extraction that doesn't need the highest-end model. Prompt blobs live in llm-prompts/; this doc is about structure — which model, how many calls, how big — not about what each prompt says.

Update this doc when: a new callsite ships (add an inventory row + a change-log line), a model/token-cap changes (update the row + change log), you observe a quality regression on a downgraded callsite (revert + record what you saw), or you add/ship a cost optimization (§6 + change log).

Model fleet

The cost model

Two levers: calls and tokens

Every Gemini interaction costs on two independent axes:

  • Call count — each call pays the acquireGeminiSlot() rate-limit cost (§9) and re-sends its system prompt. A loop firing one call per entity multiplies both slot pressure and the repeated system-prompt tokens.
  • Token count per call — input (capped external content + prompt) plus output (bounded by maxOutputTokens). The system prompt is paid on every call, so a 250-token system prompt across an N-call loop is 250 × N wasted if one call could carry all N items.

The toolkit, in rough order of leverage:

  1. Eliminate the call — cache a pure result, or replace it with deterministic code.
  2. Merge calls — fold two same-trigger, same-input calls into one; batch an N-loop into a single positional call.
  3. Gate the call — a cheap deterministic pre-filter that short-circuits the common no-op case.
  4. Exclude by user intent — if the user has declared a negative filter / blocklist over the inputs, apply it before the LLM (and ideally before the input is even stored). Classifying something the user already filtered out is pure waste — the result is never shown. This is the highest-leverage gate there is: zero quality cost, no judgment call, and it's literally what the user asked for. Easy to miss precisely because the filter lives elsewhere (a read-time feed cull), so the LLM path silently keeps paying for rows the UI will never display. (See the global negative-filter drop, §4.)
  5. Trim tokens — cap input to the signal head; size maxOutputTokens to ~4× typical output; drop fields the model doesn't need.

Where the spend concentrates

Not evenly. Two paths dominate, and they're where every opportunity in §6 lives:

  • The resume-gen pipeline (POST /api/resumes) — one generate fires 4 distinct callsites plus an N-per-entity loop. The multi-call hot path (§5).
  • Profile import — one profile-import (LITE) call per file, then one profile-synthesize call — the single most expensive call in the app (FLASH, 32k output, 80 KB input).

The two highest-volume callsites — email-parser and employment-type-classifier — are already heavily gated and batched (§4), so they're not where the remaining fat is.

The resume-gen pipeline (the hot path)

Call sequence for one generate

A single POST /api/resumes runs this sequence. Calls in best-effort steps are wrapped in try/catch — a failure degrades but never blocks the generate. Post-Tier-2: scratchpad-synth is ONE batched call regardless of entity count, and resume-rewrite ∥ resume-tagline run concurrently.

Deterministic floor is 2 calls (posting-parse + resume-rewrite). The usual case is ~5: posting-parse, bullet-tags-from-posting, one batched scratchpad-synth (when any entity is gated in), then resume-rewrite ∥ resume-tagline concurrently. A heavy generate (fresh posting, 6 scratchpad'd entities) is now ~5 calls — down from ~10 before Tier 2 collapsed the per-entity synth loop into one batched call.

Why this was the target

  • posting-parse was a pure function of the posting content, yet re-run on every generate. (Fixed — Tier 1 ✅, §6.)
  • scratchpad-synth was a per-entity loop, each iteration re-paying its ~250-token system prompt. (Fixed — batched into one positional call, Tier 2b ✅, §6.)
  • resume-tagline and resume-rewrite were two sequential post-selection calls with no data dependency. (Fixed — run concurrently, Tier 2a ✅, §6. Merging them into one FLASH call was rejected: it would move the tagline's multi-KB input onto FLASH pricing.)

Tier 1 — eliminate redundant calls ✅ (shipped 2026-05-28)

Clean wins, low risk, no quality tradeoff. Both landed — re-verified against the code on 2026-05-28.

  • ✅ Cache posting-parse per posting content. lib/resumes/posting.ts:parsePosting memoizes keyed on content: pasted text → sha256 of cleaned+capped text (never stale); URL-only → keyed on URL, computed before the fetch so a hit skips both the scrape and the LLM. 30-min TTL bounds URL staleness; 200-entry LRU cap; clones on read/write. Eliminates N−1 parses across N variants of one job. Smoke: posting-parse-cache-smoke.ts.
  • ✅ Skip profile-synthesize for single-file + empty-profile imports. lib/profile/synthesize.ts:canSkipSynthesis short-circuits the FLASH call when there's exactly one draft and no existing entities — the extraction is the canonical master (nothing to consolidate cross-file, nothing existing to reconcile, and profile-import already ran the same project-vs-role classification). Eliminates the priciest call on the common first-import path. Deliberately narrow: ≥2 files or a non-empty profile still runs the pass. Smoke: synthesize-skip-guard-smoke.ts.

Tier 2 — collapse the rewrite/tagline pair + the synth loop ✅ (shipped 2026-05-28)

  • ✅ 2a — run resume-rewrite ∥ resume-tagline concurrently (Promise.all in app/api/resumes/route.ts). The two calls have no data dependency (entityOrder is applied after both resolve), so firing them together captures the latency win. Implemented as a parallelize, NOT the originally-proposed merge. The merge (one FLASH call returning { bullets, tagline, sectionOrder, entityOrder }) looked like a call-count win but is a likely cost loss: the tagline's input is a superset of rewrite's (it carries a multi-KB profileSummary + per-entity evidence the rewrite doesn't need), so merging would move all of that onto FLASH pricing to save only a ~250-token system prompt + one slot — and the §7 non-goals already note the rate-limit slot isn't the bottleneck. Parallelizing keeps the tagline on cheap LITE, keeps its independent try/catch fallback, and leaves the employer-facing rewrite prompt untouched. Rewrite stays fatal-on-throw; the tagline stays best-effort.
  • ✅ 2b — batched scratchpad-synth N → 1. One call carries every gated entity as a delimited, numbered block with positional output ({ entries: [...] } aligned by index), replacing the former per-entity Promise.allSettled loop. Saves (N−1) repeated ~250-token system prompts + (N−1) rate-limit slots per generate; output budget scales with entity count (min(8192, 512 + N×1536)). Per-entity isolation is enforced structurally (each entity is its own block; a sibling's scratchpad never enters another's block) + instructionally (system rule 8). Count drift is tolerated (align by index; missing → empty, extra → ignored). Latency was already 1× (the old loop ran in parallel), so the win is tokens + call count. New hermetic coverage proves 2 entities → 1 call with correct attribution; Promptfoo adds a multi-entity isolation fixture.
Deploy coupling (2b). 2b changed the scratchpad-synth output shape ({ bullets } → { entries }), so the new code REQUIRES the new prompt. When LUNARY_PUBLIC_KEY is set (dev + prod), loadPrompt serves the Lunary template, so the new code must ship together with ONLY=scratchpad-synth npx tsx scripts/sync-lunary-templates.ts (pushes the new non-draft version) + a pm2 restart. Until synced, the new code hits the stale single-entity template and every synth call fails its schema → best-effort degrade to no synthesized bullets. The hermetic gate forces the disk fallback (LUNARY_PUBLIC_KEY="") so it stays green either way.

Tier 3 — prompt-text consolidation (proposed)

Cuts per-call tokens + drift, not call count. Can't merge the calls (different surfaces), but the text overlaps.

  • tagline-draft ≈ resume-tagline. ~80% shared rule block (no-fabrication / one-sentence / no-first-person / no-puffery). Factor the shared rules into a loadPrompt partial so the templates can't drift apart.
  • Bullet-drafting family shares a hallucination guard. bullet-assist-fill + bullet-assist-rewrite already share one system prompt; scratchpad-synth carries a near-identical variant. Consolidate into one shared guard block.

Tier 4 — evaluate, needs measurement (proposed)

  • bullet-tags-from-posting overlaps the deterministic scorer. lib/resumes/select.ts already does substring + tag matching for free; the LLM auto-tag's only marginal value is conceptual matching ("Built a Python API" ⇒ "Python"). Measure whether gating it to keywords the substring scorer missed (or dropping it) changes selection. Caution: it also writes the UI "pending review" tag badges, so dropping it has a UX cost beyond selection.
Rough impact (Tier 1 + 2 shipped). A heavy generate dropped from ~10 calls to ~5 — Tier 2b collapsed the per-entity synth loop into one batched call, and Tier 2a overlaps rewrite + tagline so they no longer serialize. A re-generate against the same posting drops another via the Tier 1 cache. Tier 3/4 (still open) would trim per-call tokens on top.

Cross-tier call dedup

Full design: cross-tier-llm-dedup.html. dev (:4101 / dev.db) and prod (:3101 / prod.db) — plus their two schedulers — run the SAME code on the SAME box against the SAME data, so every cacheable LLM step was calling Gemini twice for identical input. A thin wrapper, llmCached(req, compute) in lib/ai/llm-cache.ts, content-hashes the request, checks a shared SQLite file both tiers open (data/llm-cache.db, WAL), and either returns a cached result, leads the computation, or follows the other tier's in-flight one (an INSERT OR IGNORE reservation gives single-flight even when both tiers fire within the same second — the Gmail push fan-out case). Shipped 2026-05-29.

What's cached vs. what re-rolls

Caching is default-on in chatJSON (cache?: boolean, default true): deterministic extraction / classification is safe and high-value to cache. The key is content-addressed (resolved model + rendered prompt + output JSON Schema), so editing a prompt or bumping a model auto-invalidates — eviction is housekeeping, never correctness. A few intentionally-generative callsites opt out (cache: false) because they are meant to re-roll on identical input; freezing them would be a regression.

StateCallsitesWhy
Cached (default) email-parser, employment-type-classifier, posting-parse, resume-rewrite, profile-import, profile-synthesize, bullet-tags-from-posting, resume-tagline Deterministic extraction / classification / tailoring. Identical input genuinely should produce the same output — and dedups both across tiers and across the N-resume-variants-against-one-posting case.
cache: false discovery-suggest, bullet-assist-fill, bullet-assist-rewrite, tagline-draft, bullet-tags-from-profile, scratchpad-synth Generative "suggest / draft" calls the user re-asks for a fresh take. Per the anti-repetition rule they already fold the user's known items into the prompt — but they're meant to vary on re-ask with identical input, so they must opt out.

Cache-hit tracing

The rate-slot acquisition (acquireGeminiSlot) + the model call + the Lunary trace all live inside the compute() closure, which runs only for the leader. So a cache hit / follower spends no rate token, makes no API call, and is not traced as a model call in Lunary — there is no model event to attribute, because no model call happened. (No distinct cache-hit event is emitted; cost dashboards count only real calls.) The [AI] <name> … tokens=… log line likewise only prints on a real call.

Best-effort, never load-bearing

Any store-layer failure — a better-sqlite3 ABI mismatch after an nvm Node switch, a missing/locked/unwritable file, a write error — degrades to calling compute() directly (uncached) and logs one warning. The native binding is loaded via a guarded dynamic import() so a load failure can be caught instead of crashing every importer of lib/ai/gemini.ts. Errors from compute() itself are never cached (the leader DELETEs its reservation and rethrows). A daily llm-cache-prune scheduler job bounds disk (60d on done, 24h on crashed pending rows). Hermetic coverage: scripts/tests/hermetic/llm-cache-smoke.ts (single-flight, error-not-cached, stale-takeover, cache-down, prune).

Non-goals

Deliberately left alone

  • Don't over-gate email-parser. The relevance gate is load-bearing but conservative on purpose — a tighter pre-filter risks dropping real application emails (false negatives are worse than an extra LITE call). The 3 KB body cap is already tight.
  • Don't downgrade resume-rewrite off FLASH. It's the only FLASH callsite because its output is what the user emails to employers. Quality dominates cost here.
  • Don't widen profile-synthesize's skip. With ≥2 files, cross-file dedup is its core job; with a non-empty profile, existing-merge reconciliation matters. The skip is narrow by design.
  • Don't raise the rate-limit bucket to "save calls". It doesn't — ingest is sequential and Gemini's own latency dominates past ~120 RPM. Raising the bucket only helps when the bucket itself is the bottleneck.
  • Don't merge calls that fire on different triggers. tagline-draft (profile Sparkles button) and resume-tagline (resume-gen) never co-fire — consolidate their prompt text (Tier 3), not their calls.

Observability (Lunary + Promptfoo)

Three pieces wire LLM ops across every callsite (LOP-1 → LOP-9, landed 2026-05-24). Design doc: implementation §LLM observability + prompt registry.

Tracing — Lunary (managed cloud)

lib/ai/gemini.ts wraps the inner generateContent call with lunary.wrapModel at module-init, gated on LUNARY_PUBLIC_KEY — unset (dev / CI / fresh clones) ⇒ tracedGenerate === rawGenerate and Lunary code never executes. Set ⇒ every call lands in Lunary's dashboard with name, model, input messages, output, prompt+completion tokens, latency. lib/email-parser.ts bypasses chatJSON, so it tracks manually via lunary.trackEvent inside a defensive safeTrack helper — a Lunary failure logs a warn but never disrupts Gmail ingest. Init lives in instrumentation.ts next to initLogger(). Activate: drop LUNARY_PUBLIC_KEY into .env → pm2 restart mission-control-{dev,scheduler-dev,scheduler-prod} --update-env.

Prompt registry — Lunary templates

Prompt loading is uniform across all 14 callsites: each loads its prompt through lib/ai/prompts.ts:loadPrompt(slug, vars) — the 13 chatJSON callsites and email-parser (which loads its prompt the same way even though it makes the Gemini call via the Vercel AI SDK's generateObject, not chatJSON). Only the model-call package differs — Gemini SDK via chatJSON for the 13, Vercel AI SDK for the lone email-parser; the prompt-registry path is shared by all. loadPrompt prefers Lunary's renderTemplate (versioned, dashboard-editable) when LUNARY_PUBLIC_KEY is set, else falls back to parsing docs/llm-prompts/<slug>.md from disk — the path that makes hermetic smokes + Lunary-less dev work, and protects prod through Lunary blips. Editing: dashboard edits publish a new version instantly (mirror back to the .md same-day); code edits to the .md push via npx tsx scripts/sync-lunary-templates.ts (idempotent; needs LUNARY_SECRET_KEY, distinct from the public key).

Dashboard caveat. Lunary's model picker only knows its built-in OpenAI/Anthropic SKUs, so our Gemini 3.x strings show as "gpt-5.5" or similar in the dropdown. The actual extra.model is stored correctly and surfaces through renderTemplate — the dropdown is cosmetic. The playground / built-in eval is unusable for our fleet; Promptfoo is the eval path.

Eval harness — Promptfoo (manual trigger only)

eval/ with a TS custom provider (eval/provider.ts) dispatching each fixture's callsite slug to the real chatJSON-wrapped lib function. One YAML per slug under eval/suites/, each an array of fixtures with is-json + schema-shape javascript + llm-rubric assertions. Judge model pinned to MODEL_LITE_CHEAP (~$0.001/call). Run: npm run test:prompts. NOT in pre-push.sh — burns real Gemini tokens (~$0.01–0.05/full-run). A new callsite needs: name in the inventory, prompt blob in docs/llm-prompts/, handler in eval/provider.ts:HANDLERS, fixture in eval/suites/<slug>.yaml.

Defaults + rate limiting

Defaults baked into chatJSON

  • temperature → 0.4 (callers override: 0.1 conservative extraction, 0.9 exploration).
  • maxOutputTokens → 4096 (callers override; 32k only for profile import + synthesize).
  • thinkingBudget → 0 (Gemini 2.5+ defaults to thinking mode, adding 30s–2min latency and eating the output budget — all current callers are mechanical extraction, no chain-of-thought needed).
  • responseMimeType → "application/json" (every caller is structured-output).

Rate limiting

lib/ai/rate-limit.ts:acquireGeminiSlot() is a process-shared token bucket. Code default 12 req/min, burst 60 (free-tier safe). Every Gemini call MUST go through chatJSON or parseApplicationEmail — both block on the bucket before hitting the API. Bypass = quota drain risk. Env overrides: GEMINI_RATE_PER_MIN (default 12, prod ships 60 via .env.production), GEMINI_RATE_BURST (default 60). Practical ceiling: ingest is sequential, so RPM above ~120 stops mattering — Gemini's own ~1–2s response latency dominates. Raising the bucket only helps when the bucket itself is the bottleneck.

Adding a new Gemini caller

Wiring steps

  1. Route through chatJSON (lib/ai/gemini.ts) — gets you retries, the rate-limit gate, Zod validation, token-usage logging.
  2. Pick the model: enum-like with ≤5 classes → MODEL_LITE_CHEAP; output ships to a user as resume / cover-letter / high-stakes text → MODEL_FLASH; anything else → omit model (inherits MODEL_LITE).
  3. Pass maxOutputTokens matched to expected size (~4× typical output, to absorb tail variance without burning budget).
  4. Add a row to the inventory (§2) + a prompt blob in llm-prompts/ + a Promptfoo fixture in eval/suites/<slug>.yaml + handler in eval/provider.ts:HANDLERS.
  5. If sending external content (HTML, email body, resume text), cap the input length at the smallest size that still preserves the signal — tail content is almost always boilerplate.

Cost checklist

Before you ship, also ask:

  1. Is the output a pure function of the input? If so, cache it (content-keyed) so repeat calls are free. (See parsePosting.)
  2. Is there a common no-op case? Add a deterministic pre-gate that short-circuits it before the LLM. (See looksRelevant, the rewrite no-match filter.)
  3. Has the user already excluded this input? If a negative filter / blocklist applies to the input, gate the call on it — and drop the input at ingest where possible. Never classify what the user explicitly filtered out; the result would never be shown. (See the global negative-filter drop, §4.)
  4. Are you calling it in a loop? Can the items batch into one positional call so the system prompt is paid once? (See employment-type-classifier.)
  5. Does a deterministic rule cover the obvious cases? Run the regex/heuristic first; only send the ambiguous remainder to the LLM. (See inferEmploymentTypeFromTitle.)
  6. Does an existing callsite fire on the same trigger over the same input? Consider merging the output schemas into one call.
  7. Is the input capped to the signal head? External content is mostly boilerplate past the first few KB.
  8. Is maxOutputTokens sized to ~4× typical output? Too generous burns budget silently; too tight surfaces as a MAX_TOKENS error (the safer failure).
