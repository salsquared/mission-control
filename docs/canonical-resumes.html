Canonical Resumes (“Canons”)

A plan for generating one resume per role-archetype instead of one per job — so near-identical postings (every “security officer” listing, every “math tutor” gig) reuse a single LLM-built resume rather than burning 2–3 Gemini calls each to re-derive the same thing. §1–§5 frame the idea + design; §6's decision list is now fully answered (2026-05-30) — the locked spec is the Decisions summary up top.

Status: design decided (2026-05-30) — no code yet. All 12 open questions in §6 are answered (✓ Decided callouts); the Decisions summary below is the locked spec, and §4's design + §7's phasing are updated to match. The problem

Per-job generation re-derives the same resume

Today every resume is built per posting. POST /api/resumes takes one job (a Pipeline application, a URL, or pasted text), extracts that posting's keywords, then runs the full relevance pipeline: select → scratchpad-synth → rewrite (FLASH) + tagline (LITE) → render. Each generation is 2–3 Gemini calls (per docs/llm-calls.html; the per-user rate limit even assumes “1 generate = 2–3 Gemini calls”).

But Sal's reality is that jobs cluster hard into a handful of role-types that draw on the same experience:

  • Every security officer posting (Allied Universal — museum rover, mall patrol, studio, …) → same security history, same bullets, near-identical output.
  • The four no-degree side watchlists already are role-types: AI coding evaluator, Math / test-prep tutor, Bilingual CS / interpreter, Junior web dev & QA.

Generating a fresh resume for each of, say, five Interested security jobs means ~10–15 Gemini calls to produce five PDFs that are practically the same document. The cross-tier LLM cache doesn't rescue this: it's content-addressed on the exact rendered prompt, so two similar-but-not-identical postings miss the cache and each pay full price.

Sal's framing: “it doesn't make sense to create resumes per job but per canon or type of job — we really only need one resume for security-type jobs.”

The idea

A Canon = a named role-archetype with one reusable resume

Introduce a Canon: a user-named role category (“Security Officer”, “Math Tutor”, “Junior Web Dev”) that owns one canonical resume. Instead of feeding the pipeline a single posting, we feed it the canon's keyword profile (a representative keyword set standing in for “what postings of this type ask for”), generate the resume once, and reuse it for every job in that canon.

Concretely, a canon's resume is a curated subset of your full profile — the experiences and bullets relevant to that role-type, selected (and re-worded) out of the one master profile. The profile stays the single source of truth; each canon is a lens that picks the slice of it that speaks to that role. (This is why staleness is entity-scoped — §6 Q7.)

A Canon is a new granularity level that nests between the two we already have:

LevelExampleExists?Resume relationship
Trackcareer / sideYes (Application.track, Watchlist.track)Too coarse — one resume can't serve all of “side”
Canon ← new“Security Officer”, “Math Tutor”No — this proposalOne reusable resume per canon
Job“Allied Universal — Museum Rover”Yes (Application)Too fine — the wasteful status quo

So the mental model is track ⊇ canon ⊇ job: a track holds several canons; a canon holds many jobs; a canon has exactly one resume that all its jobs share.

Why “Canon” is the right unit. It matches how Sal actually applies — the same bullets, ordering, and framing across a whole role-type — and it's the level at which the expensive LLM work is genuinely shared. The content-hash cache captures “identical input”; the Canon captures “same role, different listing,” which is where the real duplication lives.

How a Canon fits the existing architecture

Hooks that already exist

Most of the machinery a Canon needs is already present — the proposal is largely about where the posting input comes from and reuse, not a new pipeline:

  • The pipeline is keyword-driven, not posting-object-driven. selectBullets(profile, keywords, {}, keywordWeights) and rewriteBullets(flat, posting) consume a keyword list + weights. A canon just supplies those from its own keyword set instead of one posting — the selector, scorer, one-page pruner, and skills-gap don't need to know the difference.
  • Side watchlists already partition into role-types. The four side watchlists map ~1:1 to canons, and each carries a keyword string. Career watchlists are company-shaped rather than role-shaped, so career canons would need a different source (see §6).
  • pinKeywords is a posting-conditional pin. Per-entity pinKeywords (on WorkRole/Project/Education) already force an entity to lead its section when a posting matches. A canon could drive this deterministically (“the security canon always leads with the security role”).
  • GeneratedResume already stores a frozen artifact + selection trace. A canonical resume is a GeneratedResume that is keyed to a canon rather than a single posting, and is reused rather than regenerated.
  • The Pipeline picker already filters by track. The GenerateResumeCard Interested-app picker has a career/side/both filter — a natural place to surface “use this canon's resume.”

Data model

A new Canon model, plus a nullable canonId on the three things that point at it. A canonical resume is a GeneratedResume tagged with canonId and an isCanonical flag.

model Canon {
  id            String   @id @default(cuid())
  userId        String
  name          String              // "Security Officer" — user-facing label
  slug          String              // normalizeRoleName(name) — match key
  track         String              // "career" | "side" — every canon nests under a track
  description   String?             // optional user note

  // The canon's "posting profile": the keyword set that replaces a single
  // posting's keywords when generating. Same shape posting-parse emits.
  keywords      String   @default("[]")     // JSON [{ keyword, importance }]
  keywordSource String   @default("manual") // "manual" | "aggregated" | "llm"

  // Per-canon framing (§6 Q8) — DEFERRED past v1: the existing pipeline
  // (selection spine + the resume-tagline LLM call) already orders sections/
  // entities, so v1 leaves these null. Kept nullable for the later "lock a
  // fixed layout" refinement.
  pinnedEntityIds String?  // JSON string[] — entities forced to lead (deferred)
  sectionOrder    String?  // JSON SectionKey[] (deferred)
  onePage         Boolean  @default(true)   // default render = PDF + one-page (§6 Q9);
                                            // full-length + docx render on demand
  currentResumeId String?                   // active (latest) canon resume version

  active        Boolean  @default(true)
  // Demand-driven, entity-scoped staleness (§6 Q7). resumeStale flips true only
  // when an entity the current canon resume DEPENDS ON changes — its bullets,
  // tags/pinKeywords, or deletion — or when the canon's own keywords/framing
  // change. Edits to profile entities NOT in resumeEntityIds leave it fresh.
  resumeStale     Boolean  @default(true)
  resumeEntityIds String?  // JSON string[] — profile entity ids in the current
                           // canon resume's selection; the staleness dependency set.
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  user          User              @relation(fields: [userId], references: [id], onDelete: Cascade)
  resumes       GeneratedResume[] // the canonical resume(s)
  applications  Application[]     // jobs assigned to this canon
  watchlists    Watchlist[]       // watchlists that feed this canon

  @@unique([userId, slug])
  @@index([userId, track, active])
}

//   Application.canonId  String?   — which canon a tracked job belongs to (manual tag, §6 Q4)
//   Watchlist.canonId    String?   — optional manual link: a role-watchlist feeds a canon; its jobs inherit
//   GeneratedResume.canonId   String?
//   GeneratedResume.isCanonical Boolean @default(false)
//   GeneratedResume.canonVersion Int?    — auto-incremented per canon; prior
//      versions are retained + downloadable (auto-versioning, no diff/rollback

Generating a canon resume

Reuse POST /api/resumes almost verbatim. Add a fourth posting-input variant { canonId } alongside url/text/applicationId. When present, the route builds a synthetic ParsedPosting from canon.keywords (title = canon name, company = null, keywords = the canon's term list) and runs the unchanged pipeline — selection, rewrite, and the resume-tagline framing call all work as-is on canon keywords. v1 passes the keywords flat (no keywordWeights); the selector treats missing weights as importance 1, which is fine (§6 Q6). The result is persisted as a GeneratedResume with isCanonical = true (+ canonVersion), and canon.resumeStale is cleared. (Per-canon framing overrides and keyword weighting are deferred refinements — §6 Q8 / Q6.)

Reusing it for a specific job

The default reuse is verbatim — downloading a job's resume just streams the canon's already-rendered artifact, zero LLM:

Touch list (where the work lands)

Following the docs/resume-pipeline.md touch-list discipline, a canon-aware change threads through: prisma/schema.prisma + migration · lib/schemas/* (canon Zod) · lib/repositories/canons.ts (new) · app/api/canons/route.ts (new CRUD) · the { canonId } branch in app/api/resumes/route.ts · a keyword-split helper (text → term array; weighting deferred, §6 Q6) · a UI surface (§6 Q10) · hermetic smokes. The selector/rewriter/pruner stay untouched — they already speak keywords.

What this buys (cost model)

Calls scale with canons, not jobs

Let a canon hold N Interested jobs. Per-job gen costs N × (2–3) Gemini calls and produces N near-identical PDFs. Canon gen costs 1 × (2–3) calls, reused N times.

ScenarioStatus quo (per-job)With Canons
5 security jobs10–15 calls, 5 PDFs2–3 calls, 1 reused PDF
Re-apply to a 6th security job+2–3 calls0 calls (serve artifact)
Profile bullet edited → all 5 need refresh10–15 calls2–3 calls (regen the canon once)

The win compounds because cost becomes a function of how many role-types Sal pursues (a handful, stable) instead of how many jobs he applies to (many, growing). It also stacks on top of the cross-tier cache rather than replacing it.

From the code — what one generate actually spends. A per-job generate today fires up to four Gemini calls in app/api/resumes/route.ts: posting-parse (LITE), a best-effort bullet-tags-from-posting auto-tag (LITE), the resume-rewrite (FLASH — the only FLASH path in the app), and resume-tagline (LITE, which also decides section/entity order) — plus an optional scratchpad-synth call when posting keywords are left uncovered. With canons those costs move from per job to per canon-regen: v1 feeds the pipeline the canon's flat keyword text (no posting-parse — §6 Q6) and otherwise runs the same stages as a per-job gen. So a canon regen is roughly the usual auto-tag + resume-rewrite + resume-tagline — paid once per canon, and every verbatim reuse after that is zero. (Later refinements could weight keywords (Q6) or store framing to drop resume-tagline (Q8), but neither is needed for the core win — cost already scales with canons, not jobs.)
The real cost question is regeneration cadence, not generation. A canon resume is cheap to make once; the spend is in how often it's rebuilt (profile edits, keyword drift, staleness policy). That's §6 Q7 — and it's where a careless “regen on every profile save” could quietly undo the savings.

Q1 — Granularity & whether it's truly “one resume”

✓ Decided (2026-05-30).
  • One resume per canon, reused verbatim, is the default — good for now.
  • Canons sit at the role-type level. Museum-rover vs mall-patrol vs studio-guard collapse into a single “Security Officer” canon — not different enough to matter on paper, and that collapse is the point.
  • Count ≈ up to 10, roughly one per role-specific watchlist. Side ones are known (AI-coding Evaluator, Math/Test-prep Tutor, Bilingual CS/Interpreter, Junior Web Dev/QA, Security Officer); career role names TBD when you set them up.
One addition — an opt-in “specialization pass” (this reshapes Q5). Beyond the verbatim default, you want the option to push a canon resume through an LLM pass that re-tailors its language to a specific job's posting keywords (the way resume-tagline already tailors the tagline). So the model is verbatim by default, with an explicit per-job specialization as an opt-in — not the rarely-used escape hatch the first draft assumed. The scope of that pass is now the substance of Q5.

Q2 — Canon vs Track

✓ Decided (2026-05-30).
  • Every canon belongs to exactly one track. A role belongs to one track and is exclusively a member of that one — canons never span tracks.
  • Career gets canons in v1 too, not side-only. Both tracks ship together.
  • Canon never replaces track; they coexist. Canon is a strict subset of track (track ⊇ canon ⊇ job) — a finer partition within a track, not a competing axis.
Consequence (this drives Q3 + Q4). Because career is in v1, the “watchlist ≈ canon” shortcut no longer covers everything. It holds for side (linkedin/indeed watchlists are role-shaped) but not career (Greenhouse/Lever/Workday watchlists are company-shaped — one employer, many roles). So Q3 (keyword source) and Q4 (job→canon assignment) now split by track: the side mechanism and the career mechanism are genuinely different, and each needs its own answer.

Q3 — Where the canon's keywords come from

The pipeline needs a keyword profile (with per-keyword importance — see Q6) to stand in for “a posting.” Per Q2 this splits by track, because the two track types carry different watchlist shapes.

Side canons

✓ Your call (6.3.d): a side canon's keywords = its feeding watchlist's keywords. A linkedin/indeed watchlist already is a role-type and its keywords field is the role query.
Caveat (the actual decision lives in Q6). That field is a single free-text OR-string (e.g. "math tutor OR SAT prep OR test prep", max 200 chars), not a weighted {keyword, importance}[] list — so “same as the watchlist” still needs a small transform to feed the pipeline. Whether that's a plain split (flat weights) or a one-time posting-parse pass (real weights) is the importance question — answer it in Q6.

Career canons

✓ Your call (6.3 career): (a) manual. You type the canon's keywords yourself, mirroring the side track (where the keywords are the watchlist string you typed). A career watchlist is a company board with no role keywords to copy, so you curate them directly on the canon — “I'll just add some myself.”
This unifies the keyword source. Both tracks now boil down to user-provided keyword text — side reads it off the feeding watchlist's keywords string, career takes it straight from you. So there's one mechanism to build, not two, and the aggregation / LLM-synthesis options drop out of v1 (a self-maintaining “aggregate from this canon's tracked postings” mode stays on the table as a later enhancement). The only keyword question left is how that text becomes weighted — Q6.

Q4 — How a job gets assigned to a canon

✓ Decided (2026-05-30): (c) manual tag. You assign the canon yourself — no auto-derivation in v1. You'll create role-based watchlists by hand (not just company boards), and a job gets its canon from a manual tag. Consistent with the rest of your choices (manual keywords, etc.): keep v1 deterministic and user-driven, skip the inference.
What “manual tag” means in the UI (Q10). Two manual touch-points, both one-click:
  • Per-job — pick the canon when you track a posting / from the kanban card (sets Application.canonId).
  • Per-watchlist (optional convenience) — tag a role-shaped watchlist to a canon once (Watchlist.canonId); jobs it spawns inherit that canon. Still “manual” — you set the link one time — but it spares you per-job tagging for role-watchlists.
Deferred automation. The auto-assign options from the first draft — watchlist-derived at trackAsApplication, normalizeRoleName matching, LLM-classify — are not in v1. They stay on the table as later conveniences if hand-tagging gets tedious.

Q5 — Reuse fidelity: verbatim default + an opt-in specialization pass

Per Q1 there are two modes, not one.

Default — verbatim (decided)

✓ Default reuse is verbatim — serve the canon's stored artifact, 0 LLM per job. It's lossless: the rendered resume contains nothing per-company — the H1 is profile.headline (your name), the subtitle is the canon's tagline, and posting.company only ever reaches the download filename / X-Resume-Company header / stored row, never the document body. The one per-job artifact is the filename, which the reuse path can stamp for free.

Opt-in — specialization pass (the open scope)

What you asked for (6.1.a): an explicit, per-job action that re-tailors the canon resume's language to a specific posting's keywords — starting from the canon's already-made selection (which bullets/entities) rather than re-deriving it. Only wording changes; the curated bullet-set + ordering stay put. Sub-decisions (all resolved by the tentative below, 6.5):
  • Which calls? Tagline-only (cheapest), or tagline + a FLASH rewriteBullets over the canon's bullets against this posting's keywords (the real “language match”)?
  • Posting keywords source: a posting-parse of the actual job page (one call) — that's the “application-page keywords” you want to match.
  • Persistence: a specialized run saves a new per-job GeneratedResume linked to that Application, leaving the canon resume untouched. (Decided: yes — canon stays the reusable base; specializations are per-job children.)
  • Phasing: ship verbatim reuse first; specialization is a fast follow that reuses the existing rewrite/tagline machinery on a pre-built selection.
✓ Decided (6.5 — go with the tentative): default verbatim; specialization = posting-parse + FLASH rewriteBullets + resume-tagline over the canon's fixed selection, saved as a per-job GeneratedResume (canon resume untouched). Cost ≈ a normal per-job gen minus the select/synth stages — and it only fires when you ask, so the fleet-level win holds.

Q6 — Do the keywords need importance weights?

Re-examined (2026-05-30): not necessary for v1. keywordWeights is optional in scoreBullet (lib/resumes/select.ts) — when it's absent, every keyword scores at importance 1 and the pipeline runs fine: it still picks matching bullets, drops off-topic entities, and renders. Importance only changes tie-breaking — which bullets win when an entity has more matches than its slots (4 per work role) and a few marginal keep/drop calls. On a small profile with a hand-picked keyword list, that difference is minor. (Earlier this question called weights “mandatory” — that was wrong; they're a quality nicety.)
Recommendation: flat for v1. Split the canon's keyword text into a term array and pass it with no weights (zero LLM). If selection quality ever disappoints, add weighting later — either one posting-parse pass per canon, or let the canon editor accept an importance per keyword, or aggregate weights from the canon's tracked postings. All three stay on the table as future upgrades; none is needed to ship.

Q7 — Regeneration & staleness

✓ Decided (2026-05-30).
  • Entity-scoped staleness. A canon goes stale only when an experience in that canon's resume changes — its bullets, its tags/keywords (pinKeywords), or the experience is deleted — or when the canon's own keyword text / framing changes.
  • Unrelated edits don't touch it. Editing a profile experience that isn't in this canon's resume (or one that's only in a different canon) leaves this canon fresh. No blanket “profile changed → everything stale.”
  • Lazy + manual. Just tag it stale and show a badge; never auto-regenerate, never on a schedule. It rebuilds only when you click regenerate (or download a stale canon).
Mechanism. “Experiences in the canon” = the distinct entity ids in the canon resume's stored selections (each row carries sourceId). Record that set at gen time (a canon.resumeEntityIds JSON snapshot, or a small CanonEntity join for an indexed lookup). The staleness hook sits on the profile-entity mutations — app/api/profile/{work-roles,projects,education} PATCH/DELETE + bullet/tag edits — each flips resumeStale=true on only the canons whose dependency set contains the touched entity.
Wrinkles to accept/decide.
  • Newly-relevant entities won't trigger it. If you beef up an experience that wasn't selected last time, the canon won't auto-mark stale (it's not in the dependency set) even though a regen might now include it. That's the intended tradeoff of the narrow rule — a manual regen always picks it up.
  • Gen-time auto-tag. Resume gen runs bullet-tags-from-posting, which writes tags onto the profile. The just-generated canon must clear resumeStale after that (so it doesn't self-stale); open: does tagging a shared entity mark sibling canons stale?
  • Extras (skills / languages / interests). You enumerated experiences (bullets/keywords/deletion). These also render on the resume but aren't entity-IDed in selections — do edits to them mark a canon stale, or stay out of the dependency model? (Default: out, to honor the narrow rule.)

Q8 — Per-canon framing overrides

Re-examined (2026-05-30): not necessary for v1. The existing pipeline already orders the resume with zero per-canon fields — selectBullets picks a work-role spine + sorts by date, and the resume-tagline LLM call returns sectionOrder + entityOrder exactly as it does for every per-job resume today. A canon resume gets a perfectly good, stable layout from that path. (Earlier this question pitched stored framing as a way to skip the tagline call — but skipping it also means losing the role-relevant tagline, so v1 just keeps the call.)
Recommendation: defer. Don't add pinnedEntityIds / sectionOrder in v1. They're a later refinement worth it only if you want to lock a fixed layout (so regens never reorder) or hand-write a per-canon tagline. The one framing decision that is settled stays: one-page default = on (Q9). Entity pinKeywords already exist if you ever want a specific experience to always lead.

Q9 — Storage, formats, versioning

✓ Decided (2026-05-30).
  • One canonical resume per canon (the current version) — one stored selection, not one artifact per combo.
  • Default render = PDF + one-page. A new canon ships with pdf + one-page as its default; the full-length variant and other formats (docx) are available on demand, rendered from the same stored selection.
  • Auto-versioning, no diff. Each regen makes a new version; prior versions are retained + downloadable, but there's no diff/rollback UI — just keep the versions. Canon.currentResumeId points at the active one.
  • §4 model confirmed (Canon + nullable canonId on Application/Watchlist/GeneratedResume + isCanonical), plus GeneratedResume.canonVersion + Canon.currentResumeId for the version chain.
From the code — on-demand re-render is free of LLM. GeneratedResume already persists profileSnapshot (frozen profile JSON) + selections (the chosen, rewritten bullets) + artifactPath. So pdf↔docx and full↔one-page variants re-run only composeResumeProps + the renderer — zero Gemini — off the stored selection. So each version persists one selection; the default (pdf, one-page) artifact is materialized up front and the rest render on click.

Non-goals

Explicitly out of scope (for v1)

  • Automatic per-job LLM tailoring — the default path does zero per-job LLM (verbatim reuse). An explicit, opt-in specialization pass (Q5) is in scope as a wanted capability, but it never fires unless you ask for it; the existing per-job URL/Paste flow also stays as a manual escape hatch.
  • Replacing the relevance pipeline — select/rewrite/prune/skills-gap are reused as-is; canons only change the input (keywords) and add reuse.
  • Cover letters (Q11) — noted as the natural home for per-company personalization, but a separate feature.
  • Auto-regeneration on every profile edit — deliberately demand-driven to protect the cost win (§5).
