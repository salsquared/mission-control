Closed-jobs feature

A Closed state on both the applications kanban and the job-listings feed — set by hand or driven by the close-detection probe, one-click "open in new tab" on kanban cards, and a sharper probe so fewer dead listings sit around as live.

Context & problem
Status: design — all open questions (OQ1–OQ9) decided (2026-06-09); Tasks checklist authored, ready to implement. Three independent pillars, all requested in one ask.

Problem

Three related pain points around job opportunities that have closed (no longer taking applications):

  1. No manual "closed" state the user can set or see.
    • Applications kanban — ApplicationsKanbanCard.tsx:43 defines 8 columns (Interested → Applied → Assessment → Interviewing → Offer → Accepted → Declined → Rejected). There is no terminal "Closed" column, so when an opportunity the user was tracking closes, they currently delete the card — losing the history. The status set itself (APPLICATION_STATUSES, lib/schemas/applications.ts:7) has no CLOSED value.
    • Job-listings feed — JobPosting.status does already support "closed" (JOB_POSTING_STATUSES, lib/schemas/watchlists.ts), but it is only ever set automatically by the scheduler's liveness probe (job-watcher.ts:496). There is no manual "mark closed" control, and the feed only renders status:"new" (NewPostingsCard.tsx:117), so a closed listing simply disappears — it can't be reviewed.
  2. The job link is hard to open. When the user opens a kanban card, the posting URL is shown by PostingSourceLine (ApplicationDetailOverlay.tsx:46) — but only when app.postingId is set (:361). Manually-added applications carry no URL at all: AppRecord (ApplicationsKanbanCard.tsx:9) has no url field, so the kanban card face has nothing to click, and the user ends up selecting + copying a URL by hand or right-click → open-in-new-tab.
  3. Close-detection misses "closed but the link still exists" jobs. The scheduler only probes a posting when it is stale — absent from the latest fetch and lastSeenAt > 6h (job-watcher.ts:461). A job that is closed but still listed in the source feed never goes stale → never gets probed → never closes. And even when probed, probeViaHttpStatus (liveness.ts:134) returns "alive" on any 2xx that doesn't hit a closure marker — and the per-ATS marker lists are self-described as "initial guesses" (liveness.ts:79). So a "200-but-closed" page (job content intact, banner says "no longer accepting applications") reads as alive. The user reports "missing lots of jobs that are closed but the links still exist," which matches both gaps.

Goals & non-goals

  • Goal: A CLOSED state the user can set and see on both the applications kanban and the job-listings feed.
  • Goal: A confirmed-closed listing cascades to its linked kanban card — the close-detection probe (and a manual posting-close) auto-moves a still-INTERESTED application to CLOSED, the same way email ingest drives application state. No more deleting cards by hand.
  • Goal: One-click "open in new tab" for the job link from the kanban (card face + detail overlay), for every application — including manually-added ones.
  • Goal: Measurably fewer missed closures — close-detection that catches the closed-but-still-listed and 200-but-closed cases the current probe misses.
  • Non-goal (this doc): An auto-classifier that emits CLOSED from emails. CLOSED is a manual decision; the email classifier is out of scope (see OQ1).
  • Non-goal: A blanket re-probe of every still-listed posting on every tick. C3 (committed, OQ5) is the budgeted, rotating version — it re-probes a capped sample per tick ordered by lastProbedAt, so a full pass spreads over many ticks rather than hammering the shared IP at once.
  • Non-goal: Reviving a closed posting back to new if the source re-lists it. Closed stays closed (matches today's behaviour; the manual control inherits it).

Design

Pillar A — manual CLOSED state on both surfaces

A1 · Applications kanban

Why a new status and not the existing Declined? The board already has distinct terminal meanings that CLOSED must not collapse into: Declined = I said no (withdrew / turned it down — the user's "things I don't want" column), Rejected = they said no, and the new Closed = the opportunity itself ended (listing pulled / no longer accepting) with no yes-or-no either way — the state the cascade sets. They are not interchangeable, so CLOSED is its own value on the Application model rather than a flag derived from the linked posting (see Alternatives considered).

The kanban renders directly off APPLICATION_STATUSES and a column table, so this is an additive change with no new abstractions:

  • Add 'CLOSED' to APPLICATION_STATUSES (lib/schemas/applications.ts:7). The read shape is status: z.string() (:47) so existing rows are unaffected; the POST/PATCH paths use ApplicationStatusSchema, which gains the value for free.
  • Add a terminal Closed column to pipelineColumns (ApplicationsKanbanCard.tsx:43), after Rejected, with a neutral/archived hue distinct from Rejected's slate — e.g. bg-zinc-500/20 text-zinc-400 border border-zinc-500/20 (placement & treatment = OQ2).
  • The detail-overlay status buttons render one per APPLICATION_STATUSES (ApplicationDetailOverlay.tsx:367), so CLOSED appears there automatically — no edit needed.
  • Drag-to-column and the button both call PATCH /api/applications, already ApplicationStatusSchema.optional() — no route change. The PATCH already writes a STATUS_CHANGED anchor at now() (route.ts:158), so a manual close is correctly the latest status anchor — the stale-email guard then protects it (an older email can't un-close it; see OQ6).
  • CLOSED is set by hand or by the cascade — the same status value is reached when a linked listing is confirmed closed (see Pillar C → A cascade below). Both paths write a STATUS_CHANGED event so the timeline shows why (provenance via syncSource: ms for manual, probe for auto).
  • Keep CLOSED out of the LLM classifier enum in lib/email-parser.ts (the comment at applications.ts:3 says the set "mirrors" the classifier — that becomes "superset of"; update the comment). See OQ1.

A2 · Job-listings feed

The status value and the API both already exist — this is a UI + surfacing change:

  • Mark-closed control on PostingRow (NewPostingsCard.tsx:626), alongside the existing hide/track buttons (:673). It calls api.postings.update(id, { status: "closed" }) — the PATCH route already accepts JobPostingStatusSchema (app/api/postings/[id]/route.ts).
  • Surface closed listings. The feed requests status:"new" only (:117); the list route already supports ?status=closed (app/api/postings/route.ts:108). Render closed ones in a way that's out of the way but reviewable — mechanism is OQ3 (collapsed "Recently closed (N)" section vs. an Open/Closed toggle vs. inline dimmed badge).
  • Stickiness is already correct. The seen-again path bumps lastSeenAt but never touches status (job-watcher.ts:330), and stale-candidate selection excludes closed/hidden (:466) — so a manually-closed posting that's still in the feed will not be revived or re-probed. No new guard needed; verify with a hermetic smoke.

Pillar B — one-click job link from the kanban

The root issue is that the kanban has no link to click for most apps. Fix the data first, then the UI:

  • Data: give every application a job URL. A manually-added app has no postingId, so deriving from JobPosting.sourceUrl alone leaves exactly the user's problem rows link-less. Add Application.url String? (populated from posting.sourceUrl at track-as-application time, and editable for manual apps), then thread it onto the applications list payload + AppRecord (ApplicationsKanbanCard.tsx:9). Whether a dedicated column is load-bearing vs. derive-only = OQ4.
  • UI — card face: a small ExternalLink icon button (mirroring NewPostingsCard.tsx:643) rendered when url is present, as a real <a href target="_blank" rel="noopener noreferrer">, with onClick stopPropagation so it doesn't open the detail overlay.
  • UI — detail overlay: promote the existing PostingSourceLine (ApplicationDetailOverlay.tsx:46) to render for any app with a URL (not just postingId), styled as an unmistakable one-click "Open posting ↗" button rather than selectable URL text.

C · What counts as a closure marker

"Marker" is overloaded in the diagram above. Concretely, the probe (lib/postings/liveness.ts) draws a closed verdict from four distinct signal tiers, in descending reliability — and most kinds use only the first. A marker is not a structured signal the ATS sends on purpose; tiers 2–3 are us scraping the public page the way a browser would and pattern-matching whatever it returns.

Two consequences worth stating plainly:

  • The generic tier (smartrecruiters, workable, recruitee, personio, clearcompany, careers-page) uses only tier 1 — probeGeneric passes no body check (liveness.ts:289). A closed job on any of those that returns 200 with a "position filled" banner is invisible today — they don't even have a marker list. That's Gap B for the entire generic tier.
  • So when this doc says "expand markers" (C1) it means tiers 2–3: curate the scraped phrases and extend the require-an-alive-marker-before-declaring-alive gate (today only LinkedIn/Indeed) to the other HTML kinds. "Prefer structured state" (C2) means promoting tier 4 where a JSON API exposes an open/closed flag — the only liveness signal the ATS actually sends on purpose.

Scoped, lowest-risk-first fixes (Sal's minimal-v1 preference — measure before widening, since a too-aggressive probe re-introduces the false-close behaviour the probe gate was built to kill):

  • C0 · Diagnose (do this first). A probe-audit script under scripts/tests/probes/ that samples postings per kind, re-probes via probePostingLiveness, and — crucially — captures the raw page text + final URL for every non-closed verdict, not just a count. Reporting the verdict alone can't reveal a missing phrase; dumping the actual body of alive/unknown pages that are really closed is what surfaces the wording LinkedIn/Indeed/etc. use ("no longer accepting applications", "this job has expired", …) so C1's tier-3 lists come from observed pages, not hunches. Seed it with a handful of known-dead URLs (the new manual-CLOSE control gives a growing supply) so the sample is guaranteed to contain closed examples to read. Output: closed-but-still-listed rate per kind + the unmatched closure phrases → tells us whether the misses are Gap A (not-probing-live) or Gap B (bad/absent markers), so C1–C3 are data-driven, not guesses.
  • C1 · Expand & curate closure markers (liveness.ts:79) informed by C0, and extend the "require an alive-marker before declaring alive" pattern (already used for LinkedIn :210 / Indeed :227) to the other HTML-scraping kinds, so a 200-but-closed page falls to unknown rather than alive. Additive, conservative.
  • C2 · Prefer structured ATS state. Greenhouse/Lever already probe the JSON API first (liveness.ts:234,:248); a removed Greenhouse/Lever job 404s on the board API. Audit (C0) whether closed-but-listed jobs expose an explicit open/closed flag in those JSON payloads (and Ashby/Workday CXS), and trust that over HTML heuristics where present.
  • C3 · Periodic re-probe of still-listed postings (committed — OQ5). A budgeted, rotating sweep that re-probes status:"new" postings still in the feed (so they never go stale) to catch Gap A directly — designed below. C0 still runs first, but only to size C3's budget and feed the C1/C2 markers; C3 ships regardless of what C0 finds.

C3 · Periodic re-probe of still-listed postings (the Gap A sweep)

C1/C2 sharpen the probe but can't help Gap A — a closed-but-still-listed posting is never stale, so the existing close-detection never selects it (job-watcher.ts:461). C3 adds a second, proactive sweep that re-probes still-listed postings on a rolling budget, so every active posting is eventually re-checked even while it's still in the feed.

  • Rotation key. Add JobPosting.lastProbedAt DateTime? (additive, nullable — same migration shape as Application.url). lastSeenAt can't be reused: the fetch bumps it on every sighting (job-watcher.ts:330), so it tracks "seen in feed," not "probed." The sweep selects status:"new" postings ordered by lastProbedAt asc (nulls first) and stamps lastProbedAt on everything it probes — a durable rolling cursor with no in-memory state to lose on restart.
  • Budget & cadence. A per-kind cap separate from and smaller than the stale-probe's maxPerTick, so the proactive sweep never starves close-detection of its rate budget. Size it from C0 so a full pass over the still-listed population completes within a target freshness window (decide the window — e.g. every active posting re-probed at least once / 24–48 h). Reuses the same PROBE_PROFILES concurrency / delay / 429-abort machinery and the arXiv-style cooldown awareness, so it stays inside politeness limits on the shared IP.
  • Verdict handling = identical to the stale path. A C3 closed verdict flips status="closed" and runs the same Pillar C → A cascade (source:"probe") — a still-listed posting found dead closes its linked INTERESTED card just like a stale one. alive bumps lastSeenAt; unknown leaves the row alone but still stamps lastProbedAt so a flaky page advances the cursor instead of jamming the rotation.
  • Composes with C1/C2. C3 decides which postings get probed; C1/C2 decide the verdict accuracy. Both are needed: C3 without good markers re-probes still-listed pages but still mis-reads a 200-but-closed banner; C1/C2 without C3 never look at still-listed pages at all.

Pillar C → A cascade — a closed listing closes its kanban card

The probe currently only flips JobPosting.status; it should also drive the linked application's status, the same way email ingest drives application state. Application.postingId String? @unique (schema.prisma:206) links one application back to the JobPosting it was tracked from, so a confirmed-closed posting id maps directly to its card.

Shared helper — one place, two callers. A new lib/applications/close-from-posting.ts:closeApplicationsForClosedPostings(postingIds, { at, source }):

  • Selects applications where postingId IN postingIds AND status is cascade-eligible — INTERESTED only (OQ7 decided 2026-06-09; the "I was watching, never applied, it closed" case the user describes). It deliberately does not touch APPLIED/ASSESSMENT/INTERVIEW*/OFFER/terminal cards: a listing routinely closes after you're already in the pipeline, so auto-closing those would wrongly bury live applications.
  • For each match: set status = "CLOSED" + lastUpdateAt = at, and create a STATUS_CHANGED ApplicationEvent (fromStatus, toStatus:"CLOSED", syncSource: source) — mirroring the manual PATCH event (app/api/applications/route.ts:158) so the timeline records why the card closed. Then broadcastEvent({ model:"Application", … }) so the board live-updates.
  • Because it only touches INTERESTED cards (which carry no newer status anchor), it sidesteps the stale-status ordering problem entirely — no guard needed.

Two callers (provenance via source):

  • Probe (auto) — in scheduler/jobs/job-watcher.ts, right after the confirmedClosedIds close updateMany (:490), call the helper with those ids and source:"probe". Runs per-tier against that tier's DB (application + posting share the DB), so no cross-tier concern.
  • Manual posting-close — in app/api/postings/[id]/route.ts PATCH, when the new status is "closed", call the helper with source:"ms" so marking a listing closed in the feed also closes its interested card. (Symmetry; OQ8 decided 2026-06-09 — both probe and manual cascade.)

Notification: the probe already fires a per-watchlist "N postings closed" standard bell row (job-watcher.ts:522). Cascade-closed cards fold into that summary rather than a per-application blast (a per-card email here would be exactly the kind of noise the circuit breaker exists to prevent) — OQ9 decided 2026-06-09.

Alternatives considered

  • Don't give Application its own CLOSED — derive the kanban's closed state from the linked JobPosting.status instead — rejected (decided 2026-06-09). Application gets its own CLOSED value because: (1) the board already separates Declined (I said no — withdrew / turned it down) from Rejected (they said no), and Closed (the opportunity itself ended — listing pulled / no longer accepting, no yes-or-no either way) is a third distinct terminal meaning that deserves its own status, not a flag derived from another model; (2) the "many apps have no linked posting" concern is minor in practice — there are few manual / email-only applications without a posting, and the user updates those by hand anyway. The cascade still reuses the existing auto-only JobPosting closed signal — through the liveness probe, never the email parser (the classifier stays out of CLOSED per OQ1, so inbound mail can't auto-close a card) — to drive that own status.
  • Derive the kanban link from postingId only (no new column) — rejected as insufficient: manually-added apps have no posting, and those are precisely the link-less rows the user is fighting. (OQ4 decided 2026-06-09: add the column.)
  • Aggressively re-probe all still-listed postings every tick — rejected: it spikes probe volume against a shared IP that arXiv-style cooldowns already taught us to respect. C3 instead does a budgeted, rotating sample (OQ5) — same Gap-A coverage, spread over many ticks; C1/C2 keep the verdicts accurate so the extra probing doesn't re-introduce false-closes.
  • Cascade by merely flagging the card (badge) instead of moving it to CLOSED — rejected: the user explicitly wants the probe to change the state, mirroring how email ingest moves cards. The INTERESTED-only guard + reversibility make the auto-move safe; a flag would still leave them deleting cards by hand.

Open Questions

Unresolved decisions

Resolve all OQs with the user before writing implementation code. Options are lettered (OQ5a, OQ5b, …); the Lean: / ✓ Decided. line names the pick by its letter first, then justifies it. Deciding one appends, never replaces: keep the lettered options under the question and swap its Lean:/Status: line for the ✓ Decided. line, flipping the banner to callout tip. All of OQ1–OQ9 are settled (2026-06-09).

OQ1 — Does CLOSED stay manual-only, or also join the email classifier enum?
  • (a) Manual-only. Add CLOSED to APPLICATION_STATUSES only and leave the lib/email-parser.ts classifier enum alone, so the status set is a superset of the classifier's. Pros: the LLM can never mislabel a rejection email as "closed"; CLOSED stays a deliberate human/probe action. Cons: an email that literally says "this role has been filled" won't move a card on its own — the user or the cascade has to close it.
  • (b) Add CLOSED to the classifier. Teach lib/email-parser.ts to emit CLOSED from closure-signalling emails, so inbound mail auto-closes a card. Pros: hands-free closure straight from the inbox. Cons: the classifier already folds "job closed" into rejection/withdrawal, so this would relabel real rejections as CLOSED and corrupt the funnel stats.

✓ Decided (2026-06-09): (a) manual-only — the misclassification risk in (b) is real and the closure signal is already captured as a rejection/withdrawal, so keeping the classifier untouched and reaching CLOSED only by hand or via the probe cascade is the safe call.

OQ2 — Kanban placement & treatment for the Closed column?
  • (a) Standalone 9th column. Add a terminal Closed column to pipelineColumns after Rejected, muted zinc hue. Pros: fully additive, reuses the existing column model, ships without touching board layout. Cons: the board grows to 9 columns — more horizontal scroll on narrow screens.
  • (b) Collapsed terminal group. Fold Closed/Rejected/Declined into one collapsible "terminal" group so the board stays narrow. Pros: keeps the board compact as terminal states multiply. Cons: a new grouping component to build + an extra click to reach terminal cards.

✓ Decided (2026-06-09): (a) the standalone muted column (zinc, e.g. bg-zinc-500/20 text-zinc-400 border border-zinc-500/20) — it's the additive, ship-today option; the grouping in (b) is deferred until the board actually feels too wide.

OQ3 — How are closed listings surfaced in the feed?
  • (a) Collapsed "Recently closed (N)" section. Add a collapsed section to the feed that queries ?status=closed (app/api/postings/route.ts:108). Pros: keeps the default feed focused on actionable new postings; closures stay reviewable in one click. Cons: closures are a click away rather than glanceable.
  • (b) Open/Closed segmented toggle. A toggle that swaps the feed query between new and closed. Pros: a clean single-list mental model. Cons: hides new postings whenever you're viewing closed; more view state to manage.
  • (c) Inline dimmed badge. Render closed postings inline with a dimmed "Closed" badge. Pros: zero navigation — everything in one list. Cons: pushes dead listings into the primary view, the exact noise the feed should suppress.

✓ Decided (2026-06-09): (a) the collapsed section — it surfaces closures without letting them crowd the new-postings feed that (b) and (c) both compromise.

OQ4 — Kanban link: new Application.url column, or derive from postingId only?
  • (a) Add Application.url String?. New nullable column, populated from posting.sourceUrl at track-as-application time and editable for manual apps; threaded onto the list payload + AppRecord. Pros: every app carries a clickable link — including manual apps and apps whose posting was later pruned (the user's exact complaint). Cons: an additive migration on both DBs (the pm2-stop-to-unlock dance).
  • (b) Derive from postingId only. Render the link from the linked JobPosting.sourceUrl, no new column. Pros: no migration. Cons: manually-added apps have no postingId, so they stay link-less — precisely the rows the user is fighting — and the link breaks if the posting is pruned.

✓ Decided (2026-06-09): (a) add the column — (b)'s no-migration win doesn't matter when it leaves the exact rows the user is complaining about link-less, and the column also survives posting pruning.

OQ5 — Add periodic re-probing of still-listed postings (Gap A), or rely on markers + manual close?
  • (a) Budgeted live re-probe (C3). Add a proactive scheduler sweep that re-probes still-listed status:"new" postings on a rolling, per-kind budget ordered by a new JobPosting.lastProbedAt. Pros: directly closes Gap A — a closed-but-still-listed job never goes stale, so nothing else ever re-checks it; gives continuous freshness. Cons: added probe volume on the shared IP (mitigated by the budget + cooldown) and a second additive migration (lastProbedAt).
  • (b) Measure-first + manual close. Ship only the marker work (C0/C1/C2) and let the new manual-CLOSE control cover the still-listed tail. Pros: no extra probing, smallest footprint. Cons: Gap A stays open — a closed-but-listed job is invisible until the user notices and closes it by hand.

✓ Decided (2026-06-09): (a) build C3, committed (not gated on C0) — markers only sharpen the probe on stale postings, so nothing else keeps the still-listed-but-closed population current, and the user prioritized continuous freshness. C0 still runs first, but only to size C3's budget and feed the markers. Designed in Pillar C → C3.

OQ6 — Can a later email override a manually-CLOSED kanban card?
  • (a) Hard terminal-lock. Once a card is CLOSED, no inbound email can move it — special-case CLOSED in the ingest path. Pros: a manual close is absolutely final. Cons: a legitimately newer email (the role reopened, an interview invite arrives) can't revive the card, and it needs bespoke ingest code nothing else needs.
  • (b) Stale-status anchor guard. Treat the manual close as a normal status anchor: the PATCH writes a STATUS_CHANGED at now(), and ingest's existing stale-guard already lets only a genuinely newer email move it. Pros: reuses machinery that's already there, zero special-casing; the rare newer-email override is usually legitimate. Cons: not an absolute lock — a sufficiently new email can still move the card.

✓ Decided (2026-06-09): (b) — guard-governed. The existing anchor logic already does the right thing, so a manual close at now() only yields to a genuinely newer email, getting the stickiness without the special-casing (a) would demand.

OQ7 — Cascade eligibility: which kanban statuses does a closed listing auto-close?
  • (a) INTERESTED-only. The cascade closes a linked card only while it's still INTERESTED. Pros: matches the "I was watching, never applied, it closed" case; never buries a card you've actively progressed (APPLIED and beyond). Cons: if you applied and the listing then vanishes, the card isn't auto-closed — you tidy it by hand.
  • (b) INTERESTED + APPLIED. Also auto-close cards you'd applied to. Pros: catches the "applied, then the role was pulled" case automatically. Cons: a listing routinely closes after you're already in the pipeline (interviews happen off-listing), so this would bury live applications, and it collides with the stale-status guard on cards carrying newer anchors.

✓ Decided (2026-06-09): (a) INTERESTED-only — auto-closing an APPLIED card would bury a live application a listing routinely outlives; the INTERESTED-only set is also the one that carries no newer status anchor, so it sidesteps the stale-status ordering problem entirely.

OQ8 — Cascade trigger: does a manual "mark closed" in the feed also cascade, or only the probe?
  • (a) Probe + manual both cascade. Both the auto-probe close and a user marking a listing closed in the feed run the same closeApplicationsForClosedPostings helper (provenance probe vs ms). Pros: symmetric — closing a listing always closes its interested card, however the closure was decided. Cons: a mistaken manual feed-close also closes the linked card (reversible by drag-back, but an extra surprise).
  • (b) Probe-only. Only the liveness probe cascades; a manual feed-close leaves the linked card alone. Pros: manual feed actions stay scoped to the feed, no surprise card movement. Cons: asymmetric — after manually closing a listing the user must close its card separately, re-introducing by-hand bookkeeping.

✓ Decided (2026-06-09): (a) both cascade — the whole point is to stop hand-closing cards, so a manual posting-close should behave like a probe close; the move is INTERESTED-only and reversible, so the surprise cost is small.

OQ9 — Cascade notification: how is a cascade-closed card surfaced?
  • (a) Fold into the closure summary. Count cascade-closed cards in the probe's existing per-watchlist "N postings closed" bell row (job-watcher.ts:522). Pros: reuses the existing notification, no new surface, stays inside the circuit-breaker's noise budget. Cons: less granular — the user sees "N closed," not which cards moved.
  • (b) Per-application row. Emit a dedicated notification per cascade-closed card. Pros: explicit per-card visibility of what auto-moved. Cons: exactly the per-card blast the notification circuit breaker exists to prevent — noisy when a watchlist closes many at once.

✓ Decided (2026-06-09): (a) fold into the existing summary — a per-card blast (b) is the noise the circuit breaker was built to stop, and the rollup is enough for an INTERESTED-only, reversible move.

Execution — agent track briefs

The checklist above is the what; this is how to dispatch it across agents. The plan is re-cut so the three parallel tracks are fully file-disjoint: a serial foundation commit absorbs every file the contention table flags as shared (prisma/schema.prisma, lib/schemas/applications.ts, lib/repositories/applications.ts, app/api/applications/route.ts), after which Tracks A/B/C never touch the same file. Merge is therefore low-conflict by construction — the only residual shared file is the append-only SUITES list in scripts/pre-push.sh.

Shared guardrails (every agent).
  • No AI commit attribution — no Co-Authored-By / "Generated with" lines.
  • Always git commit -- <explicit paths> — a bare commit sweeps in other tracks' staged files off the shared index.
  • Stay in your lane — edit only files in your "Owns" list; if you think you need another track's file, stop and flag it.
  • Don't run npm run dev/start — PM2 owns the ports; verify against the running dev server (:4101).
  • DB path: DATABASE_URL="file:./dev.db", never file:./prisma/dev.db (Prisma resolves relative file: URLs from prisma/).
