Closed-jobs feature

A Closed state on both the applications kanban and the job-listings feed — set by hand or driven by the close-detection probe, one-click "open in new tab" on kanban cards, and a sharper probe so fewer dead listings sit around as live.

Context & problem
Status: design — OQ1–OQ4 decided (2026-06-09); OQ5–OQ7 still open. Three independent pillars, all requested in one ask. The remaining open questions (OQ5–OQ7) must be resolved with the user before any implementation code is written.

Problem

Three related pain points around job opportunities that have closed (no longer taking applications):

  1. No manual "closed" state the user can set or see.
    • Applications kanban — ApplicationsKanbanCard.tsx:43 defines 8 columns (Interested → Applied → Assessment → Interviewing → Offer → Accepted → Declined → Rejected). There is no terminal "Closed" column, so when an opportunity the user was tracking closes, they currently delete the card — losing the history. The status set itself (APPLICATION_STATUSES, lib/schemas/applications.ts:7) has no CLOSED value.
    • Job-listings feed — JobPosting.status does already support "closed" (JOB_POSTING_STATUSES, lib/schemas/watchlists.ts), but it is only ever set automatically by the scheduler's liveness probe (job-watcher.ts:496). There is no manual "mark closed" control, and the feed only renders status:"new" (NewPostingsCard.tsx:117), so a closed listing simply disappears — it can't be reviewed.
  2. The job link is hard to open. When the user opens a kanban card, the posting URL is shown by PostingSourceLine (ApplicationDetailOverlay.tsx:46) — but only when app.postingId is set (:361). Manually-added applications carry no URL at all: AppRecord (ApplicationsKanbanCard.tsx:9) has no url field, so the kanban card face has nothing to click, and the user ends up selecting + copying a URL by hand or right-click → open-in-new-tab.
  3. Close-detection misses "closed but the link still exists" jobs. The scheduler only probes a posting when it is stale — absent from the latest fetch and lastSeenAt > 6h (job-watcher.ts:461). A job that is closed but still listed in the source feed never goes stale → never gets probed → never closes. And even when probed, probeViaHttpStatus (liveness.ts:134) returns "alive" on any 2xx that doesn't hit a closure marker — and the per-ATS marker lists are self-described as "initial guesses" (liveness.ts:79). So a "200-but-closed" page (job content intact, banner says "no longer accepting applications") reads as alive. The user reports "missing lots of jobs that are closed but the links still exist," which matches both gaps.

Goals & non-goals

  • Goal: A CLOSED state the user can set and see on both the applications kanban and the job-listings feed.
  • Goal: A confirmed-closed listing cascades to its linked kanban card — the close-detection probe (and a manual posting-close) auto-moves a still-INTERESTED application to CLOSED, the same way email ingest drives application state. No more deleting cards by hand.
  • Goal: One-click "open in new tab" for the job link from the kanban (card face + detail overlay), for every application — including manually-added ones.
  • Goal: Measurably fewer missed closures — close-detection that catches the closed-but-still-listed and 200-but-closed cases the current probe misses.
  • Non-goal (this doc): An auto-classifier that emits CLOSED from emails. CLOSED is a manual decision; the email classifier is out of scope (see OQ1).
  • Non-goal (lean, see OQ5): Periodic re-probing of every still-listed posting. That doubles probe volume against shared-IP rate limits; deferred in favour of "measure first + manual close for the long tail."
  • Non-goal: Reviving a closed posting back to new if the source re-lists it. Closed stays closed (matches today's behaviour; the manual control inherits it).

Design

Pillar A — manual CLOSED state on both surfaces

A1 · Applications kanban

The kanban renders directly off APPLICATION_STATUSES and a column table, so this is an additive change with no new abstractions:

  • Add 'CLOSED' to APPLICATION_STATUSES (lib/schemas/applications.ts:7). The read shape is status: z.string() (:47) so existing rows are unaffected; the POST/PATCH paths use ApplicationStatusSchema, which gains the value for free.
  • Add a terminal Closed column to pipelineColumns (ApplicationsKanbanCard.tsx:43), after Rejected, with a neutral/archived hue distinct from Rejected's slate — e.g. bg-zinc-500/20 text-zinc-400 border border-zinc-500/20 (placement & treatment = OQ2).
  • The detail-overlay status buttons render one per APPLICATION_STATUSES (ApplicationDetailOverlay.tsx:367), so CLOSED appears there automatically — no edit needed.
  • Drag-to-column and the button both call PATCH /api/applications, already ApplicationStatusSchema.optional() — no route change. The PATCH already writes a STATUS_CHANGED anchor at now() (route.ts:158), so a manual close is correctly the latest status anchor — the stale-email guard then protects it (an older email can't un-close it; see OQ6).
  • CLOSED is set by hand or by the cascade — the same status value is reached when a linked listing is confirmed closed (see Pillar C → A cascade below). Both paths write a STATUS_CHANGED event so the timeline shows why (provenance via syncSource: ms for manual, probe for auto).
  • Keep CLOSED out of the LLM classifier enum in lib/email-parser.ts (the comment at applications.ts:3 says the set "mirrors" the classifier — that becomes "superset of"; update the comment). See OQ1.

A2 · Job-listings feed

The status value and the API both already exist — this is a UI + surfacing change:

  • Mark-closed control on PostingRow (NewPostingsCard.tsx:626), alongside the existing hide/track buttons (:673). It calls api.postings.update(id, { status: "closed" }) — the PATCH route already accepts JobPostingStatusSchema (app/api/postings/[id]/route.ts).
  • Surface closed listings. The feed requests status:"new" only (:117); the list route already supports ?status=closed (app/api/postings/route.ts:108). Render closed ones in a way that's out of the way but reviewable — mechanism is OQ3 (collapsed "Recently closed (N)" section vs. an Open/Closed toggle vs. inline dimmed badge).
  • Stickiness is already correct. The seen-again path bumps lastSeenAt but never touches status (job-watcher.ts:330), and stale-candidate selection excludes closed/hidden (:466) — so a manually-closed posting that's still in the feed will not be revived or re-probed. No new guard needed; verify with a smoke (P1.2.3).

Pillar B — one-click job link from the kanban

The root issue is that the kanban has no link to click for most apps. Fix the data first, then the UI:

  • Data: give every application a job URL. A manually-added app has no postingId, so deriving from JobPosting.sourceUrl alone leaves exactly the user's problem rows link-less. Add Application.url String? (populated from posting.sourceUrl at track-as-application time, and editable for manual apps), then thread it onto the applications list payload + AppRecord (ApplicationsKanbanCard.tsx:9). Whether a dedicated column is load-bearing vs. derive-only = OQ4.
  • UI — card face: a small ExternalLink icon button (mirroring NewPostingsCard.tsx:643) rendered when url is present, as a real <a href target="_blank" rel="noopener noreferrer">, with onClick stopPropagation so it doesn't open the detail overlay.
  • UI — detail overlay: promote the existing PostingSourceLine (ApplicationDetailOverlay.tsx:46) to render for any app with a URL (not just postingId), styled as an unmistakable one-click "Open posting ↗" button rather than selectable URL text.

Scoped, lowest-risk-first fixes (Sal's minimal-v1 preference — measure before widening, since a too-aggressive probe re-introduces the false-close behaviour the probe gate was built to kill):

  • C0 · Diagnose (do this first). A probe-audit script under scripts/tests/probes/ that samples current status:"new" postings per kind, re-probes them via probePostingLiveness, and reports the closed-but-still-listed rate + which markers did/didn't fire. This tells us whether the misses are Gap A (not-probing-live) or Gap B (bad markers) — so C1–C3 are data-driven, not guesses.
  • C1 · Expand & curate closure markers (liveness.ts:79) informed by C0, and extend the "require an alive-marker before declaring alive" pattern (already used for LinkedIn :210 / Indeed :227) to the other HTML-scraping kinds, so a 200-but-closed page falls to unknown rather than alive. Additive, conservative.
  • C2 · Prefer structured ATS state. Greenhouse/Lever already probe the JSON API first (liveness.ts:234,:248); a removed Greenhouse/Lever job 404s on the board API. Audit (C0) whether closed-but-listed jobs expose an explicit open/closed flag in those JSON payloads (and Ashby/Workday CXS), and trust that over HTML heuristics where present.
  • C3 · (conditional, OQ5) periodic live re-probe of a budgeted sample of still-listed postings to catch Gap A directly — only if C0 shows a large closed-but-listed population that markers can't catch. Otherwise the new manual CLOSE control is the deterministic complement for that long tail.

Pillar C → A cascade — a closed listing closes its kanban card

The probe currently only flips JobPosting.status; it should also drive the linked application's status, the same way email ingest drives application state. Application.postingId String? @unique (schema.prisma:206) links one application back to the JobPosting it was tracked from, so a confirmed-closed posting id maps directly to its card.

Shared helper — one place, two callers. A new lib/applications/close-from-posting.ts:closeApplicationsForClosedPostings(postingIds, { at, source }):

  • Selects applications where postingId IN postingIds AND status is cascade-eligible — lean: INTERESTED only (the "I was watching, never applied, it closed" case the user describes). It deliberately does not touch APPLIED/ASSESSMENT/INTERVIEW*/OFFER/terminal cards: a listing routinely closes after you're already in the pipeline, so auto-closing those would wrongly bury live applications. Eligibility set = OQ7.
  • For each match: set status = "CLOSED" + lastUpdateAt = at, and create a STATUS_CHANGED ApplicationEvent (fromStatus, toStatus:"CLOSED", syncSource: source) — mirroring the manual PATCH event (app/api/applications/route.ts:158) so the timeline records why the card closed. Then broadcastEvent({ model:"Application", … }) so the board live-updates.
  • Because it only touches INTERESTED cards (which carry no newer status anchor), it sidesteps the stale-status ordering problem entirely — no guard needed.

Two callers (provenance via source):

  • Probe (auto) — in scheduler/jobs/job-watcher.ts, right after the confirmedClosedIds close updateMany (:490), call the helper with those ids and source:"probe". Runs per-tier against that tier's DB (application + posting share the DB), so no cross-tier concern.
  • Manual posting-close — in app/api/postings/[id]/route.ts PATCH, when the new status is "closed", call the helper with source:"ms" so marking a listing closed in the feed also closes its interested card. (Symmetry; gated by OQ7.)

Notification: the probe already fires a per-watchlist "N postings closed" standard bell row (job-watcher.ts:522). Lean: fold cascade-closed cards into that summary rather than a per-application blast (a per-card email here would be exactly the kind of noise the circuit breaker exists to prevent). Exact surface = part of OQ7.

Alternatives considered

  • Reuse the existing auto-only JobPosting closed for the kanban too — rejected: the kanban tracks my application's lifecycle (a distinct model with its own status set); overloading the posting status would conflate "the listing is gone" with "I closed out my pursuit."
  • Derive the kanban link from postingId only (no new column) — rejected as insufficient: manually-added apps have no posting, and those are precisely the link-less rows the user is fighting. Kept alive as OQ4 in case a derive-only v1 is preferred.
  • Aggressively re-probe all postings every tick — rejected for v1: it doubles probe volume against a shared IP that arXiv-style cooldowns already taught us to respect, and risks re-introducing false-closes. Measure first (C0); manual close covers the tail.
  • Cascade by merely flagging the card (badge) instead of moving it to CLOSED — rejected: the user explicitly wants the probe to change the state, mirroring how email ingest moves cards. The INTERESTED-only guard + reversibility make the auto-move safe; a flag would still leave them deleting cards by hand.

Rollback

Enum/UI/probe changes are revert + npm run build + pm2 restart; CLOSED is just a string, so existing rows are untouched on revert. The Application.url migration is additive + nullable — no down-migration data loss; a revert leaves a harmless unused column. Probe changes are scheduler-only behaviour; revert + restart both schedulers.

Open Questions

Unresolved decisions

Resolve all OQs with the user before writing implementation code. Options are lettered (OQ5a, OQ5b, …); the Lean: / ✓ Decided. line names the pick by its letter first, then justifies it. Deciding one appends, never replaces: keep the lettered options under the question and swap its Lean:/Status: line for the ✓ Decided. line, flipping the banner to callout tip. OQ1–OQ4 are settled (2026-06-09); OQ5–OQ7 remain open.

OQ1 — Does CLOSED stay manual-only, or also join the email classifier enum?
  • (a) Manual-only (APPLICATION_STATUSES a superset of the classifier enum).
  • (b) Add CLOSED to the lib/email-parser.ts classifier too.

✓ Decided (2026-06-09): (a) manual-only — APPLICATION_STATUSES becomes a superset of the classifier enum and lib/email-parser.ts is left untouched, because an email announcing a closed job is already captured as a rejection/withdrawal and teaching the classifier a CLOSED label would only risk mislabeling real rejections.

OQ2 — Kanban placement & treatment for the Closed column?
  • (a) A standalone 9th column after Rejected (muted zinc).
  • (b) A collapsed "terminal" group (Closed/Rejected/Declined) so the board doesn't grow to 9 columns wide.

✓ Decided (2026-06-09): (a) a standalone muted column after Rejected (zinc hue, e.g. bg-zinc-500/20 text-zinc-400 border border-zinc-500/20) — it's fully additive and reuses the existing column model, so it ships without restructuring the board; the collapsed terminal group (b) is deferred until the board actually feels too wide.

OQ3 — How are closed listings surfaced in the feed?
  • (a) A collapsed "Recently closed (N)" section.
  • (b) An Open/Closed segmented toggle that swaps the query.
  • (c) Inline with a dimmed "Closed" badge.

✓ Decided (2026-06-09): (a) a collapsed "Recently closed (N)" section (queries ?status=closed via app/api/postings/route.ts:108) — it keeps the default feed focused on actionable new postings while still letting the user review closures in one click, whereas the toggle (b) and inline badge (c) both push closed noise into the primary view.

OQ4 — Kanban link: new Application.url column, or derive from postingId only?
  • (a) Add a nullable Application.url column (covers manual apps + pruned postings).
  • (b) Derive-only from postingId (no migration, but link-less manual apps remain).

✓ Decided (2026-06-09): (a) add the nullable Application.url column, populated from posting.sourceUrl at track-as-application time and editable for manual apps — derive-only (b) was rejected because it leaves manually-added apps (the user's exact complaint) with nothing to link to, and a real column also survives the posting being pruned. Needs an additive migration on both DBs (P2.1.1).

OQ5 — Add periodic re-probing of still-listed postings (Gap A), or rely on markers + manual close?
  • (a) A budgeted live re-probe in the scheduler.
  • (b) Measure-first + manual close.

Lean: (b) — no live re-probe in v1: run the C0 audit, widen markers (C1/C2), and only revisit (a) if the audit shows a large closed-but-listed population, because a blanket re-probe doubles probe volume against a shared IP and risks re-introducing the false-closes the probe gate was built to prevent. Status: open (gated on C0 data).

OQ6 — Can a later email override a manually-CLOSED kanban card?
  • (a) A hard terminal-lock.
  • (b) Governed by the existing stale-status anchor guard.

Lean: (b) — guard-governed: a manual close writes a STATUS_CHANGED anchor at now(), so only a genuinely newer email can move it (rare, and usually legitimate when it happens), which gets the desired stickiness without the special-casing a hard lock (a) would demand. Status: open.

OQ7 — Cascade scope: which kanban statuses does a closed listing auto-close, and does manual posting-close cascade too?
  • (a) Eligibility — INTERESTED only, vs. INTERESTED + APPLIED.
  • (b) Trigger — probe-only, vs. a manual "mark closed" in the feed also cascading to the card.
  • (c) Notification — fold into the per-watchlist closure summary, vs. a per-card row.

Lean: (a) INTERESTED-only, because auto-closing an APPLIED card would bury a live application a listing routinely outlives; (b) both probe and manual posting-close cascade, for symmetry between the two ways a listing gets closed; (c) fold into the existing closure summary, since a per-card blast is exactly the noise the notification circuit breaker exists to prevent. Status: open.
