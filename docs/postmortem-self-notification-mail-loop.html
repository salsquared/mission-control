mission-control

Post-Mortem — The Self-Notification Mail Loop

On 2026-06-02, mission-control sent itself ~552 emails about a single CalSAWS interview in ~20 minutes, created roughly that many Google Calendar events, and burned ~$0.66 of Gemini tokens — entirely on its own outbound mail, classified as if it were new. This is the full account: what happened, why every existing defense failed to stop it, what was cleaned up, what was not, and how to keep it from ever recurring.

Summary (TL;DR)

What happened

mission-control sends its application notifications (OFFER / REJECTION / INTERVIEW_SCHEDULED / ASSESSMENT_REQUESTED) from → to the user's own Gmail (lib/email/send.ts:notificationToEmail — both from and to are user.email). Those emails land back in the same INBOX the Gmail push webhook watches. With no marker to distinguish our own outbound mail from a genuine inbound email, the webhook re-ingested mission-control's own notification, the LLM classified it as a fresh interview, fired another notification, sent another email — and looped.

One real CalSAWS interview reminder seeded a self-amplifying loop that ran 552× in ~20 minutes (2026-06-02 03:29–03:49 UTC, ~1,102 webhook deliveries in the window). It was not stopped by any of our code — only by Gmail's own outbound send throttle (User-rate limit exceeded … (Mail sending)). Until the fix shipped, it was dormant, not solved: any new genuine notification could have re-triggered it.

The damage, in one line each

  • Real inbox: ~552 actual [mission-control] … emails delivered to the user's Gmail.
  • Real calendar: on the order of ~552 Google Calendar events created (one per looped INTERVIEW_SCHEDULED event — each a distinct ApplicationEvent, so gcal's idempotency did not collapse them).
  • Tokens: ~$0.66 of Gemini spend on uncached classifier calls — a working LLM cache would have made it ~$0.00.
  • Database: 1,229 prod + 668 dev junk ApplicationEvent rows and 559 prod + 323 dev Notification rows across 4 prod apps, since cleaned.

Status

The code fix is shipped and live (commit 7ae5a6c, both tiers redeployed). The database is clean. But two side-effects that escaped the database — the real Google Calendar events and the real Gmail messages — were never cleaned and have no programmatic cleanup path (see §9). Those are the only open items.

Root cause & why every defense failed

The loop

The architecture has a closed cycle: notifications are self-addressed, the inbox is watched, and the webhook ingests everything in the inbox — including our own outbound mail. Nothing marked an email as "this is mine, don't re-ingest it."

Why the per-message dedup didn't stop it

Ingest is idempotent per message via @@unique([applicationId, emailMsgId, kind]). But every loop iteration was a brand-new outbound email with a brand-new msgId, so each re-ingest was — to that constraint — a legitimately distinct message. The dedup that protects against webhook redelivery is structurally blind to a loop that generates new messages.

Why quiet hours didn't throttle it

INTERVIEW_SCHEDULED dispatches at the critical tier (lib/repositories/applicationEvents.ts → dispatchNotification({ tier: "critical" })), and critical tier bypasses quiet hours by design (lib/notifications/dispatch.ts: the quiet-hours gate is skipped when tier === "critical"). So the loop ran at full speed regardless of time of day.

The amplifier — free-mail was not a dedup blocklist root

The sender-domain dedup fallback (lib/applications/sender-domain.ts) blocked multi-tenant ATS roots (greenhouse.io, lever.co, …) but not consumer free-mail roots like gmail.com. The user's own notifications are from their gmail.com address, so every re-ingested self-notification resolved to senderDomain = gmail.com, and the fallback funneled them onto whatever app happened to carry that domain. That's how a CalSAWS loop leaked junk events onto Delta Air Lines, the LA Dodgers, and a test row too.

The decisive failure was the missing self-identification. Dedup, quiet hours, and the sender-domain heuristic are all downstream mitigations — none of them is designed to recognize "this email is mine." The system had no way to tell its own voice from the world's.

The $0.66 — token-cost mechanism

Where the money went

Every re-ingested email ran the LLM classifier lib/email-parser.ts:parseApplicationEmail (model: gemini-3.1-flash-lite; ~600 input + ~180 output tokens per call — the 3 KB-trimmed body plus prompt wrapper, returning the full application schema). At ≥552 calls (and likely ~2× that, since the ~1,102 webhook deliveries imply both the dev and prod tiers were ingesting the same mailbox), that adds up to the observed ~$0.66. Classifier retries also pay the cost.

Why the cache barely helped (corrected)

parseApplicationEmail routes through llmCached(), whose key is content-addressed: model + rendered prompt + output-schema fingerprint. The rendered prompt includes the email's own send time as the date-resolution anchor (sentAt.toISOString()), and each of the 552 looped self-notifications was sent at a distinct moment — so they had distinct cache keys and were not collapsible into one. (An earlier draft of this doc wrongly claimed a working cache would have driven the cost to ~$0.00 — corrected here after noticing the parsed interview times in the calendar varied per call.) What the cache would have deduped is the part that genuinely repeats: the Gmail push fan-out delivers each single message to both the dev and prod webhooks ~simultaneously (same msgId / sentAt / body → same key → single-flight: one tier leads, the other follows). That roughly halves the ~1,102 calls back toward ~552, so a healthy cache would have made the loop cost ~$0.33, not ~$0.00. The cache is a duplicate-suppressor, not a loop-breaker — only the self-notification guard prevents the loop.

Why the calendar events landed at many different times. The cache being off meant dev and prod parsed each looped email independently, doubling the parses — but the spread of times (the ~552 events landed on 399 distinct times across four months, not one) comes from a date feedback loop plus a timezone-serialization flip in the LLM's output. Full forensic breakdown in §6.
The cache was disabled at the time — and had been for days. Prod logs showed [llm-cache] disabled … no such column: value. This was schema drift, not the better-sqlite3 ABI mismatch first suspected: the 2026-05-31 refactor that re-homed the LLM cache onto the generic base renamed its payload column result → value and added expiry, but kept the same table name in the same data/llm-cache.db file. CREATE TABLE IF NOT EXISTS no-ops on the pre-existing old-shape table, so the first prepared statement referencing value threw no such column: value, which the best-effort store caught and degraded to uncached, permanently and invisibly. With the cache live, the loop's token cost would have been roughly halved (the simultaneous dev+prod fan-out of each message deduped), not eliminated — the distinct per-message send-time anchors kept the 552 looped emails from collapsing further. The disabled cache amplified the bill ~2×; it didn't cause it. (Fixed 2026-06-04 — the base now detects column drift on init and rebuilds the table rather than silently disabling; see §10.)

Rate limiting did not help the cost

acquireGeminiSlot() (default 12 req/min) paced the calls but does not reduce per-call token spend — cost is calls × tokens-per-call regardless of how they're spread over time. Pacing only changed how long the loop took to burn the money, not how much it burned.

Why it drifted — three bugs compounding

So: is it "the LLM parsing dates wrong" or "something else"? Both, specifically. The LLM mis-serializes the time, but the app keeps handing the LLM its own previous output to re-parse. Neither alone produces a four-month walk.

  • 1 — Feedback fuel: the notification body re-encodes the parsed time. lib/repositories/applicationEvents.ts:61 builds the INTERVIEW_SCHEDULED notification body as event.scheduledAt.toLocaleString(). So every self-notification email literally contained the previous iteration's parsed timestamp as M/D/YYYY, h:mm:ss AM/PM. Proof: 550 of the 552 events' rawText (the wording the LLM says it read) are in that exact toLocaleString shape; the only 2 that aren't are the genuine human seeds (06/03/2026 at 11:00 AM and an earlier 04/22/2026 at 9:15 AM).
  • 2 — Timezone flip: the LLM serializes the wall-clock time as UTC half the time. The parser asks for ISO 8601. Given a zone-less wall-clock time (5/9/2026, 9:00:00 AM), the LLM inconsistently emitted it as local (…T09:00:00) or UTC (…T09:00:00Z). Reading a Pacific wall-clock time as UTC shifts the stored instant earlier by Pacific's offset. Proof: every drifted rawText resolved to exactly two times, 7 h apart (106 rawTexts, the PDT dates) or 8 h apart (36, the PST dates) — i.e. precisely Pacific's UTC offset for that date's season. A random misread wouldn't be season-correct; a local↔UTC confusion is, exactly.
  • 3 — 552 iterations random-walk the date backward. Each UTC-misread subtracts ~7–8 h; the toLocaleString of the new (earlier) time then feeds the next email, so the error compounds. Crossing midnight again and again marched the date backward ~117 days, from the true 6/3 to 2/6. The "clusters" you saw are the stops on that walk: each intermediate time is a small cluster, doubled by its local-vs-UTC pair → 399 distinct values.
Latent beyond the loop. Bug #2 isn't loop-specific — even a single genuine interview email can land 7–8 h off if the LLM appends a stray Z. The loop merely sampled it 552× and made it impossible to miss. Hardening the date serialization (and not feeding a re-parseable timestamp back into notification bodies) is tracked in §10.

How this was reconstructed

The loop's DB rows were deleted in the cleanup (§8) and the live calendar is now partially hand-deleted, so this was derived from the pre-cleanup backup — ~/backups/mission-control/prod.db.pre-mailloop-cleanup-20260602T042424Z — grouping the 552 CalSAWS INTERVIEW_SCHEDULED rows by scheduledAt and by notes (= the LLM's rawText). Each surviving calendar event's description still carries that rawText, so any remaining events can be spot-checked the same way.

The fix

Two-signal self-notification guard (commit 7ae5a6c)

The real fix gives the system a way to recognize its own voice. New module lib/applications/self-notification.ts defines the shared constants and the predicate:

export const MC_NOTIFICATION_HEADER = "X-Mission-Control";
export const MC_NOTIFICATION_HEADER_VALUE = "notification";
export const MC_SUBJECT_PREFIX = "[mission-control] ";

export function isSelfNotificationEmail(args: { subject?: string | null; mcHeader?: string | null }): boolean {
  if ((args.mcHeader ?? "").trim().toLowerCase() === MC_NOTIFICATION_HEADER_VALUE) return true; // primary
  if ((args.subject ?? "").startsWith(MC_SUBJECT_PREFIX)) return true;                          // fallback
  return false;
}
  • Stamp on send. lib/email/send.ts now emits an X-Mission-Control: notification header on every notification (EmailMessage gained a headers? field; buildRfc822Message writes them into the RFC 822 header block) and uses the shared MC_SUBJECT_PREFIX.
  • Skip on ingest. lib/applications/ingest.ts checks isSelfNotificationEmail and returns { action: "skipped", reason: "self_notification" } before the looksRelevant filter and the LLM classifier — so a self-notification costs zero tokens. The header is the primary signal; the subject prefix is the fallback for notification mail already sitting in the inbox that pre-dates the header.
  • Kill the amplifier. New FREE_MAIL_DOMAINS in lib/applications/relevance.ts (20 consumer roots) is folded into sender-domain.ts's BLOCKED_ROOTS, so gmail.com et al. resolve to null and never funnel mail onto one app. This is a dedup-blocklist-only change — deliberately not added to SENDER_DOMAINS, which doubles as a positive relevance signal (widening it would flood the classifier with personal mail).

Defense in depth — two independent signals

The header is robust to a user genuinely titling a real email [mission-control] …; the subject prefix catches notifications that predate the header. Either one trips the skip.

Tests

New hermetic smoke self-notification-loop-guard-smoke.ts (20/20) — including a closed-loop round-trip that asserts what notificationToEmail produces is exactly what isSelfNotificationEmail flags after passing through buildRfc822Message — plus an extended sender-domain-smoke.ts (free-mail → null). Both wired into scripts/pre-push.sh; full hermetic gate 101/101, tsc --noEmit clean.

Unaddressed remediation

The clutter the user still sees

The database cleanup removed the rows, but the loop's two external side-effects were never cleaned and currently have no programmatic cleanup path. This is the gap between "the bug is fixed" and "the mess is gone."

~552 orphaned Google Calendar events remain. deleteEventFromGcal(userId, gcalEventId) exists in lib/calendar/sync.ts, but it is only called from the manual per-event DELETE handler in app/api/applications/events/route.ts — there is no batch cleanup. Worse, the calendar event IDs lived in ApplicationEvent.gcalEventId, and those rows were deleted in §8, so the IDs are lost. Programmatic cleanup-by-stored-ID is no longer possible; it now requires a Calendar API sweep that matches the loop window / title pattern, or manual deletion in Google Calendar.

~552 orphaned Gmail messages remain in the inbox. Notifications do not persist the Gmail messageId of what they sent (Notification records emailSentAt / emailError but not the message ID), so there is no reverse mapping to delete them. Cleanup is a manual Gmail search-and-delete (e.g. subject:"[mission-control]" in the loop window), or a one-off script over the Gmail API matching that query.

The reason for the gap

Both side-effects are fire-and-forget: the system writes to Calendar / Gmail and records only a success checkpoint, never a handle it could use to undo. Cleaning the DB rows is what severed the only stored link to the external artifacts. A future "delete-by-handle" capability (store the sent messageId; sweep gcal by stored IDs before deleting their rows) is what would have made this recoverable.

Prevention & follow-ups

Shipped (this incident)

  • Self-notification guard — the loop is structurally broken: our own mail is recognized and skipped before classification.
  • Free-mail dedup block — the sender-domain amplifier that spread junk across apps is closed.
  • Hermetic coverage — a closed-loop round-trip smoke now fails the push if the guard ever regresses.

Recommended hardening (not yet done)

  • Notification circuit breaker (defense in depth). The guard fixes this loop, but a per-app / per-window cap on outbound notifications (e.g. "no more than N sends for one application in M minutes") would catch the next unforeseen amplification before it sends 552 emails. The decisive lesson is that the only thing that stopped this was an external throttle — we should own that limit.
  • Fix the LLM cache [done — 2026-06-04]. Root cause was schema drift (column result → value + added expiry from the 2026-05-31 refactor), not a better-sqlite3 ABI mismatch — and the failure was silent: a drifted column threw at db.prepare(), was caught as a generic init error, and disabled the cache for days. Fixed in lib/shared-sqlite-cache.ts:reconcileSchema(): on init the base now compares the on-disk columns to the expected set and, on a mismatch, drops + rebuilds the table loudly (the data is throwaway) instead of disabling. A new hermetic smoke shared-cache-schema-drift-smoke.ts seeds the old schema and asserts the self-heal + warning + actual caching. This repairs the live cache on the next deploy and makes any future drift self-correct rather than billing silently. Note its scope: the cache suppresses duplicate-input spend (e.g. the dev+prod fan-out of one message), so it would have ~halved this loop's cost — but it would not have stopped the loop, whose emails each had a distinct send-time anchor → distinct key. The self-notification guard is the loop fix; the cache is the duplicate-spend defense.
  • Make date extraction timezone-safe (and stop feeding the parser its own output). The four-month calendar drift (§6) had two roots beyond the loop itself: (a) the INTERVIEW_SCHEDULED notification body is scheduledAt.toLocaleString() (lib/repositories/applicationEvents.ts:61), so the email re-encoded the parsed time for the next parse to chew on; and (b) the LLM inconsistently serialized a zone-less wall-clock time as UTC vs local (a 7–8 h flip). The self-notification guard kills the feedback, but the timezone sloppiness is latent — a single genuine interview email can still land 7–8 h off. Pin the parser to emit zone-explicit ISO (or normalize extracted times against the email's own timezone), and reconsider embedding a re-parseable timestamp in notification bodies.
  • Store undo handles. Persist the sent notification's Gmail messageId, and sweep gcal events by stored ID before deleting their ApplicationEvent rows — so a future cleanup can actually remove external artifacts (see §9).
  • One-off cleanup of the existing clutter. A throwaway Gmail + Calendar API sweep over the 2026-06-02 03:29–03:49 UTC window matching [mission-control] / the CalSAWS interview title would clear the ~552 calendar events and inbox messages that still remain.
