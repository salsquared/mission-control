Post-Mortem — The Self-Notification Mail Loop

On 2026-06-02, mission-control sent itself ~552 emails about a single CalSAWS interview in ~20 minutes, created roughly that many Google Calendar events, and burned ~$0.66 of Gemini tokens — entirely on its own outbound mail, classified as if it were new. This is the full account: what happened, why every existing defense failed to stop it, what was cleaned up, what was not, and how to keep it from ever recurring.

Summary (TL;DR)

What happened

mission-control sends its application notifications (OFFER / REJECTION / INTERVIEW_SCHEDULED / ASSESSMENT_REQUESTED) from → to the user's own Gmail (lib/email/send.ts:notificationToEmail — both from and to are user.email). Those emails land back in the same INBOX the Gmail push webhook watches. With no marker to distinguish our own outbound mail from a genuine inbound email, the webhook re-ingested mission-control's own notification, the LLM classified it as a fresh interview, fired another notification, sent another email — and looped.

One real CalSAWS interview reminder seeded a self-amplifying loop that ran 552× in ~20 minutes (2026-06-02 03:29–03:49 UTC, ~1,102 webhook deliveries in the window). It was not stopped by any of our code — only by Gmail's own outbound send throttle (User-rate limit exceeded … (Mail sending)). Until the fix shipped, it was dormant, not solved: any new genuine notification could have re-triggered it.

The damage, in one line each

  • Real inbox: ~552 actual [mission-control] … emails delivered to the user's Gmail.
  • Real calendar: on the order of ~552 Google Calendar events created (one per looped INTERVIEW_SCHEDULED event — each a distinct ApplicationEvent, so gcal's idempotency did not collapse them).
  • Tokens: ~$0.66 of Gemini spend on uncached classifier calls — a working LLM cache would have made it ~$0.00.
  • Database: 1,229 prod + 668 dev junk ApplicationEvent rows and 559 prod + 323 dev Notification rows across 4 prod apps, since cleaned.

Status

The code fix is shipped and live (commit 7ae5a6c, both tiers redeployed). The database is clean, and as of 2026-06-04 the two external side-effects are cleaned too: the calendar turned out to be recoverable (only ~28 of the looped events ever reached Google Calendar — the loop synced only future-dated events while the date-drift walked backward into the past — and those were deleted), and the looped Gmail messages had already been deleted by the user (see §9). The only open items now are the defensive hardening follow-ups — a notification circuit breaker, timezone-safe date extraction, and undo handles — planned in §11.

Root cause & why every defense failed

The loop

The architecture has a closed cycle: notifications are self-addressed, the inbox is watched, and the webhook ingests everything in the inbox — including our own outbound mail. Nothing marked an email as "this is mine, don't re-ingest it."

Why the per-message dedup didn't stop it

Ingest is idempotent per message via @@unique([applicationId, emailMsgId, kind]). But every loop iteration was a brand-new outbound email with a brand-new msgId, so each re-ingest was — to that constraint — a legitimately distinct message. The dedup that protects against webhook redelivery is structurally blind to a loop that generates new messages.

Why quiet hours didn't throttle it

INTERVIEW_SCHEDULED dispatches at the critical tier (lib/repositories/applicationEvents.ts → dispatchNotification({ tier: "critical" })), and critical tier bypasses quiet hours by design (lib/notifications/dispatch.ts: the quiet-hours gate is skipped when tier === "critical"). So the loop ran at full speed regardless of time of day.

The amplifier — free-mail was not a dedup blocklist root

The sender-domain dedup fallback (lib/applications/sender-domain.ts) blocked multi-tenant ATS roots (greenhouse.io, lever.co, …) but not consumer free-mail roots like gmail.com. The user's own notifications are from their gmail.com address, so every re-ingested self-notification resolved to senderDomain = gmail.com, and the fallback funneled them onto whatever app happened to carry that domain. That's how a CalSAWS loop leaked junk events onto Delta Air Lines, the LA Dodgers, and a test row too.

The decisive failure was the missing self-identification. Dedup, quiet hours, and the sender-domain heuristic are all downstream mitigations — none of them is designed to recognize "this email is mine." The system had no way to tell its own voice from the world's.

The $0.66 — token-cost mechanism

Where the money went

Every re-ingested email ran the LLM classifier lib/email-parser.ts:parseApplicationEmail (model: gemini-3.1-flash-lite; ~600 input + ~180 output tokens per call — the 3 KB-trimmed body plus prompt wrapper, returning the full application schema). At ≥552 calls (and likely ~2× that, since the ~1,102 webhook deliveries imply both the dev and prod tiers were ingesting the same mailbox), that adds up to the observed ~$0.66. Classifier retries also pay the cost.

Why the cache barely helped (corrected)

parseApplicationEmail routes through llmCached(), whose key is content-addressed: model + rendered prompt + output-schema fingerprint. The rendered prompt includes the email's own send time as the date-resolution anchor (sentAt.toISOString()), and each of the 552 looped self-notifications was sent at a distinct moment — so they had distinct cache keys and were not collapsible into one. (An earlier draft of this doc wrongly claimed a working cache would have driven the cost to ~$0.00 — corrected here after noticing the parsed interview times in the calendar varied per call.) What the cache would have deduped is the part that genuinely repeats: the Gmail push fan-out delivers each single message to both the dev and prod webhooks ~simultaneously (same msgId / sentAt / body → same key → single-flight: one tier leads, the other follows). That roughly halves the ~1,102 calls back toward ~552, so a healthy cache would have made the loop cost ~$0.33, not ~$0.00. The cache is a duplicate-suppressor, not a loop-breaker — only the self-notification guard prevents the loop.

Why the calendar events landed at many different times. The cache being off meant dev and prod parsed each looped email independently, doubling the parses — but the spread of times (the ~552 events landed on 399 distinct times across four months, not one) comes from a date feedback loop plus a timezone-serialization flip in the LLM's output. Full forensic breakdown in §6.
The cache was disabled at the time — and had been for days. Prod logs showed [llm-cache] disabled … no such column: value. This was schema drift, not the better-sqlite3 ABI mismatch first suspected: the 2026-05-31 refactor that re-homed the LLM cache onto the generic base renamed its payload column result → value and added expiry, but kept the same table name in the same data/llm-cache.db file. CREATE TABLE IF NOT EXISTS no-ops on the pre-existing old-shape table, so the first prepared statement referencing value threw no such column: value, which the best-effort store caught and degraded to uncached, permanently and invisibly. With the cache live, the loop's token cost would have been roughly halved (the simultaneous dev+prod fan-out of each message deduped), not eliminated — the distinct per-message send-time anchors kept the 552 looped emails from collapsing further. The disabled cache amplified the bill ~2×; it didn't cause it. (Fixed 2026-06-04 — the base now detects column drift on init and rebuilds the table rather than silently disabling; see §10.)

Rate limiting did not help the cost

acquireGeminiSlot() (default 12 req/min) paced the calls but does not reduce per-call token spend — cost is calls × tokens-per-call regardless of how they're spread over time. Pacing only changed how long the loop took to burn the money, not how much it burned.

Why it drifted — three bugs compounding

So: is it "the LLM parsing dates wrong" or "something else"? Both, specifically. The LLM mis-serializes the time, but the app keeps handing the LLM its own previous output to re-parse. Neither alone produces a four-month walk.

  • 1 — Feedback fuel: the notification body re-encodes the parsed time. lib/repositories/applicationEvents.ts:61 builds the INTERVIEW_SCHEDULED notification body as event.scheduledAt.toLocaleString(). So every self-notification email literally contained the previous iteration's parsed timestamp as M/D/YYYY, h:mm:ss AM/PM. Proof: 550 of the 552 events' rawText (the wording the LLM says it read) are in that exact toLocaleString shape; the only 2 that aren't are the genuine human seeds (06/03/2026 at 11:00 AM and an earlier 04/22/2026 at 9:15 AM).
  • 2 — Timezone flip: the LLM serializes the wall-clock time as UTC half the time. The parser asks for ISO 8601. Given a zone-less wall-clock time (5/9/2026, 9:00:00 AM), the LLM inconsistently emitted it as local (…T09:00:00) or UTC (…T09:00:00Z). Reading a Pacific wall-clock time as UTC shifts the stored instant earlier by Pacific's offset. Proof: every drifted rawText resolved to exactly two times, 7 h apart (106 rawTexts, the PDT dates) or 8 h apart (36, the PST dates) — i.e. precisely Pacific's UTC offset for that date's season. A random misread wouldn't be season-correct; a local↔UTC confusion is, exactly.
  • 3 — 552 iterations random-walk the date backward. Each UTC-misread subtracts ~7–8 h; the toLocaleString of the new (earlier) time then feeds the next email, so the error compounds. Crossing midnight again and again marched the date backward ~117 days, from the true 6/3 to 2/6. The "clusters" you saw are the stops on that walk: each intermediate time is a small cluster, doubled by its local-vs-UTC pair → 399 distinct values.
Latent beyond the loop. Bug #2 isn't loop-specific — even a single genuine interview email can land 7–8 h off if the LLM appends a stray Z. The loop merely sampled it 552× and made it impossible to miss. Hardening the date serialization (and not feeding a re-parseable timestamp back into notification bodies) is tracked in §10.

How this was reconstructed

The loop's DB rows were deleted in the cleanup (§8) and the live calendar is now partially hand-deleted, so this was derived from the pre-cleanup backup — ~/backups/mission-control/prod.db.pre-mailloop-cleanup-20260602T042424Z — grouping the 552 CalSAWS INTERVIEW_SCHEDULED rows by scheduledAt and by notes (= the LLM's rawText). Each surviving calendar event's description still carries that rawText, so any remaining events can be spot-checked the same way.

The fix

Two-signal self-notification guard (commit 7ae5a6c)

The real fix gives the system a way to recognize its own voice. New module lib/applications/self-notification.ts defines the shared constants and the predicate:

export const MC_NOTIFICATION_HEADER = "X-Mission-Control";
export const MC_NOTIFICATION_HEADER_VALUE = "notification";
export const MC_SUBJECT_PREFIX = "[mission-control] ";

export function isSelfNotificationEmail(args: { subject?: string | null; mcHeader?: string | null }): boolean {
  if ((args.mcHeader ?? "").trim().toLowerCase() === MC_NOTIFICATION_HEADER_VALUE) return true; // primary
  if ((args.subject ?? "").startsWith(MC_SUBJECT_PREFIX)) return true;                          // fallback
  return false;
}
  • Stamp on send. lib/email/send.ts now emits an X-Mission-Control: notification header on every notification (EmailMessage gained a headers? field; buildRfc822Message writes them into the RFC 822 header block) and uses the shared MC_SUBJECT_PREFIX.
  • Skip on ingest. lib/applications/ingest.ts checks isSelfNotificationEmail and returns { action: "skipped", reason: "self_notification" } before the looksRelevant filter and the LLM classifier — so a self-notification costs zero tokens. The header is the primary signal; the subject prefix is the fallback for notification mail already sitting in the inbox that pre-dates the header.
  • Kill the amplifier. New FREE_MAIL_DOMAINS in lib/applications/relevance.ts (20 consumer roots) is folded into sender-domain.ts's BLOCKED_ROOTS, so gmail.com et al. resolve to null and never funnel mail onto one app. This is a dedup-blocklist-only change — deliberately not added to SENDER_DOMAINS, which doubles as a positive relevance signal (widening it would flood the classifier with personal mail).

Defense in depth — two independent signals

The header is robust to a user genuinely titling a real email [mission-control] …; the subject prefix catches notifications that predate the header. Either one trips the skip.

External-artifact cleanup [resolved — 2026-06-04]

What was still outstanding after the DB cleanup

The §8 database cleanup removed the rows, but the loop's two external side-effects — the real Google Calendar events and the real Gmail messages — survived it and, at the time this post-mortem was first written, had no programmatic cleanup path (the row deletion had severed the only stored link to them). Both have since been cleaned; the detail below is preserved because it explains why the calendar turned out far smaller than feared, and seeds the "store undo handles" hardening in §11.

Calendar — cleaned (only ~28 events ever reached gcal, not ~552). The loop synced a calendar event only for a future-dated scheduledAt (the ingest.ts gate), and the date-drift random-walk (§6) marched the parsed times backward into the past — so the vast majority never qualified for a gcal write. Reconstruction from the pre-cleanup backup confirmed only ~28 looped events (14 prod + 14 dev, all dated 6/2–6/3) were ever created on the calendar. The calendar event id is sha1(ApplicationEvent.id) (gcalIdempotencyId), validated against the 14 stored gcalEventIds, so they were addressable after all. 27 were deleted via the Calendar API (the 28th the user had already removed); the window now holds only genuine events. (The ~552 figure is the count of looped emails / DB rows, not calendar events.)

Gmail — already cleaned by the user. The looped [mission-control] … messages had already been deleted from the inbox by the time cleanup ran; a subject:"[mission-control]" search returns zero CalSAWS threads. The remaining matches are ~8 unrelated mid-May smoke-test emails, deletable manually. Note the app cannot delete them programmatically anyway: its Gmail scope is gmail.readonly + gmail.send (no gmail.modify), and the interactive MCP Gmail connection is read-only too — so inbox deletion is user-side until a scope bump (tracked as deferred in §11, Fix C).

The lesson that survives — fire-and-forget side-effects need stored handles

This was recoverable largely by luck (the future-dated gate kept the calendar small; the user had already cleared Gmail). The underlying gap is real: both side-effects are fire-and-forget — the system writes to Calendar / Gmail and records only a success checkpoint, never a handle it could use to undo. Notifications don't persist the sent Gmail messageId (only emailSentAt / emailError), and deleting an ApplicationEvent row throws away its gcalEventId before any gcal sweep. The "store undo handles" item in §11 (Fix C) closes this so a future incident is cleanable by design, not by luck.

Prevention & follow-ups

Shipped (this incident)

  • Self-notification guard — the loop is structurally broken: our own mail is recognized and skipped before classification.
  • Free-mail dedup block — the sender-domain amplifier that spread junk across apps is closed.
  • Hermetic coverage — a closed-loop round-trip smoke now fails the push if the guard ever regresses.

Recommended hardening (not yet done)

Three of these remain open: the notification circuit breaker, timezone-safe date extraction, and storing undo handles. They are worked into an actionable build plan — design decisions all settled 2026-06-04 (§12) and a phased task checklist — in §11.

  • Notification circuit breaker (defense in depth). The guard fixes this loop, but a cap on outbound notifications would catch the next unforeseen amplification before it sends 552 emails. The decisive lesson is that the only thing that stopped this was an external throttle — we should own that limit. Design now settled (§11 / §12): two layers — a global per-user backstop (≤ 10 emails / 1 min) plus an extensible per-feature registry, today only applications scoped per-application (≤ 1 email / 10 min).
  • Fix the LLM cache [done — 2026-06-04]. Root cause was schema drift (column result → value + added expiry from the 2026-05-31 refactor), not a better-sqlite3 ABI mismatch — and the failure was silent: a drifted column threw at db.prepare(), was caught as a generic init error, and disabled the cache for days. Fixed in lib/shared-sqlite-cache.ts:reconcileSchema(): on init the base now compares the on-disk columns to the expected set and, on a mismatch, drops + rebuilds the table loudly (the data is throwaway) instead of disabling. A new hermetic smoke shared-cache-schema-drift-smoke.ts seeds the old schema and asserts the self-heal + warning + actual caching. This repairs the live cache on the next deploy and makes any future drift self-correct rather than billing silently. Note its scope: the cache suppresses duplicate-input spend (e.g. the dev+prod fan-out of one message), so it would have ~halved this loop's cost — but it would not have stopped the loop, whose emails each had a distinct send-time anchor → distinct key. The self-notification guard is the loop fix; the cache is the duplicate-spend defense.
  • Make date extraction timezone-safe (and stop feeding the parser its own output). The four-month calendar drift (§6) had two roots beyond the loop itself: (a) the INTERVIEW_SCHEDULED notification body is scheduledAt.toLocaleString() (lib/repositories/applicationEvents.ts:61), so the email re-encoded the parsed time for the next parse to chew on; and (b) the LLM inconsistently serialized a zone-less wall-clock time as UTC vs local (a 7–8 h flip). The self-notification guard kills the feedback, but the timezone sloppiness is latent — a single genuine interview email can still land 7–8 h off. Design settled (§11 / §12): pin the parser with temperature: 0 and normalize extracted times against USER_TIMEZONE app-side (B1 + B2-ii); the re-parseable toLocaleString() timestamp stays — traced server-side-only, the guard skips it before the parser so it never re-feeds the LLM (OQ6).
  • Store undo handles. Persist the sent notification's Gmail messageId, and sweep gcal events by stored ID before deleting their ApplicationEvent rows — so a future cleanup can actually remove external artifacts (see §9).
  • One-off cleanup of the existing clutter [done — 2026-06-04]. The ~28 looped calendar events that actually reached gcal were deleted via the Calendar API, and the looped inbox messages had already been removed by the user — see §9 for the full account (the "~552 events" feared here turned out to be ~28, because the loop only synced future-dated events while the drift walked into the past).

Goals & non-goals

  • Goal. Own the outbound-volume limit that today only Gmail's external throttle owns; make extracted interview times timezone-correct independent of any loop; make Calendar/Gmail side-effects undoable by storing a handle at write time.

Fix A — notification circuit breaker

Why. Nothing in the app caps outbound notifications. The decisive lesson of this incident is that the only thing that stopped the loop at 552 was Gmail's own send-rate limit — an external throttle we don't control. We should own that limit.

Where. lib/notifications/dispatch.ts:dispatchNotification — the email side-channel fires at lines 122–128, after the in-app row is created. The breaker gates that side-channel.

v1 design — two layers, both must pass (deterministic, DB-backed). This is the project's first outbound-notification limiter, so it's built as a two-tier abstraction from the start: a global backstop every feature inherits, plus a per-feature registry. Before firing the email channel the breaker evaluates both limits; if either is at/over cap it skips the send, records emailError = "circuit breaker: <layer> <N>/<window>", still creates the in-app row, and emits one console.warn.

  1. Global (per-user) backstop — ≤ 10 emails / 1 min across all of mission-control. Counts every email-engaged Notification for the user in the trailing minute, regardless of feature. This is the catch-all that would have stopped the 552-loop at 10. Cheap: the existing @@index([userId, createdAt]) covers a count of in-window rows whose channels still include email. Counting note (impl, 2026-06-08): the breaker counts rows that engaged the email channel (channels contains email), not rows with a non-null emailSentAt as an earlier draft proposed. Two reasons: (1) a breaker must trip on attempts, not successes — when sends are muted (EMAIL_ENABLED=0) or failing (Gmail down), emailSentAt is null on every row yet the runaway still needs stopping; (2) a breaker- or quiet-hours-suppressed row has email stripped from channels, so it self-excludes from the count — correct, it didn't send.
  2. Per-feature limiter (registry) — today only applications, scoped per-application at ≤ 1 email / 10 min. A small registry maps a feature → { scopeKey(dispatch), cap, windowMs }. The applications feature keys on the applicationId in the dispatch payload, so a single job card can email at most once per 10 minutes — the surgical limit the loop most needed. Future features (postings per-watchlist, system alerts, …) register their own rule later; until they do, they're covered by the global backstop alone.

Both checks are per-tier (dev.db / prod.db are separate), so each tier self-caps independently — total outbound is bounded at 2 × each cap, which is enough; the goal is "not 552," not a cross-tier global lock. Reuses the existing Notification table as the ledger — no token bucket, no extra SQLite file — plus one nullable indexed scopeKey column for the per-feature layer (OQ3).

Why DB-count, not the in-memory rate buckets. The existing acquireGeminiSlot / acquireArxivSlot buckets are process-memory (or single-row best-effort) and pace throughput; a breaker needs a durable count that survives restart and is queryable for the "tripped" decision. The Notification rows already are that ledger (channels-engaged + createdAt for the global layer; an indexed scopeKey for the per-feature layer — OQ3).

Fix B — timezone-safe date extraction

Why (latent beyond the loop). Per §6, the parser asks for a bare "ISO 8601" timestamp and the LLM inconsistently serialized a zone-less wall-clock as UTC (…Z) vs local — a 7–8 h flip for Pacific times. The self-notification guard kills the feedback loop, but a single genuine interview email can still land 7–8 h off. Independent bug, still live.

Where. lib/email-parser.ts: the schema's extractedDates[].startsAt describe-string (lines 129–134) and the generateObject call (line 212, no temperature → model default → non-deterministic jitter). The extracted instant is mapped verbatim to ApplicationEvent.scheduledAt at lib/applications/ingest.ts:633, which becomes the gcal event start at lib/calendar/sync.ts:74.

  • B1 — pin determinism (cheap). Set temperature: 0 on the generateObject call. Cuts the run-to-run jitter that smeared the 552 parses across 399 times. The cross-tier cache key is model + prompt + schema — temperature does not affect it, so caching is unchanged.
  • B2 — make the instant zone-correct (the real fix). Chosen: (ii) app-side normalization (OQ5). Three approaches were weighed: (i) prompt-only — pass the user's IANA tz into the prompt and instruct "emit an explicit UTC offset; never bare-Z a wall-clock time" (rejected — trusting the LLM to be tz-deterministic is what caused the drift); (ii) app-side normalization — when startsAt is zone-less or bare-Z but rawText carried no timezone, re-interpret the wall-clock components in USER_TIMEZONE (already derived at lib/calendar/sync.ts:29) ← chosen: deterministic, no schema/eval churn; (iii) schema change — return wall-clock + a separate tz field and combine app-side (rejected — most robust but adds a Promptfoo update for no deterministic gain).
  • B3 — stop feeding the parser its own output (now mostly moot). Resolved: keep it (OQ6). The INTERVIEW_SCHEDULED notification body is event.scheduledAt.toLocaleString() (lib/repositories/applicationEvents.ts:61), which is what re-encoded the prior parse for the next one. The self-notification guard already breaks that loop, so this is belt-and-suspenders — and the timestamp was traced server-side-only (DB + email render + bell), skipped at ingest.ts:134 before parseApplicationEmail, so it never feeds the LLM. Kept for its genuine value in the bell/email.
Output-shape gate. Any change to the parser schema or prompt (B2-i / B2-iii, possibly B1's prompt) requires amending eval/suites/email-parser.yaml and running npm run test:prompts before push (per CLAUDE.md — not in the pre-push gate, burns real Gemini tokens). The chosen path (B1 + B2-ii) touches neither schema nor prompt, so this gate does not apply to v1 — retained only as a guard if B2-iii is ever revisited.

Fix C — store undo handles

Why. §9 was recoverable by luck. The system writes to Gmail/Calendar and keeps only a success checkpoint, never a handle to undo. Store the handle at write time so a future incident is cleanable by design.

  • C1 — persist the sent Gmail messageId. SendResult.messageId already exists (lib/email/send.ts:101, set from res.data.id at line 134) but dispatchNotificationEmail drops it — the success branch writes only emailSentAt / emailError (lines 219–222). Add Notification.emailMessageId String? (migration on both dev.db + prod.db) and write it on the success path. Gives notifications → inbox-message a reverse map.
  • C2 — sweep gcal before deleting rows. deleteEventFromGcal(userId, gcalEventId) exists (lib/calendar/sync.ts:258) but is wired only to the manual per-event DELETE handler. Add a reusable purgeApplicationEvents(eventIds) that deletes the gcal events by their stored gcalEventId first, then the ApplicationEvent rows — and route any future bulk cleanup through it (the §8 cleanup deleted rows directly, which is exactly what lost the IDs).
  • C3 — deferred. A gmail.modify scope bump to enable programmatic inbox deletion (re-consent required — OQ7), and a batch "undo a loop window" admin tool. Out of v1 scope.

Open Questions

Decisions to settle before coding

Resolve these with the user before writing any code (per the repo's design-doc convention). Each carries a proposal; the build order in §13 assumes the proposals unless overridden. All seven resolved 2026-06-04 — every item below is settled and the §13 build order reflects the final decisions.

  • OQ1 — breaker scope. [RESOLVED 2026-06-04] Two layers, both enforced. A global per-user limit across all of mission-control (the backstop), and a per-feature limiter via an extensible registry — today only the applications feature, scoped per-application. Rationale: this is the project's first outbound-notification system; the user wants other features (postings, system, …) to use the same notification/email plumbing later, so the global cap is a feature-agnostic backstop and each feature declares its own scoped rule. For now only applications is registered.
  • OQ2 — breaker cap/window. [RESOLVED 2026-06-04] Global (per-user): ≤ 10 emails / 1 min across the whole app. Per-application (the applications feature): ≤ 1 email / 10 min per job card. Throttling a genuine burst above the global rate is acceptable — critical-tier bursts are rare, and the in-app row still lands even when the email is suppressed.
  • OQ3 — breaker counting (per-feature layer). [RESOLVED 2026-06-04] Option (b): an indexed generic Notification.scopeKey String? column. The per-feature layer counts by scope (for applications, by applicationId), which today lives only inside the opaque Notification.payload JSON string — unindexable and not groupable. Rather than a brittle payload LIKE %"applicationId":"…"% substring scan (option a), the dispatcher writes a namespaced scopeKey in the form "<feature>:<id>" (e.g. application:<applicationId>), indexed via @@index([userId, scopeKey, createdAt]), so the breaker does an exact indexed match. Rationale: the registry model (OQ1) makes per-feature scoping a recurring, first-class need — one generic indexed column serves every future feature (postings:<watchlistId>, system:<type>, …) instead of each reinventing a JSON probe. Distinct from dedupKey: dedupKey is @unique = at-most-once-ever (event:<id>); scopeKey is a non-unique indexed grouping label for a rate window (application:<id>). A single notification carries both.
  • OQ4 — trip behavior. [RESOLVED 2026-06-04] (1) Drop the email channel only, keep the in-app row — mirrors how quiet-hours already strips email from channels in dispatch.ts (the bell still shows it). (2a) Quiet suppression — record emailError = "circuit breaker: <layer> <N>/<window>" on the row + one console.warn; no "breaker tripped" system notification in v1. Rationale: a self-notification about suppression is the same class of thing that fed the original loop; the emailError field + warn log are enough to diagnose a trip. Upgradable to a loud system notification later if a trip ever goes unnoticed.
  • OQ5 — Fix B approach. [RESOLVED 2026-06-04] B1 (temperature: 0) + B2-ii (app-side normalization against USER_TIMEZONE). The parser's generateObject call (lib/email-parser.ts:212) currently sets no temperature; pinning it to 0 removes nondeterminism but is not sufficient on its own — the model still serializes a zone-less wall-clock as bare-Z UTC. The real fix is B2-ii: take whatever the LLM emits and re-interpret it against USER_TIMEZONE in code before storing scheduledAt. Chosen over B2-i (prompt-only — trusting the LLM to be deterministic about tz is exactly what caused the 7–8 h drift) and B2-iii (schema + dedicated tz field — adds Promptfoo/schema churn for no deterministic gain). Net: no schema or eval-suite change.
  • OQ6 — Fix B3. [RESOLVED 2026-06-04] Keep the toLocaleString() timestamp. Traced its fan-out: the value goes into Notification.body (applicationEvents.ts:61) and from there to exactly three sinks — the DB column, the rendered outbound email, and the in-app bell — none an LLM. The outbound email re-enters via the Gmail webhook, but the self-notification guard returns skipped: self_notification at ingest.ts:134, before parseApplicationEmail at ingest.ts:151 — so the body never reaches the parser and burns zero tokens in normal operation. The only path that would feed it to the LLM is a guard failure (header stripped and subject prefix changed), the exact loop the guard prevents. Since it's effectively server-side/user-facing only and genuinely useful context in the bell/email, keep it; dropping it would be marginal defense against an already-closed hole.
  • OQ7 — Fix C scope. [RESOLVED 2026-06-04] Defer the gmail.modify scope bump; store handles only in v1. v1 = persist Notification.emailMessageId (already in SendResult.messageId, currently dropped) + a gcal-sweep-before-delete helper (purgeApplicationEvents) so event-row deletes trash their calendar events instead of orphaning gcalEventIds. The batch "undo a loop window" tool that would programmatically trash the looped emails needs gmail.modify — a re-consent flow that widens mailbox write access — and is deferred: this incident's Gmail side was already cleaned by hand, and storing the handles now gives the durable foundation to build that tool later without coupling the scope bump to v1.
