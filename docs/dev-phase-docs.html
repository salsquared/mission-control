Documentation Standard

Documentation Across Dev Phases

Which kind of document to write at each phase of development — plan → architecture → implementation — and the living-state layer that keeps them honest. Each phase answers a different question; the docs are the artifacts of those answers.

Living standard Applies to: all repos Last updated 2026-05-28

01Purpose

A document is only useful if it answers the question you actually have. The three development phases each ask a different question, so each produces a different kind of doc — with a different audience, level of detail, and shelf life. This standard names those doc types so a new doc has an obvious home and a reader knows what to expect.

The backbone, stated as a relay of questions:

PhaseAnswersThis repo's example
PlanWhat are we building, and why?docs/user-stories.md
ArchitectureHow does the system fit together? What did we decide and trade off?docs/architecture.md
ImplementationHow, concretely, and in what order?docs/implementation.md
Living stateWhere did we leave off, right now?docs/next_steps.md
Companion This doc governs which doc to write. Its sibling, HTML Documentation Standards, governs how to format whichever doc you land on.

02The phase flow

Phases flow forward but loop back — implementation surfaces facts that revise the architecture; real use revises the plan. The living-state doc sits beside all three, recording where the loop currently is.

03Phase 1 — Plan

The plan phase establishes what we're building and why, before committing to how. It is deliberately implementation-free: a plan doc that names a database table has leaked into the next phase. Three doc types live here, from loosest to most structured.

General / overview

The orienting document — vision, goals, operating constraints, non-goals. One page a newcomer reads first. In this repo the role is filled by the opening of user-stories.md and the "Goals and Operating Context" section of architecture.md (single-user, self-hosted, small hardware budget, flaky external APIs). Those constraints are why later decisions look the way they do.

  • In: vision, success criteria, hard constraints, explicit non-goals.
  • Out: schemas, file paths, step-by-step plans.

Sketchpad

The thinking-out-loud doc: rough ideas, open questions, options not yet chosen, half-formed designs. Low ceremony, explicitly disposable — nothing here is a commitment. In this repo, sketchpad thinking lives inside next_steps.md ("Open questions", "design wave" notes) and the "Open product calls" asides scattered through implementation.md rather than a standalone file. Promote a sketch into a user story or a design decision once it firms up; prune it once it's resolved.

Do Date sketchpad entries (absolute ISO) and mark unknowns as open questions. A sketchpad's value is capturing the option you didn't take and why.

User stories

The structured heart of the plan: discrete, prioritized statements of need in "As a <role>, I want <capability> so that <benefit>" form. This repo's user-stories.md is the model — numbered stories (S1.1, S2.3) grouped by feature area, each tagged with a priority and a status:

MarkerMeaningKind
🔴Must-have for next shipPriority
🟡ImportantPriority
🔵Nice-to-have / laterPriority
✅Shipped end-to-endStatus
◐ / ⛔Partial / user-declinedStatus
Keep priority ≠ status A 🔴 story can still be unbuilt. Don't collapse "how badly we want it" and "is it done" into one symbol — user-stories.md tracks both independently, and so should yours.

04Phase 2 — Architecture

The architecture phase answers how the system fits together — the structure, the decisions, and the trade-offs behind them. It is graph-heavy by design: a system-context diagram, a layering diagram, and data-flow diagrams carry more than paragraphs can. docs/architecture.md is the exemplar — it opens with a Mermaid system-context graph, then a layered-architecture table, then per-subsystem data flows, and is explicit that "where this document and the code disagree, the code wins."

What an architecture doc contains

  • Design choices / decision records — what we picked, the alternatives, and why (e.g. "markdown-as-source-of-truth", "in-process cache that survives HMR via globalThis"). Capture the reasoning so it isn't re-litigated.
  • Diagrams, lots of them — system context (boundaries & external integrations), layered/component structure, and data-flow / sequence for the non-obvious paths. Mermaid only (see the format rules).
  • Trade-offs & risks — what the design is bad at, and the constraints that forced it.
Surface vs. synthesis Architecture splits into focused reference docs (this repo: apis.md, frontend_terminology.md, hosting.md — one subsystem each) and a synthesis doc (architecture.md) that ties them together. Big systems want both; small ones can start with synthesis only.

05Phase 3 — Implementation

The implementation phase answers how, concretely, and in what order. This is the granular layer: numbered milestones broken into sub-steps, each with file paths, schema/API shapes, acceptance criteria, and a status. docs/implementation.md is the exemplar — it states up front that user-stories.md says what/why while it says how / in what order, then walks milestones like M7.7 down to ordered sub-tasks (M7.7.1 schema → M7.7.2 rewrite → …).

What an implementation doc contains

  • Milestones & ordered sub-steps — small enough to do in one sitting; the order encodes dependencies.
  • Concrete shapes — the touched file paths, table columns, API contracts, the migration name. Specific enough to act on without re-deriving.
  • Acceptance criteria — how you know a step is done (a passing smoke, an observed behavior).
  • A status legend — this repo uses ✅ shipped · ◐ partial · 🟢 in progress · ⏳ planned · 💤 deferred · ⛔ declined · ❌ killed. Mark each milestone so the doc never overstates progress.
  • Story back-references — every milestone names the user stories it satisfies, closing the loop to Phase 1.
Do Make steps small, ordered, and individually verifiable. Cite the exact file (lib/resumes/select.ts) and the migration name. Mark status honestly.
Don't Restate the why here (that's the plan) or re-explain the structure (that's architecture). Don't let a shipped milestone keep a ⏳ — a stale status is the failure mode this phase is most prone to.
Focused implementation docs A single thorny mechanism can earn its own implementation doc — this repo has resume-pipeline.md (the selection algorithm + a "touch list" for adding a field) and close-detection-probe.md (a probe-gated rollout + runbook). Spin one off when a subsystem's how-to outgrows a milestone entry.

06The living-state layer

Across all three phases sits one perpetually-current doc: the living state. It is not a phase — it's the running log of where the loop is right now: what the last session did, what's in flight, what's blocked, today's critical path, parked TODOs. This repo's next_steps.md fills the role and is read at the start of every session and updated at the end of every one.

Do — living state Capture state-derivable facts: uncommitted work, decisions deferred to "next time", critical-path ordering. Use absolute ISO dates. Keep it short and current.
Don't — living state Store code-derivable facts (architecture, conventions) here — those belong in the architecture doc or a project guide. A living doc that accretes durable facts stops being readable.

07Which doc do I write?

Match the question you're holding to the doc that answers it:

If you're asking…Write / updatePhaseDetail level
"Should we even build this? What's the goal?"General / overviewPlanHigh-level
"What are the options? I'm not sure yet."SketchpadPlanRough, disposable
"What exactly do users need, and how badly?"User storiesPlanDiscrete, prioritized
"How do the pieces fit? What are we trading off?"Architecture / design choicesArchitectureStructural, diagram-heavy
"What are the concrete steps, in order?"Implementation / milestonesImplementationGranular, file-level

One question per doc When a doc starts answering a second question — a user-stories file sprouting schema tables, an implementation doc re-arguing the why — that's the signal to split it. Cross-link instead of duplicating.

08Lifecycle & maturity

A topic moves through the phases over time, and each doc carries a maturity badge in its .meta row so readers know how much to trust it:

BadgeMeaningTypical phase
DraftStill forming; expect churnPlan / sketchpad
LivingContinuously updated; current by designLiving state
StableSettled; changes are deliberateArchitecture / shipped implementation
ArchivedHistorical; superseded but kept for contextany (moved to docs/archive/)

The forward arrow isn't one-way: a shipped milestone teaches you something that revises the architecture, and real-world use revises the plan. Keep the feedback dotted-arrow alive — when implementation contradicts the architecture doc, fix the doc (or note the divergence) rather than letting it rot.
